# VETO 与迭代规则

*与 REQUIREMENTS §3.4.1、§3.4.2 一致；Story 4.1 产出*

---

## 1. 角色一票否决权（§3.4.2）

除评分规则的一票否决（影响环节得分与等级）外，**批判审计员**与**AI 代码教练**均拥有对**迭代结束/达标**的一票否决权。

| 角色 | 一票否决权含义 | 触发条件 | 后果 |
|------|----------------|----------|------|
| **批判审计员** | 对审计闭环收敛的一票否决 | 审计结论为「本轮存在 gap」或「未通过」 | 本轮不收敛，该 stage 迭代不结束；须修复后再次审计，直至连续 3 轮无 gap |
| **AI 代码教练** | 对全流程迭代达标的一票否决 | 教练输出 `iteration_passed: false` 或等效判定 | 本轮迭代不达标；全流程迭代不达标；须按教练建议改进后重新触发教练诊断 |

- 两者**独立行使**：批判审计员否决阶段级审计收敛；教练否决全流程级迭代达标。
- 实施时两者均可触发，任一否决则对应层级不通过。

## 2. OWASP Top 10 高危、硬编码敏感信息（§3.4.1）

**一票否决触发条件**：
- **OWASP Top 10 高危**：发现与 A01–A10 任一类别对应的可利用漏洞。
- **硬编码敏感信息（CWE-798）**：生产代码或配置中包含硬编码的密码、密钥、API Key、数据库凭证等。

**判定方式**：消费 code-reviewer-config 的 veto_items（veto_owasp_high、veto_cwe798）；由 scoring 解析器从 check_items 产出 veto 类 item_id 并传入 veto 判定模块。

## 3. 多次迭代阶梯式扣分（§3.4.5）

| 迭代次数 | 阶梯系数 |
|----------|----------|
| 1 次通过 | 100% |
| 2 次通过 | 80% |
| 3 次通过 | 50% |
| ≥4 次通过 | 0% |

**问题严重等级差异化**：
- **致命问题**：整改≥3 次 → 阶梯系数强制 0
- **严重问题**：整改≥2 次 → 系数降一档

---

## 4. Epic 级一票否决 8 项条件（§3.4.3）

任一条件触发即 Epic 级 veto（综合等级降为 L1 或标记「不达标」）：

| 序号 | 条件 | 默认阈值 |
|------|------|----------|
| ① | 单阶段 veto 次数 | ≥3 |
| ② | 需求交付率 | < 80% |
| ③ | 高危漏洞 | ≥ 2 |
| ④ | 测试通过率 | < 80% |
| ⑤ | 整改≥4 次未通过 Story 数 | ≥1 |
| ⑥ | 一次通过率 | < 50% |
| ⑦ | 整改≥3 次 Story 数 | ≥2 |
| ⑧ | 致命问题整改≥3 次 Story 数 | ≥1 |
