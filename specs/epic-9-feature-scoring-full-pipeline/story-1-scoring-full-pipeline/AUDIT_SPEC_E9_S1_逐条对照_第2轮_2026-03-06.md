## 3. 批判审计员结论（占比 ≥50%）

### 3.1 遗漏需求点

**审计**：Story 9.1 的 10 个 AC 及 Dev Notes 在 spec 中均有映射，无整条 AC 遗漏。TASKS 的 T1–T9、T11 在 spec 中均有对应节，且上一轮遗漏的「aggregateByBranch 不实现」「完整 run 定义」「退化逻辑」「补跑参数」「等效机制」「验收 fixture」均已纳入 spec。

**结论**：无遗漏需求点。✅

### 3.2 边界与术语定义

**审计**：§3.2.3 的 run_id 共享机制已由「等效」改为「二选一实现」，边界清晰。§3.4.2 的「完整 run」已定义（至少 3 stage、implement 计入规则），退化逻辑已写「若无完整 run 则显示数据不足」。「已知 fixture」已约定路径（`__tests__/fixtures/` 或等价）、内容（phase_score、dimension_scores）、断言规则（±1 舍入）。

**结论**：边界与术语均已定义。✅

### 3.3 验收可执行性

**审计**：§3.4.2 的 fixture 验收可执行——实施方在 plan/tasks 中指定 fixture，单测中硬编码预期并断言。§3.2.3 的 run_id 共享验收可由「二选一之一满足」判定。§3.3.1 的补跑参数已明确，验收无歧义。

**结论**：验收可独立执行。✅

### 3.4 与前置文档一致性

**审计**：spec 与 Story 9.1、TASKS 无直接矛盾。实施范围、任务划分、源文件列表一致。关于退化逻辑：TASKS T8 原文为「若无完整 run，退化为按单 record 最新 timestamp 取可用的 stage 子集」；spec 采用「若无完整 run 则显示数据不足」——此为设计简化，与用户期望的「若无完整 run 则显示数据不足」一致，且更易实施与验证。

**结论**：与前置文档一致，无隐性冲突。✅

### 3.5 规格可操作性

**审计**：T1–T9、T11 的规格均可直接操作。补跑参数、完整 run 定义、退化逻辑、aggregateByBranch 排除、fixture 约定均已写入，实施时无需回查 TASKS 推断。

**结论**：规格可操作。✅

### 3.6 潜在风险与建议（批判审计员保留意见）

**风险 1**：§3.4.2 验收 fixture 路径「scoring/data 下预置 `__tests__/fixtures/` 或等价目录」——「下预置」与「或等价」组合时，可解读为「scoring/data/__tests__/fixtures/」或「scoring/data 目录下任一等价结构」。建议：若 plan 阶段采用单测 fixture，可明确为 `scoring/__tests__/fixtures/` 或 `scoring/data/__fixtures__/` 之一，以消除实施阶段歧义。**不影响本轮通过**，属可接受的轻度模糊。

**风险 2**：§3.2.3「二选一实现」——若同时实现两种机制，验收标准未定义优先顺序。建议：实施时选定一种为主，另一种为辅或文档化备选；或明确「至少实现其一」即可。**不影响本轮通过**。

**结论**：上述为可选增强，不构成 blocker。
