# plan-E999-S004：Stage Validator Allowed Next Stages 实现方案

**Epic**：E999 test-epic
**Story**：S004

---

## 1. 实现目标

以最小变更完成 allowed next stages 能力落地，并确保 `StoryStateManager` 在推进阶段时统一复用该语义与结构化错误结果。

## 2. 实施步骤

### Phase 1：测试先行（RED）

1. 在 `tests/story-workflow/StageValidator.test.ts` 增加 allowed next stages 查询断言。
2. 在 `tests/story-workflow/StageValidator.test.ts` 增加：
   - 合法 next stage 通过断言
   - 非法 target stage 返回结构化错误断言
   - 错误中包含 `allowedNextStages` 的断言
3. 在 `tests/story-workflow/StoryStateManager.test.ts` 增加：
   - `advanceStage()` 经过共享语义推进的断言
   - manager 暴露 validator 错误细节的断言
4. 在 `tests/story-workflow/integration.test.ts` 增加：
   - 从 allowed next stages 查询到 manager 推进复用的链路验证
   - 非法推进错误中 `allowedNextStages` 一致性的断言
5. 运行目标测试，确认 RED 失败，证明 allowed next stages 能力与复用语义尚未完整实现。

### Phase 2：最小实现（GREEN）

1. 在 `src/story-workflow/StageValidator.ts` 新增 allowed next stages 查询方法。
2. 让 `validateTransition()` 统一基于 allowed next stages 执行合法性判断。
3. 在结构化错误 details 中写入 `allowedNextStages`。
4. 让 `validateEntry()` 复用更新后的 `validateTransition()`。
5. 在 `src/story-workflow/StoryStateManager.ts` 中通过共享 validator 完成阶段推进校验，保持 manager 不维护第二套阶段语义。

### Phase 3：回归验证

1. 重新运行三份 story-workflow 测试。
2. 确认新增 allowed next stages 场景与既有阶段推进场景同时通过。
3. 核对 spec / plan / gaps / tasks 与代码、测试文件映射一致。

---

## 3. 风险与约束

- 不能破坏既有 `validateEntry` 的返回格式。
- 不能让 `StoryStateManager` 与 `StageValidator` 各自维护不同阶段语义。
- 错误 details 必须稳定包含 `allowedNextStages`，避免测试与调用方依赖消息文本猜测。
- 需要保持阶段历史与日志行为不变。
