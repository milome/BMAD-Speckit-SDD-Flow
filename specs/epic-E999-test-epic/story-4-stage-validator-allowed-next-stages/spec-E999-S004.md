# Spec E999-S004：Stage Validator Allowed Next Stages

**Epic**：E999 test-epic
**Story**：S004 stage-validator-allowed-next-stages

---

## 1. 概述

本 story 以最小真实实现方式，为 `src/story-workflow` 引入单一事实来源的 allowed next stages 能力，并要求 `StoryStateManager` 在阶段推进时统一复用该语义与错误结构。

目标范围：

1. `StageValidator` 对外提供当前阶段的 allowed next stages 查询能力。
2. 合法转换必须基于 allowed next stages 统一判定，不允许散落第二套阶段判断逻辑。
3. 非法转换必须返回结构化 `StageTransitionError`。
4. 错误中至少包含 `fromStage`、`toStage`、`allowedNextStages`、可读错误消息。
5. `StoryStateManager.advanceStage()` 必须复用 `StageValidator` 的 allowed next stages 语义。
6. 相关单元测试与集成测试真实通过。

非目标：
- 不扩展 `src/story-workflow` 之外的模块。
- 不改动 Story 阶段模型定义与工作流主线。
- 不新增与 allowed next stages 无关的状态编排能力。

## 3. 功能规格

### 3.1 StageValidator allowed next stages 能力

`StageValidator` 必须提供稳定、可断言、可复用的 allowed next stages 结果，作为 story-workflow 阶段推进的单一事实来源。

#### 3.1.1 allowed next stages 查询

新增公开能力，返回当前阶段允许进入的下一阶段集合。

约束：

- `create` 的 allowed next stages 为 `['audit']`
- `audit` 的 allowed next stages 为 `['dev']`
- `dev` 的 allowed next stages 为 `['post-audit']`
- `post-audit` 的 allowed next stages 为 `[]`
- 返回值顺序稳定，便于测试直接断言
- 返回值不得原地暴露可变内部引用

#### 3.1.2 基于 allowed next stages 的转换校验

`validateTransition(context)` 必须统一复用 allowed next stages 查询结果完成合法/非法判断。

当 `targetStage` 位于当前阶段 allowed next stages 集合中，且 prerequisite 满足时：

- 返回成功结果

当 `targetStage` 不在当前阶段 allowed next stages 集合中时：

- 返回 `err(StageTransitionError)`
- 错误包含：
  - `fromStage = currentStage`
  - `toStage = targetStage`
  - `details.allowedNextStages = 当前阶段 allowed next stages`
  - `message` 为可读错误消息

当 prerequisite 未满足时：

- 返回 `err(StageTransitionError)`
- `code = ErrorCodes.PREREQUISITE_NOT_MET`
- `details.allowedNextStages` 仍然存在，保证调用方与测试统一消费同一语义

#### 3.1.3 与既有入口校验的兼容

`validateEntry` 继续保留原有 `{ valid, errors }` 返回格式，但内部必须复用 allowed next stages 驱动的 `validateTransition`，避免双重逻辑漂移。

### 3.2 StoryStateManager 统一复用语义

`StoryStateManager.advanceStage()` 在完成当前阶段并计算下一阶段后，必须：

1. 基于当前 state 计算 `completedStages`
2. 获取当前阶段 allowed next stages
3. 通过共享 `StageValidator` 完成转换校验
4. 若失败，直接返回 validator 产出的结构化错误
5. 若成功，才继续更新 `stageHistory`、`currentStage`、日志记录

约束：

- `StoryStateManager` 不得独立维护与 `StageValidator` 不一致的 allowed next stages 集合
- 推进失败时不得吞掉 `allowedNextStages` 等错误上下文
- 合法推进路径行为保持可用

6. 目标测试集全部通过
