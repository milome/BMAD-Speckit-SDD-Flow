# Spec: E001-S001 Email Validator

## Story
作为用户，我需要验证输入的邮箱格式是否正确，以便在注册时提供有效的联系方式。

## 功能需求

| 需求ID | 描述 | 优先级 | 说明 |
|--------|------|--------|------|
| FR-001 | 验证邮箱格式符合 RFC 5322 标准 | P0 | 格式验证，不包含 DNS 检查 |
| FR-002 | 验证域名格式符合标准 | P1 | 仅格式验证（如 example.com），非 MX 记录验证 |
| FR-003 | 提供清晰的错误提示信息 | P1 | 英文错误消息（中文本地化支持不在本 Story 范围） |

## 验收标准

### AC-1: 有效邮箱验证
- **Given** 用户输入有效的邮箱地址
- **When** 调用验证函数
- **Then** 返回验证通过结果

**测试用例**:
- `user@example.com` → valid
- `test.user@domain.co.uk` → valid
- `user+tag@example.com` → valid

### AC-2: 无效邮箱验证
- **Given** 用户输入无效的邮箱地址
- **When** 调用验证函数
- **Then** 返回验证失败及错误原因

**测试用例**:
- `invalid-email` → invalid (缺少 @ 符号)
- `@nodomain.com` → invalid (缺少用户名)
- `user@` → invalid (缺少域名)
- `user@@double@at.com` → invalid (多个 @ 符号)

### AC-3: 错误提示
- **Given** 邮箱验证失败
- **When** 获取错误信息
- **Then** 返回清晰的英文错误描述（中文本地化支持不在本 Story 范围）

**错误码定义**:
| 错误码 | 场景 | 英文消息 |
|--------|------|----------|
| `EMPTY` | 输入为空字符串 | "Email cannot be empty" |
| `INVALID_FORMAT` | 格式不符合 RFC 5322 | "Invalid email format" |
| `TOO_LONG` | 长度超过 254 字符 | "Email exceeds maximum length" |

## 技术约束

- 使用 TypeScript 实现
- 支持 Node.js 18+ 和浏览器环境
- 单元测试覆盖率 ≥ 90%
- 零外部依赖

## 边界条件

| 场景 | 预期行为 | 备注 |
|------|----------|------|
| 空字符串输入 | 返回无效，提示"Email cannot be empty" | 立即返回 |
| 超长邮箱 (>254字符) | 返回无效，提示"Email exceeds maximum length" | RFC 5321 限制 |
| Unicode 字符 | 不在本 Story 范围：当前仅支持 ASCII (RFC 5322) | 将在后续 Story 实现 |
| 大小写敏感 | 域名部分不区分大小写 | 验证时统一转小写 |

## 接口定义

```typescript
interface EmailValidationResult {
  valid: boolean;
  error?: string;
  errorCode?: 'EMPTY' | 'INVALID_FORMAT' | 'TOO_LONG';
}

function validateEmail(email: string): EmailValidationResult;
```

## 依赖

无外部依赖。仅使用标准 JavaScript API。

## 非功能需求

- 性能: 单次验证 < 1ms
- 包大小: < 2KB (gzip)

## 不在本 Story 范围的功能

以下功能不在当前 Story 范围内：

| 功能 | 描述 | 关联需求 |
|------|------|----------|
| Unicode 支持 | RFC 6532 国际化邮箱支持 | 边界条件/Unicode |
| 多语言错误消息 | i18n 本地化错误提示 | AC-3/错误提示 |
| MX 记录验证 | DNS 查询验证域名可送达性 | FR-002/域名验证 |
| 常见邮箱域名白名单 | gmail.com, outlook.com 等 | FR-002/域名验证 |
