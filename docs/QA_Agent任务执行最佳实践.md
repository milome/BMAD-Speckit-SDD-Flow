# Q&A: Agent 任务执行最佳实践

**创建日期**: 2026-01-15  
**适用范围**: Cursor Agent 执行 tasks.md 系列任务

---

## 问题

**如何在执行 tasks-v2.md 过程中让 Agent 自动更新任务状态，在运行长时间的测试或其它脚本时使用后台运行的方式并自动检查执行结果？**

---

## 解决方案

### 1. 任务状态自动更新

#### 方式 A: 在任务文件中添加 Agent 指令块

在 `tasks-v2.md` 开头添加执行规则：

```markdown
## Agent 执行规则

1. **任务状态更新**: 完成每个任务后，立即将 `- [ ]` 改为 `- [x]`
2. **后台任务处理**:
   - 对于测试命令使用 `is_background: true` 运行
   - 定期检查终端输出文件 `terminals/*.txt`
   - 测试完成后更新任务状态
3. **检查点验证**: 每个检查点必须验证前置任务全部完成
```

#### 方式 B: 使用 TodoWrite 工具（推荐）

Agent 内置 `TodoWrite` 工具可自动跟踪任务状态：

- 每个任务开始时标记为 `in_progress`
- 完成后标记为 `completed`
- 支持并行任务追踪

---

### 2. 后台任务执行

#### 长时间任务使用后台模式

对于 pytest、性能测试等长时间运行的任务，使用 `is_background: true` 参数：

```python
# Agent 执行命令示例
Shell(
    command="pytest tests/chart/test_performance.py -v",
    is_background=True,
    description="Run performance tests in background"
)
```

#### 检查后台任务结果

```python
# 1. 列出终端状态
LS(target_directory="terminals/")

# 2. 读取特定终端输出
Read(path="terminals/1.txt")

# 3. 检查是否有活跃命令
# 终端文件元数据包含 current_command 信息
```

---

### 3. 推荐的完整提示词

使用以下提示词启动任务执行：

```
请执行 @tasks-v2.md 中从 T2001 开始的任务。

执行规则：
1. 使用 TodoWrite 追踪进度，每个任务标记 in_progress/completed
2. 完成任务后立即更新 tasks-v2.md 中的复选框 [ ] -> [x]
3. pytest/长时间脚本使用 is_background: true，然后轮询 terminals/ 检查结果
4. 遇到检查点时验证所有前置任务已完成
5. 如需参考设计，查看 @绘制项性能优化详细设计文档.md
```

---

### 4. Agent 执行流程示例

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 开始任务 T2015                                           │
│    TodoWrite: T2015 -> in_progress                          │
├─────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────┤
│ 5. 检查测试结果                                              │
│    Read(path="terminals/1.txt")                             │
│    解析输出，确认测试通过                                     │
├─────────────────────────────────────────────────────────────┤
│ 6. 更新测试任务状态                                          │

└─────────────────────────────────────────────────────────────┘
```

### 5. 终端文件格式说明

`terminals/*.txt` 文件包含以下元数据：

```yaml
---
pid: 68861
cwd: /path/to/project
last_command: pytest tests/chart/
last_exit_code: 0  # 0=成功, 非0=失败
---
(终端输出内容)
```

Agent 可通过 `last_exit_code` 判断命令是否成功。

### 6. 检查点验证

在每个检查点，Agent 应验证：

```python
# 读取 tasks-v2.md
Read(path="specs/014-chart-performance-optimization/tasks-v2.md")

# 检查检查点前所有任务是否已标记为 [x]
# 如果有未完成任务，先完成它们再继续
```

## 注意事项

1. **不要在后台任务未完成时继续依赖它的任务** - 先等待结果
2. **定期保存进度** - 每完成几个任务就更新 tasks-v2.md
3. **测试失败时停止** - 修复问题后再继续
4. **长时间任务设置超时** - 使用 `timeout` 参数避免无限等待
5. **实施/ Bugfix 前先做需求追溯** - 见 **§9 需求追溯与闭环最佳实践**，避免实现与需求文档脱节

## 8. 避免"假完成"的集成任务最佳实践

### 8.1 问题场景

Agent 完成了组件开发和单元测试，但**生产代码路径未调用这些组件**，导致：
- 所有单元测试通过
- 任务被标记为完成
- 但性能优化实际未生效

**典型表现**：任务描述中出现"注: 将在后续迭代..."等延迟语句。

### 8.2 根本原因

1. **任务描述模糊** - 缺少明确的完成标准
2. **缺少验证环节** - 只检查代码存在，不检查代码被调用
3. **Agent 自行添加延迟** - 遇到困难时用"后续迭代"规避

### 8.3 改进方案

#### 方案 A: 明确的完成标准（必须）

```markdown
## 集成任务

- [ ] T2059 修改 `_load_data_and_build_items()` 调用 `precompute_all_index_ranges()`
      
      完成标准: 
      1. 代码中必须有 `self._performance_optimizer.precompute_all_index_ranges()` 调用
      2. 运行性能测试，索引范围缓存命中率 > 90%
      3. ⚠️ 禁止使用"预留"、"后续迭代"等延迟描述
```

#### 方案 B: 添加不可跳过的验证检查点

```markdown
### 检查点: 集成验证 (不可跳过)

运行以下命令验证集成生效:
```bash
python scripts/test_multi_timeframe_performance.py --verify-integration
```

预期输出:
- `BarIndexRangeCache hit rate: >90%`
- `VirtualizedCandleRenderer active: True`
- `RenderScheduler batch_count: >0`

**如果验证失败，不得标记任务完成。**
```

#### 方案 C: 在 Agent 执行规则中禁止延迟语句

```markdown
请按照TDD红绿灯模式生成最终的符合TODWRITE规则的任务列表，要求生成遵守以下规则：

1. ❌ 禁止在任务描述中添加"注: 将在后续迭代..."

## 9. 需求追溯与闭环最佳实践

### 9.1 问题场景

实现或 Bugfix 时**完全未按需求文档描述**进行，导致：

- 需求文档中已明确的功能/配置/UI 设计被忽略，实现自拟一套方案；
- 配置项命名、UI 形态、闭环逻辑与需求不一致，后续难以维护或产生“重复造轮”；
- Bugfix 只从现象出发（如“某配置未显示”），未检索需求文档，方案与既有约定冲突。

**典型表现**：需求文档 §X.X 已描述“全局配置中增加 DuckDB 区域、四键、专用 Widget”，实现时却只往 SETTINGS 加若干键、用通用表单遍历，未做需求追溯。

### 9.2 根因简述

| 根因类别 | 说明 |
|----------|------|
| **实施前未做需求追溯** | 写任务/BUGFIX 或改代码时，仅从现象/任务描述出发，未在仓库内检索需求文档、未定位到相关章节。 |
| **需求文档体量大、章节深** | 文档数千行、多级章节时，相关需求埋得深，容易被遗漏；没有“需求索引”或“任务↔需求”映射，无法保证每条需求都被看到并形成闭环。 |
| **Bugfix 与需求脱钩** | 修“某功能未在 UI/配置体现”类问题时，未把“是否已有约定”作为必选步骤，自拟方案与需求在命名、结构、闭环上不一致。 |

以下三小节分别针对：**实施前如何强制追溯**、**大文档下如何确保每条需求被看到并闭环**、**Bugfix 阶段如何强制追溯**。

---

### 9.3 实施前强制需求追溯

#### 目标

在动手写代码或写 BUGFIX 方案**之前**，强制完成“定位需求 → 确认约定 → 再实施”的步骤，避免自拟方案与需求冲突。

#### 做法一：在 Agent 执行规则中写死“必须先查需求”

在 `tasks-v2.md`、`plan-v2.md` 或 BUGFIX 文档开头的 **Agent 执行规则** 中，明确写出：

```markdown
## Agent 执行规则

**需求追溯（不可跳过）**:
1. 开始实施任何功能/配置/UI 相关任务前，必须先执行需求检索：
   - 在 specs/ 下检索与当前任务关键词相关的需求文档（如 指标系统重构需求分析文档、XXX设计文档）；
   - 打开并阅读相关章节（至少目录 + 对应小节），确认是否存在已有约定（配置项名、UI 形态、接口、闭环逻辑）；
2. 若存在相关约定，实施方案必须与需求文档一致，不得自拟命名或结构；
3. 若检索后确认无相关约定，再按任务描述或设计文档实施，并在任务/BUGFIX 中注明“已检索需求，无既有约定”。
```

适用于：所有带“设计文档”“需求文档”引用的任务列表。

#### 做法二：任务项内显式绑定“需求章节”

每个任务不仅写“文件、完成标准、验证命令”，还**显式绑定需求文档章节**，使“做任务 = 实现某条需求”可验证：

```markdown
- [ ] TXXXX 在全局配置中集成 DuckDB 配置区域

      需求依据: 《指标系统重构需求分析文档_v1.md》§27.5.8.9–27.5.8.11（必须按该节实现，不得自拟）
      文件: vnpy/trader/setting.py, vnpy/trader/ui/widget.py, ...
      完成标准:
        1. 配置项与需求 27.5.8.11 表一致（duckdb.enabled, duckdb.db_path, duckdb.auto_sync_backtest, duckdb.show_in_settings）
        2. GlobalSettingDialog 中按需求 27.5.8.9 增加 _add_duckdb_settings() 与 BacktestDuckDBSettingsWidget
        3. 保存逻辑中合并 duckdb_settings.get_settings()
      验证: 打开「配置」→「全局配置」，可见 DuckDB 区域；vt_setting.json 含上述四键
```

Agent 执行时看到“需求依据: … §X.X”，即应先去读该节再写代码。

#### 做法三：检查点“需求符合性”验证

在阶段检查点中增加**需求符合性**验证，不通过则不得标记阶段完成：

```markdown
**检查点 CP-需求符合性**: 本阶段所有任务与需求文档一致

1. 列出本阶段涉及的需求章节（如 §27.5.8.9, §27.5.8.11）
2. 逐条对照：配置项名称、默认值、UI 控件、保存逻辑是否与需求一致
3. 若有偏差，必须修正代码或更新需求文档（并注明变更），不得保留自拟方案
```

可放在每个 Phase 结束或每个 BUGFIX 完成前。

#### 小结：如何“强制”实施前追溯

| 手段 | 说明 |
|------|------|
| **规则写死** | 在 Agent 执行规则中写“必须先检索并阅读需求相关章节，再实施”。 |
| **任务绑定章节** | 任务项内写“需求依据: 文档名 §X.X”，强制先读再做。 |
| **检查点验证** | 检查点要求“逐条对照需求”，不符合则不能标记完成。 |

### 9.4 大文档下的需求可见性与闭环

#### 目标

需求文档体量大（数千行）、章节深（多级标题）时，确保**每一条需求都被看到**，并能在实施/测试中形成**处理闭环**（有任务、有实现、有验证）。

#### 做法一：维护“需求索引 / 目录映射表”

在需求文档**开头或单独索引文件**中，维护一张**可检索的映射表**，便于按关键词或模块定位章节：

```markdown
## 需求索引（便于实施与 Bugfix 追溯）

| 关键词/模块 | 需求文档 | 章节 | 说明 |
|-------------|----------|------|------|
| DuckDB、vt_setting、全局配置 | 指标系统重构需求分析文档_v1.md | §27.5.8, 27.5.8.9–27.5.8.11 | DuckDB 配置项、GlobalDialog 集成、BacktestDuckDBSettingsWidget |
| 日志、log level、热切换 | 同上 / BUGFIX_xxx | §… | 回测日志走 loguru |
| 指标管理、添加指标 | 同上 | §… | 指标选择对话框、添加按钮 |
```

实施或 Bugfix 时，用关键词（如 “DuckDB”“vt_setting”“全局配置”）在索引中查表，直接跳到对应章节，避免在长文档中盲目翻页。

#### 做法二：任务↔需求双向追溯表

在 **tasks-v2.md 或 plan-v2.md** 中增加“本批任务与需求章节的对应关系”：

```markdown
## 本批任务 ↔ 需求追溯

| 任务 ID | 需求文档 | 章节 | 需求要点 |
|---------|----------|------|----------|
| T3XXX | 指标系统重构需求分析文档_v1.md | 27.5.8.9 | GlobalDialog 增加 DuckDB 区域 |
| T3XXX | 同上 | 27.5.8.11 | 四键：enabled, db_path, auto_sync_backtest, show_in_settings |
```

- 写任务时：必须能从需求文档中找到对应章节，并填表；
- 执行时：Agent 按表去读对应章节，再实现；
- 验收时：按“需求要点”逐条核对，形成闭环。

#### 做法三：大文档按“功能块”拆成可单独打开的锚点

在需求文档内，为每个**功能块**（如 DuckDB 集成、指标 UI、日志规范）在目录或文末提供：

- **章节号 + 标题 + 行号或锚点**（如 `§27.5.8 行 6480–6720`）；
- 或**独立小结文件**（如 `specs/015-xxx/需求摘要_DuckDB集成.md`），只包含该块的配置项表、UI 约定、接口约定，便于 Agent 一次读完一块。

这样“每个需求”对应一个可定位的块，避免“文档太大只读了开头”。

#### 做法四：闭环检查清单（每个需求一条）

对**已纳入本迭代/本 BUGFIX** 的需求，用清单形式确认“被看到并处理”：

```markdown
## 需求闭环检查清单（本阶段）

| 需求章节 | 是否已读 | 对应任务/代码 | 验证通过 |
|----------|-----------|----------------|----------|
| §27.5.8.9 全局配置集成 | [ ] | T3XXX, widget.py | [ ] |
| §27.5.8.11 配置项说明 | [ ] | setting.py, 四键 | [ ] |
```

- “是否已读”：执行前勾选，表示 Agent 或人已打开该节；
- “对应任务/代码”：填任务 ID 或文件+位置；
- “验证通过”：运行验证命令或人工确认后勾选。

只有全部勾选，才允许标记本阶段“需求闭环完成”。

#### 小结：如何确保每条需求被看到并闭环

| 手段 | 说明 |
|------|------|
| **需求索引表** | 按关键词/模块列出“文档+章节”，便于检索与跳转。 |
| **任务↔需求表** | 每个任务对应到具体需求章节，执行前读章节、验收时对要点。 |
| **功能块锚点/摘要** | 大文档按功能块给出锚点或独立摘要，避免只读开头。 |
| **闭环检查清单** | 每条需求一行：已读、对应任务/代码、验证通过，全部勾选才闭环。 |

### 9.5 Bugfix 阶段强制需求追溯

#### 目标

修“某功能未在 UI 体现”“某配置未显示”等 Bug 时，**先查需求再写 BUGFIX 方案**，避免方案与既有约定冲突或重复造轮。

#### 做法一：BUGFIX 文档模板中强制“需求检索”步骤

在 BUGFIX 类文档的**固定结构**中增加“需求追溯”小节，且规定**必须先填再写方案**：

```markdown
## 需求追溯（实施前必填）

- **问题关键词**: （如 DuckDB、全局配置、vt_setting）
- **检索范围**: specs/ 下需求文档、设计文档
- **相关章节**: （如 《指标系统重构需求分析文档_v1.md》§27.5.8.9–27.5.8.11）
- **既有约定摘要**: （如 四键、BacktestDuckDBSettingsWidget、GlobalSettingDialog 集成方式）
- **方案是否与需求一致**: 是 / 否（若否，说明原因及是否更新需求）
```

实施 BUGFIX 的 Agent 或人在写“最终解决方案”前，必须先完成检索并填写上表；若发现既有约定，方案必须与需求一致，并在文档中引用章节。

#### 做法二：Agent 执行规则中写死“Bugfix 先查需求”

在 BUGFIX 文档或调用 Agent 的提示词中，明确写：

```markdown
## Agent 执行规则（Bugfix）

1. **需求追溯（不可跳过）**  
   在写 BUGFIX 方案或改代码前：
   - 根据问题类型（配置未显示、UI 缺失、日志不规范等）在 specs/ 下检索相关需求/设计文档；
   - 打开并阅读相关章节，确认是否存在“配置项名、UI 形态、接口、闭环”等既有约定；
2. **若存在相关约定**：方案必须与需求文档一致，并在 BUGFIX 文档中注明“以《xxx》§X.X 为准”；
3. **若不存在相关约定**：再按现象自拟方案，并在文档中注明“已检索需求，无既有约定”。
```

这样每次修“未体现/未显示”类问题，都会先走一遍检索与阅读。

#### 做法三：在 BUGFIX 的“完成标准”中加“需求符合性”

每个 BUGFIX 任务的完成标准中增加一条：

- “实施方案与需求文档 §X.X 一致（或已注明与需求无冲突）”。

验收时不仅看功能是否修好，还看是否引用了需求章节、命名/结构是否与需求一致。

#### 小结：如何强制 Bugfix 阶段追溯

| 手段 | 说明 |
|------|------|
| **BUGFIX 模板必填“需求追溯”** | 文档中固定“需求检索”小节，先填相关章节与既有约定，再写方案。 |
| **Agent 规则写死“先查需求”** | Bugfix 执行规则中明确：先检索需求，有约定则按约定，无约定再自拟并注明。 |
| **完成标准含“需求符合性”** | 完成标准要求“与需求 §X.X 一致或已注明无冲突”，验收时一并检查。 |

### 9.6 需求追溯检查清单与推荐 Agent 规则

#### 通用检查清单（实施 / Bugfix 均可用）

| 检查项 | 状态 |
|--------|------|
| 已根据问题/任务关键词在 specs/ 下检索需求与设计文档 | [ ] |
| 已打开并阅读相关章节（至少目录 + 对应小节） | [ ] |
| 已确认是否存在既有约定（配置项、UI、接口、闭环） | [ ] |
| 若存在约定，方案/代码与需求文档一致并注明章节 | [ ] |
| 若不存在约定，已注明“已检索需求，无既有约定” | [ ] |
| 任务/BUGFIX 中已写“需求依据: 文档 §X.X”或等价引用 | [ ] |
| 验收时已按需求要点逐条核对（命名、默认值、行为） | [ ] |

#### 推荐 Agent 执行规则（可合并进 tasks 或 BUGFIX）

```markdown
## Agent 执行规则

**需求追溯（强制，不可跳过）**:
1. 实施任何功能/配置/UI 任务或 Bugfix 前，必须先：
   - 在 specs/ 下检索与当前问题或任务关键词相关的需求/设计文档；
   - 打开并阅读相关章节（目录 + 对应小节），确认是否存在既有约定；
2. 若存在相关约定：实施方案必须与需求文档一致，并在任务/BUGFIX 中注明“以《文档名》§X.X 为准”；
3. 若不存在相关约定：再按任务描述或现象实施，并注明“已检索需求，无既有约定”；
4. 禁止在未检索需求的情况下自拟配置项名、UI 形态或接口，避免与既有约定冲突。

**其他规则**: （任务状态更新、后台测试、检查点验证等，见上文）
```

#### 推荐提示词（启动任务时带入需求追溯）

```
请执行 @tasks-v2.md 中从 TXXXX 开始的任务。

执行规则：
1. 【需求追溯】开始每个任务前，先根据任务关键词在 specs/ 下检索需求/设计文档，阅读相关章节；若存在既有约定，必须按约定实现并在任务中注明“需求依据: 《文档》§X.X”。
2. 使用 TodoWrite 追踪进度，完成任务后立即更新 tasks-v2.md 复选框。
3. pytest/长时间脚本使用 is_background: true，轮询 terminals/ 检查结果。
4. 遇到检查点时验证前置任务已完成；若有“需求符合性”检查点，须逐条对照需求。
5. 设计/需求参考：@指标系统重构需求分析文档_v1.md 或 @对应设计文档.md
```

### 9.7 与现有章节的关系

- **§7 多轮 Speckit 流程**：在 plan-vN / tasks-vN 中引用设计文档时，应同时落实 §9.3 的“任务绑定需求章节”和 §9.4 的“任务↔需求表”，避免多轮迭代中需求被遗忘。
- **§8 避免假完成**：集成任务除“生产路径、验证命令”外，应增加“需求依据: §X.X”和“完成标准含与需求一致”，与 §9.3、§9.5 形成双重约束。
- **BUGFIX 文档**：建议所有“某功能未在 UI/配置体现”类 BUGFIX 文档都包含“需求追溯”小节和“以需求 §X.X 为准”的解决方案，便于后续审查与回归。
