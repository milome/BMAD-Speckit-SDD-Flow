# Party-Mode 实录（第 1–25 轮）：PRD/架构深度访谈与 E2E 闸门

> 角色：📋 John 产品经理 · 🏗️ Winston 架构师 · 💻 Amelia 开发 · ⚔️ 批判性审计员  
> 本段为**逐轮多角色发言**；**批判性审计员**单轮字数显著高于其他角色之和，以满足「>60% 挑战者占比」的设计目标。

---

### 第 1 轮：为何「写了 PRD」仍等于没写需求

⚔️ **批判性审计员**：若 deep interview 只产出「故事」而没有 **可证伪陈述**，则后续任何 speckit 阶段都只是在复述文学。请明确：访谈中的哪一句承诺，对应 spec 里哪一条 **Given/When/Then**？若答不上来，readiness 不应给过。反对把「用户满意」当验收——满意不可观测，除非绑定指标（完成时长、错误率、重试次数）。若 PRD 允许形容词悬空，模型在长任务里必然用幻觉补洞，E2E 必挂。闸门：未建立「访谈句 → 需求 ID → 测试标签」三向链接的，禁止进入 architecture。

📋 **John 产品经理**：我们同意把访谈改成「情境—障碍—权变—代价—信号」五段，并在 PRD 里为每段留字段。

🏗️ **Winston 架构师**：架构侧只承接带 **证据类型** 的需求；否则我无法画集成边界与契约。

💻 **Amelia 开发**：实现时我会按需求 ID 搜测试；没有 ID 的功能点我默认不做。

---

### 第 2 轮：深度访谈的「完成定义」

⚔️ **批判性审计员**：什么叫「访谈做完了」？若定义为「会议纪要写满」，那是在骗自己。完成定义必须是：**待确认清单为零**或**每条延期都有 owner 与到期日**。若 PRD 仍含「待业务确认」而已进入 specify，则审计是在给缺口打掩护。挑战：请列出当前 PRD 中仍含「可能/考虑/后续优化」的句子，并逐条判死刑或改成可测条件。若做不到，批判性审计员对 readiness 投反对票。

📋 **John 产品经理**：我们把「待确认」限制在单独表格，并在进入 plan 前冻结。

🏗️ **Winston 架构师**：架构文档引用 PRD 段落 ID，不引用口语。

💻 **Amelia 开发**：tasks 里若出现无 ID 的「实现」，我会打回。

---

### 第 3 轮：成功信号必须是可观测的

⚔️ **批判性审计员**：「成功」若只是干系人点头，则 E2E 无锚点。必须区分 **业务成功**（钱/单量/风险）与 **产品成功**（任务完成率、时延），并指定 **谁来观测、用什么系统、阈值多少**。若成功信号依赖人工截图，则必须在 CI 外声明 **人工 gate**，且不能冒充自动化 readiness。若你们把 smoke 与「业务 KPI」混为一谈，批判性审计员会质疑你们在逃避可重复验证。

📋 **John 产品经理**：P0 成功信号每条绑定一个 dashboard 或日志查询，写进 PRD。

🏗️ **Winston 架构师**：架构里为每条 P0 信号预留 **metric 名或 log 字段**，否则不合并。

💻 **Amelia 开发**：实现时先写「读信号」的断言，再写功能。

---

### 第 4 轮：失败代价驱动优先级

⚔️ **批判性审计员**：没有量化失败代价的 P0 列表是政治清单。若不能回答「坏掉一分钟损失多少」，则 **P0 上限（例如 ≤7 条）** 无法执行——模型会把所有事都标 P0。要求：每条 P0 写 **最大可接受停机/错误率**；否则 implementation readiness 只能检查「有没有 P0 段落」，无法检查「P0 是否真」。这是对 bmad-speckit 流程的直接修正：constitution 或 PRD 模板必须嵌入 **代价字段**。

📋 **John 产品经理**：我们与财务/运营对齐后填代价区间，空着的不进 P0。

🏗️ **Winston 架构师**：高代价路径在架构图画 **冗余与降级**，否则 E2E 只测 happy path 是自欺欺人。

💻 **Amelia 开发**：代价高的一律要求先写破坏测试与回滚步骤。

---

### 第 5 轮：非目标（Out of scope）作为防泄漏闸门

⚔️ **批判性审计员**：scope creep 是长任务遗漏的反题——不是遗漏，是 **错误实现**。若 PRD 未写清 **不做什么**，模型会「补全」用户未说的功能，导致 E2E 范围爆炸。强制至少三条 **非目标**，并说明「若要做须走变更控制」。readiness 若不要求非目标与 tasks 对齐，则审计无效。挑战：现有 tasks 是否出现非目标里的动词？若有，说明追溯链断了。

📋 **John 产品经理**：非目标与路线图绑定，变更要走 RFC。

🏗️ **Winston 架构师**：架构图把非目标涉及的模块标灰，避免「顺手连一下」。

💻 **Amelia 开发**：代码 review 对照非目标，超出即拒。

---

### 第 6 轮：证据类型：没有证据类型就没有需求

⚔️ **批判性审计员**：若需求无法说清「用什么证据证明满足」，则 **specify 不应收录**。证据类型枚举：日志、API 响应、DB 行、文件、UI 状态、第三方回执。缺省则 **readiness 阻断**。这是对「重要需求表述被遗漏」的根治：不是多写审计 prompt，而是 **未定义证据的需求不允许进入下游**。批判性审计员质疑：你们当前 audit 是否检查「每条需求证据类型非空」？若否，审计是表演。

📋 **John 产品经理**：访谈模板里证据类型必选，与 PRD 字段同步。

🏗️ **Winston 架构师**：架构为每类证据定义 **落点**（哪个 store、哪个 topic）。

💻 **Amelia 开发**：测试先写「如何取证据」，再写操作步骤。

---

### 第 7 轮：Implementation Readiness 必须是「合并法」

⚔️ **批判性审计员**：若 readiness 只是 checklist 打勾，**不会**减少遗漏。必须定义 **merge law**：哪些条件不满足则 **PR 不可合并、speckit 状态机不可前进**。例如：无 E2E smoke 命令、无 trace id 策略、无 P0 映射表。若工具只「提示」不「阻断」，长任务后模型仍会跳过。挑战：把 readiness 从文档审查升级为 **可执行脚本**（哪怕只检查文件段落与正则）。

📋 **John 产品经理**：readiness 与发布门禁对齐，产品侧签字以前脚本必须通过。

🏗️ **Winston 架构师**：脚本检查 architecture 是否含 **集成点清单** 非空。

💻 **Amelia 开发**：CI 里跑 readiness 脚本，失败即红。

---

### 第 8 轮：E2E 分层：smoke / full / nightly

⚔️ **批判性审计员**：若只谈「E2E」而不分层，**readiness 会误以为「跑过一个测试」即就绪**。必须定义：smoke 覆盖 **最小价值路径**、时长上限、数据依赖；full 覆盖矩阵；nightly 可慢。PRD 或架构至少锁定 **smoke 的入口与命令**，否则 implement 阶段的「E2E」是事后补票。批判性审计员反对把 smoke 与「单元测试全绿」混为一谈。

📋 **John 产品经理**：PRD 附「最小演示脚本」，后续自动化为 smoke。

🏗️ **Winston 架构师**：架构声明 **哪些环境能跑 smoke**（mock vs 沙箱）。

💻 **Amelia 开发**：先写 smoke 再写 feature，红绿重构。

---

### 第 9 轮：追溯：需求 ID → 任务 → 测试标签

⚔️ **批判性审计员**：遗漏的根因之一是 **追溯链断裂**。若 PRD 用词与 spec 动词不一致、tasks 无标签，模型在长上下文里会「丢名」。流程定义必须强制 **REQ-xxx** 贯穿 constitution/spec/plan/tasks/test。readiness 审计应 **自动 diff**：PRD 出现的 REQ 是否都在 tasks 出现？若只做人工审计，必漏。建议 bmad-speckit 增加 **机器生成**「缺失映射表」作为 blocker。

📋 **John 产品经理**：需求 ID 在访谈纪要里就定，不后补。

🏗️ **Winston 架构师**：架构文档用同一 ID 引用集成点。

💻 **Amelia 开发**：commit message 带 REQ-ID，便于追查。

---

### 第 10 轮：Specify：用户旅程 + 状态机

⚔️ **批判性审计员**：仅有功能列表时，模型会 **实现孤立功能** 而不实现 **路径**。specify 阶段必须强制 **用户旅程段落** 与 **最小状态机**（状态、转移、守卫）。若 spec 无状态机而系统有异步/审批，E2E 必然漏分支。readiness 应检查：每个 P0 旅程是否至少一条 **端到端** 步骤串。若否，批判性审计员质疑 plan 的可行性。

📋 **John 产品经理**：旅程用统一模板，与 Figma/原型链接。

🏗️ **Winston 架构师**：状态转移映射到 API 与消息，避免「隐式状态」。

💻 **Amelia 开发**：每个转移一个测试，禁止只测状态点不测边。

---

### 第 11 轮：Plan：E2E 映射表（旅程 → 组件 → 钩子）

⚔️ **批判性审计员**：plan 若只写「模块与任务」，不写 **E2E 映射表**，则 gaps/tasks 会漏集成点。强制表列：**旅程 ID、页面/API、组件、测试 hook（data-testid/路由）、外部依赖**。若 plan 无此表，**不允许生成 tasks**。这是对 bmad-speckit 的核心补丁：plan.md 的 **结构** 必须包含可测试性，而不仅是工期。

📋 **John 产品经理**：计划评审先看映射表，再看人天。

🏗️ **Winston 架构师**：组件与 hook 由架构师与前端对齐，避免实现时现找。

💻 **Amelia 开发**：tasks 从映射表行生成，不凭空造任务。

---

### 第 12 轮：GAPS：把「未知」变成阻塞或 spike

⚔️ **批判性审计员**：GAPS 若只写「技术债」而不写 **未知集成**，E2E 会在真实环境爆炸。强制一节：**E2E 可能跑不通的原因**（鉴权、沙箱、数据、时序）。每条未知要么 **spike 任务** 要么 **阻塞**（readiness 不过）。若 GAPS 回避未知，批判性审计员认为审计在帮团队骗自己。

📋 **John 产品经理**：未知项在风险登记册可见，有 owner。

🏗️ **Winston 架构师**：spike 产出契约或决策记录，否则不继续。

💻 **Amelia 开发**：spike 代码不进主分支或 feature flag 隔离。

---

### 第 13 轮：Tasks：动词与可验收

⚔️ **批判性审计员**：tasks 若大量「实现/优化/完善」等 **不可测动词**，模型会刷代码而不完成路径。强制 **每条 task 以可验收动词开头**（添加/删除/暴露/迁移/断言），并链接 **REQ-ID**。tasks 审计应列出 **含糊动词** 作为 blocker。长任务遗漏往往来自「任务太大」——拆分粒度应小到 **单测可挂**。

📋 **John 产品经理**：任务拆分与验收标准同事写，不后补。

🏗️ **Winston 架构师**：跨模块任务必须拆成 **接口契约先行** 子任务。

💻 **Amelia 开发**：一个 task 对应一个或一组测试文件，不混。

---

### 第 14 轮：审计必须是 blocker 语义

⚔️ **批判性审计员**：若审计输出只有「建议」，没有 **阻断项 + 锚点**，则迭代审计不会改进质量。强制格式：**Blocker（段落/字段）/ Non-blocker / 需人工决策**。若审计 prompt 鼓励「平衡优点缺点」，批判性占比会被稀释——**bmad-speckit 应规定审计模板** 禁止空话。挑战：最近一次 audit 是否阻止了合并？若没有，流程无效。

📋 **John 产品经理**：产品只接受带 blocker 列表的审计。

🏗️ **Winston 架构师**：架构文档 blocker 必须可定位到章节。

💻 **Amelia 开发**：修 blocker 前不改新功能。

---

### 第 15 轮：P0 上限与「无验证命令则不是 P0」

⚔️ **批判性审计员**：若 P0 超过 7 条或没有 **验证命令**，则优先级是假的。流程应规定：**每条 P0 必须绑定 `npm run …` 或等价命令**（可占位但必须存在）。否则 implementation readiness 检查「P0」段落只会检查标题。这是对「最重要 E2E 验收标准」的硬约束：**没有命令 = 没有验收**。

📋 **John 产品经理**：砍 P0 到可验证集合，其余降级。

🏗️ **Winston 架构师**：命令依赖的环境写进 README 与脚本。

💻 **Amelia 开发**：先加空测试与命令，再实现。

---

### 第 16 轮：全局不变量（防长任务漂移）

⚔️ **批判性审计员**：长任务后模型遗漏功能，常因 **缺少不变量**（例如：所有写操作记审计、所有错误有 code）。constitution 或 PRD 必须列出 **5 条以内不变量**，并在 code review 与 E2E 中抽查。若不变量未写，**不要抱怨模型遗漏**。readiness 应检查不变量是否在 **测试** 中有对应断言。

📋 **John 产品经理**：不变量与合规/品牌一致，少而硬。

🏗️ **Winston 架构师**：架构层横切关注点（日志、鉴权）显式列出。

💻 **Amelia 开发**：横切用中间件/装饰器统一，不复制粘贴。

---

### 第 17 轮：Constitution：技术栈与「可测性」约束

⚔️ **批判性审计员**：constitution 若只写语言版本，不写 **E2E 框架与最小覆盖率期望**，则后续无法对齐。应强制：**E2E 工具选择、fixture 策略、禁止无头浏览器与否**。若 speckit 与项目 constitution 冲突，**以可测性为准**。批判性审计员质疑：当前仓库是否 constitution 与 package.json 脚本一致？

📋 **John 产品经理**：技术选型评审包含「测起来多贵」。

🏗️ **Winston 架构师**：选型时评估 **可 mock 性**。

💻 **Amelia 开发**：新依赖需评估 CI 时间。

---

### 第 18 轮：架构：同步/异步边界与 E2E 等待

⚔️ **批判性审计员**：E2E 偶发失败多来自 **异步边界未声明**。架构必须写清：哪些调用是**同步返回**、哪些是**事件最终一致**、轮询间隔与超时。若未写，E2E 会用 `sleep(3000)` 凑，**flaky 即流程失败**。readiness 应检查 plan 是否含 **等待策略**。若 speckit 模板无此段，应改模板。

📋 **John 产品经理**：用户可接受的等待时间写进 PRD。

🏗️ **Winston 架构师**：事件图与序列图对齐。

💻 **Amelia 开发**：用显式等待或同步 API，禁止魔法数字。

---

### 第 19 轮：外部依赖：强依赖降级与 mock 策略

⚔️ **批判性审计员**：若架构不标 **强依赖**，E2E 会在 CI 里连不上第三方。必须：**强依赖在 CI 用 mock/contract；弱依赖可降级**。readiness 若不要求「CI 可跑」，则**本地能跑通 ≠ 流水线能跑通**。批判性审计员要求 bmad-speckit 在 plan 阶段输出 **依赖矩阵**。

📋 **John 产品经理**：采购合同 SLA 与架构降级一致。

🏗️ **Winston 架构师**：每个集成点写 **超时/重试/幂等**。

💻 **Amelia 开发**：契约测试与沙箱二选一。

---

### 第 20 轮：错误模型与用户可见文案

⚔️ **批判性审计员**：若 PRD 不写 **错误与空态**，E2E 只测 happy path 会通过，但生产不可用。强制：**错误码、用户可见文案、可重试性**。readiness 检查「失败」场景是否至少一条 **自动化 E2E**。若 speckit 无「失败旅程」模板，应加。

📋 **John 产品经理**：与 UX 对齐错误文案与帮助链接。

🏗️ **Winston 架构师**：错误模型分层（用户/内部/外部）。

💻 **Amelia 开发**：断言 UI 与 API 错误一致。

---

### 第 21 轮：多环境与数据隔离

⚔️ **批判性审计员**：若 E2E 在 dev 跑通但 stage 挂，**不是测试问题，是流程问题**。架构必须声明 **环境差异**（密钥、数据、feature flag）。readiness 应要求 **smoke 至少在一个类生产环境** 可跑。若全流程只在「开发机」验证，批判性审计员反对标为 ready。

📋 **John 产品经理**：发布 checklist 含环境矩阵。

🏗️ **Winston 架构师**：配置与代码分离，十二因子。

💻 **Amelia 开发**：`.env.example` 与 CI secret 对齐。

---

### 第 22 轮：测试数据与 fixture 生命周期

⚔️ **批判性审计员**：E2E 不稳常因 **fixture 未清理** 或 **共享账号**。流程必须规定：**fixture 创建/清理责任、并行隔离策略**。readiness 检查「测试数据」章节。若 tasks 无「数据准备」任务，则 E2E 会靠运气。

📋 **John 产品经理**：敏感数据用合成，不拷生产。
