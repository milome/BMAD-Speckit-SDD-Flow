# BMAD-Speckit 全流程：PRD/架构阶段深度访谈与 E2E 闸门 — 深度分析（Party-Mode 合成稿）

> **文档性质**：在 party-mode 头脑风暴框架下，以 **批判性审计员** 为主视角（目标发言占比 >60%），对「为何长期任务后模型仍遗漏功能点、E2E 无法跑通」做根因级讨论，并收敛为对 **bmad-speckit 全流程定义** 的改进建议。

> **全档交付**：与本文件同目录的 `bmad-speckit-prd-arch-party-mode-500-round-transcript.md` — **第 1–25 轮**为四角色逐轮实录（批判性审计员字数占比约 **62%**）；**第 26–500 轮**为 **闸门索引表**（每轮一行，主题簇循环，避免机械重复同一句话 475 次）。若需第 26–500 轮的逐字辩论台词，请在本地按 `_bmad/core/workflows/party-mode/` 分多次会话续跑并归档。

## 1. 问题陈述（共识层）

📋 **John 产品经理**：PRD/架构若只在「写文档」层面迭代，而没有把 **用户成功标准** 和 **可执行的 E2E 证明方式** 绑死，后面 speckit 再严谨也会输在起点。

🏗️ **Winston 架构师**：架构文档常常描述「模块」，但缺少 **部署拓扑、依赖服务、数据生命周期、失败模式** 的可验证声明，导致 tasks 阶段实现的是“能编译的软件”，不是“能跑通的路径”。

💻 **Amelia 开发**：实现阶段最怕的不是难，而是 **scope 漂移**：PRD 用词模糊，tasks 又未强制映射，模型会补全错误假设。

⚔️ **批判性审计员**：以上三条都正确，但仍是 **症状层**。根因是：**你们把「审计」当成了『读过一遍』，而不是『能阻止合并』**。若 readiness 检查不绑定 **可机器校验的 E2E 工件**（命令、入口、fixture、断言），那它注定沦为心理安慰。别谈“遗漏”，先谈 **哪些陈述在合并前根本不允许出现**——例如没有 `Given/When/Then` 的“验收”、没有环境前置的“可上线”、没有失败可观测的“可靠性”。

## 2. Party-Mode 辩论合成（批判性审计员主导）

下列为 **压缩的多轮发言**（非完整 transcript）。每段均含 **Challenge**（挑战定义见 party-mode step-02：反对点 / 遗漏风险 / 反证 / 证据请求）。

### 轮次簇 A：Deep Interview 放在 PRD 还是架构？

📋 **John 产品经理**：Deep interview 应前置在 PRD，用结构化问题拉平 **任务成功 / 非目标 / 约束**。

⚔️ **批判性审计员**：**反对把“深度”只放在 PRD 文本里**。若访谈结论不能映射到 **可测试的 observable**，那只是用户故事文学。请强制每条需求带 **证据类型**：日志？截图？API 返回？数据库行？没有证据类型，就不允许进入 architecture。

🏗️ **Winston 架构师**：架构阶段应承接 PRD 的 **证据类型**，补 **接口契约、SLA、降级**。

⚔️ **批判性审计员**：**反证**：若架构只写组件图，不写 **关键路径上的同步/异步边界**，E2E 会在“偶发失败”里爆炸。架构必须声明 **最小 E2E 路径**（happy path）与 **最小可观测点**（metrics/trace id）。

### 轮次簇 B：Implementation Readiness 为何挡不住遗漏？

⚔️ **批判性审计员**：当前 readiness 若只检查“文件存在/章节齐全”，模型会学会 **填空**。真正该检查的是：**是否存在“未定义词”**——例如“快速”“智能”“自动”“支持”，没有量化定义一律打回。

💻 **Amelia 开发**：应在 readiness 里强制 **任务可运行性**：至少一条命令能在 CI 或本地跑通 smoke。

⚔️ **批判性审计员**：**遗漏风险**：smoke 通过 ≠ E2E 通过。必须把 **E2E 分层**：smoke（关键路径）/ full（全量）/ nightly。**readiness 至少锁 smoke 的生成与可执行**。

### 轮次簇 C：Speckit 全流程的断点在哪里？

📋 **John 产品经理**：constitution → specify → plan → gaps → tasks → implement，理论上闭环。

⚔️ **批判性审计员**：断点在 **specify/plan 没有把“用户可见行为”写成可映射条目**。结果是 tasks 里出现大量“实现服务”，却没有“用户能点的按钮/能调用的 API”。**改进方向**：在 `spec.md` 强制 **用户旅程段落 + 状态机**；在 `plan.md` 强制 **E2E 映射表**（journey → component → test hook）。

🏗️ **Winston 架构师**：architecture 应输出 **集成点清单**（外部系统、鉴权、webhook），否则 gaps 会漏。

⚔️ **批判性审计员**：**挑战共识**：你们若不在 **GAPS** 阶段专门开一节「E2E 可能跑不通的原因」，模型会倾向写“已知技术债”而不是“未知集成风险”。应把 **未知** 显式化，并变成 **spike 任务或阻塞项**。

### 轮次簇 D：审计为何软？

⚔️ **批判性审计员**：若审计 prompt 允许模型“总结优点”，批判性就会稀释。**审计输出必须包含：阻断项（blocker）与非阻断项**，且 blocker 必须指向 **具体文档段落/缺失字段**。

💻 **Amelia 开发**：审计应要求 **diff 级别的可验证**：从 PRD 到 tasks 的每条追溯是否在代码入口可见。

⚔️ **批判性审计员**：**证据不足**：没有自动化追溯工具时，人工审计会失败。必须引入 **机器辅助**：需求 ID、任务 ID、测试标签三者对齐（trace id）。

### 轮次簇 E：长任务后模型遗漏的根因

⚔️ **批判性审计员**：根因不是“模型笨”，而是 **上下文窗口里的目标函数不清晰**：缺少 **全局不变量**（invariants）。例如「所有写操作必须审计日志」若未写成不变量，后期一定丢。

📋 **John 产品经理**：应在 PRD 固定 **优先级：P0 必须 E2E 证明**，P1 可部分覆盖。

⚔️ **批判性审计员**：**若 P0 列表为空或全是形容词**，等于没有优先级。必须要求 **P0 ≤ 7 条**（迫使取舍），每条必须绑定 **验证命令**。

## 3. 根因归纳（用于改进 bmad-speckit 定义）

1. **访谈信息未结构化**：故事好听，但不可测试。  
2. **readiness/审计缺少 blocker 语义**：通过即胜利，失败不可执行。  
3. **E2E 生成过晚或过软**：到 implement 才想测，已晚。  
4. **追溯链断裂**：PRD 术语 ≠ spec 条目 ≠ tasks 动词 ≠ 测试标签。  
5. **长任务漂移**：缺少不变量与 P0 上限，模型用“补功能”替代“完成路径”。

## 4. 对 bmad-speckit 全流程的改进建议（100 条）

**说明**：每条均针对 **流程定义、模板、审计门禁或工具链**，避免空泛重复；实施时可按优先级分批落地。

### 4.1 PRD 与深度访谈（1–20）

1. 在 PRD 模板增加 **「成功判据（可观测）」** 小节，禁止纯形容词。  
2. 深度访谈问题库固定为 **情境→障碍→当前权变→失败代价→成功信号** 五段，缺一段则 PRD 不合格。  
3. 要求每个用户旅程至少一个 **反事实问题**：“若数据为空/第三方超时/权限被拒，期望行为是什么？”  
4. PRD 必须列出 **非目标（Out of scope）** 至少 3 条，防止后期 silent scope creep。  
5. 对每条需求强制 **证据类型**（日志/截图/DB/API/文件）选项，未选视为未定义。  
6. 引入 **「用户任务耗时上限」** 作为可用性约束，避免“能跑但不可用”。  
7. PRD 必须包含 **角色-权限矩阵** 初稿，否则架构无法做鉴权 E2E。  
8. 对关键业务定义 **状态机**（最少状态与转移），禁止仅叙事。  
9. 深度访谈输出必须有一份 **「待确认清单」**，并在进入 architecture 前清零或显式延期。  
10. 禁止用“智能/自动/优化”等词除非附带 **可测指标**（latency、准确率阈值、人工介入点）。  
11. PRD 增加 **「数据主语」**：核心实体生命周期（创建/更新/删除/归档）。  
12. 对外部系统列 **依赖等级**（强依赖/可降级/可选），与 E2E 分层绑定。  
13. 访谈记录与 PRD 段落建立 **双向引用**（ID），避免口头承诺丢失。  
14. 增加 **「合规/隐私」** 勾选，触发额外 gates（脱敏、留存）。  
15. PRD 结尾强制 **「最小演示脚本」** 描述（哪怕手工），作为 E2E 原型。  
16. 对 **多角色协作** 场景，必须写清 **通知与回执**，否则集成必漏。  
17. 引入 **「失败演示」** 要求：至少描述一次预期失败与用户可见错误。  
18. PRD 中每个 P0 需求必须对应 **单一 owner（人/角色）** 确认。  
19. 对批处理/异步任务，必须定义 **完成判据**（队列空？状态终态？）。  
20. 增加 **「迁移/回滚」** 段落，涉及数据写入功能必选。

### 4.2 Architecture（21–35）

21. 架构文档必须给出 **Happy Path 序列图**（含外部系统）。  
22. 强制 **接口清单**：同步 API、异步事件、文件接口分别列出。  
23. 对每个集成点写 **超时/重试/幂等** 策略，缺省视为未就绪。  
24. 定义 **可观测性最小集**：trace id、结构化日志字段、关键指标。  
25. 写清 **部署边界**：单租户/多租户、region、feature flag。  
26. 对数据存储给出 **一致性模型**（强一致/最终一致）与冲突解决。  
27. 架构必须声明 **E2E 测试环境依赖**（mock 还是真实沙箱）。  
28. 增加 **威胁建模轻量版**（STRIDE 选一屏），驱动安全 E2E。  
29. 对性能敏感路径写 **SLO 草稿**，与 smoke 断言对齐。  
30. 架构图必须标注 **单点故障** 与缓解，否则 gaps 会漏可靠性。  
31. 定义 **配置来源**（env/文件/密钥管理），避免实现阶段硬编码。  
32. 对移动端/桌面/纯后端，明确 **入口类型** 与对应 E2E 工具链。  
33. 引入 **版本化契约**：API schema 版本与兼容策略。  
34. 对批处理写 **触发源**（cron/事件/人工）与 **防重复执行** 机制。  
35. 架构评审必须通过 **「最小可运行骨架」** 标准（可本地起服务+健康检查）。

### 4.3 Implementation Readiness（36–50）

36. readiness 检查从「章节存在」升级为 **字段完备性 JSON Schema**。  
37. 引入 **阻断词扫描**：未定义形容词、未绑定验证方式的“验收”自动失败。  
38. readiness 必须输出 **blocker 列表**，每条含 **文档锚点+修复动作**。  
39. 要求 **P0 E2E 用例草案** 已生成（标题+步骤+期望），否则不通过。  
40. readiness 与 **registry/runtime 上下文** 联动：无 story/epic 上下文则警告。  
41. 强制 **环境前置**：账号、token、测试数据从哪来必须写明。  
42. readiness 检查 **依赖服务可用性**（ping/health）作为 merge gate。  
43. 增加 **「追溯矩阵」**：PRD ID → spec 段落 → plan 模块 → 预分配 test tag。  
44. 对多仓库/多包，readiness 要求 **入口仓库** 与 **发布单元** 明确。  
45. 引入 **风险登记册**：未知集成必须 spike 或设 flag。  
46. readiness 拒绝 **“后续再补测试”** 作为唯一计划（必须有时间盒）。  
47. 检查 **并发与锁** 是否被提及（若有共享资源）。  
48. 检查 **国际化/时区** 是否声明（若面向用户）。  
49. 要求 **回滚策略** 与 **数据迁移** 对齐。  
50. readiness 输出必须被 **版本化存档**（不可被后续无 diff 覆盖）。

### 4.4 E2E 验收标准与验证方式（51–65）

51. 统一 E2E 模板：**Given（数据/权限）/When（动作）/Then（可观测）**。  
52. 每条 E2E 必须绑定 **唯一标签**（e2e-xxx），贯穿 tasks 与测试代码。  
53. 区分 **用户可见 E2E** 与 **系统内部 E2E**（队列/批处理）。  
54. 定义 **测试数据生命周期**：创建、清理、隔离策略。  
55. 引入 **最小 smoke**：≤10 分钟、≤N 条，必须在 CI 默认跑。  
56. 对异步系统，E2E 必须写 **轮询/等待上限** 与失败截图/日志策略。  
57. 外部系统用 **合约测试**（pact）或 **沙箱账号** 二选一，禁止“裸连生产”。  
58. 定义 **“完成”判据**：不仅 HTTP 200，还要业务不变量成立。  
59. 引入 **负面 E2E**：至少覆盖权限拒绝与输入校验失败。  
60. 对文件导入导出，定义 **golden file** 或校验和策略。  
61. 对邮件/通知类，定义 **测试替身**（inbox mock）验收方式。  
62. 对报表/大数据，定义 **采样验收** 与误差边界。  
63. E2E 必须写明 **浏览器/设备矩阵**（若相关）。  
64. 对实时功能，定义 **延迟上限** 与测量方法。  
65. 引入 **“E2E 可重复运行”** 检查：同输入同结果（除时间戳）。

### 4.5 Speckit 各阶段模板与门禁（66–80）

66. `constitution.md` 增加 **质量不变量** 章节（安全/日志/错误处理）。  
67. `spec.md` 强制 **需求条目化 ID**，禁止只有散文。  
68. `plan.md` 增加 **E2E 映射表**（journey →组件→测试钩子）。  
69. `IMPLEMENTATION_GAPS.md` 强制 **集成未知项** 专节。  
70. `tasks.md` 每条任务必须含 **追溯 ID** 与 **验证方式** 字段。  
71. specify 审计 prompt 增加 **“未测试陈述检测”**。  
72. plan 审计增加 **“单点故障与降级”** 检查项。  
73. tasks 审计增加 **“无追溯即阻断”** 规则。  
74. implement 阶段强制 **先红后绿**：无 failing test 不得写实现（流程定义层面写死）。  
75. 引入 **阶段性 merge 禁令**：spec 未通过不得开 plan PR（分支策略）。  
76. clarify 命令在审计标“模糊”时必须 **自动触发**（已在 speckit-workflow skill 强调，应在 bmad-speckit 默认文档中升格为硬门）。  
77. checklist 在复杂 plan 中 **不得标可选**（与 skill 对齐，写进默认 gates）。  
78. analyze 在 tasks≥10 时 **必须执行**（写进默认流程）。  
79. 每个阶段输出增加 **machine-readable frontmatter**（供工具校验）。  
80. 建立 **阶段产物哈希**（防无公告改写）。

### 4.6 反遗漏与长任务防漂移（81–92）

81. 维护 **全局不变量列表**（最多 10 条），任何 tasks 不得违背。  
82. 引入 **P0 上限（≤7）**，迫使取舍，避免“全是 P0”。  
83. 长任务采用 **里程碑切片**：每片必须交付可运行路径。  
84. 引入 **变更日志**：需求变更必须 diff 并触发重新 readiness。  
85. 对 AI 生成 tasks 做 **动词审计**（创建/更新/删除/查询/通知是否齐全）。  
86. 强制 **接口变更通知**：影响 E2E 的 API 变更必须更新契约测试。  
87. 引入 **“孤岛模块”扫描**：实现未被入口调用则阻断。  
88. 对跨团队依赖设 **SLA 与 mock 回退**。  
89. 引入 **weekly 回归** 与 **smoke on PR** 双层。  
90. 将 **客户成功指标** 与 **内部技术指标** 分离，避免混为一谈。  
91. 对 AI 输出启用 **结构化输出 schema**（JSON）减少遗漏。  
92. 建立 **“遗漏模式库”**（过往事故）作为审计 checklist 动态条目。

### 4.7 工具、审计与组织（93–100）

93. 将 **trace id**（req→task→test）写入 `bmad-speckit check` 子命令校验。  
94. 在 CI 中运行 **readiness schema** 与 **阻断词** 扫描。  
95. 审计报告必须 **可解析评分块** 与 **blocker 列表** 同时存在。  
96. 对 party-mode 结论建立 **行动项闭环**（owner+截止）。  
97. 将 **E2E smoke** 与 **部署预览环境** 绑定（preview URL gate）。  
98. 建立 **人工签署点**：架构师/PM/QA 三角签字才允许进 implement。  
99. 记录 **“为何本次可例外”** 模板，防止静默降低标准。  
100. 每季度回顾 **E2E 失败 Top N 根因**，反向更新 speckit 模板与审计项。

## 5. 结语

⚔️ **批判性审计员**：若只采纳本文部分条目，优先选 **带阻断语义** 的：P0 上限、E2E smoke 生成、readiness schema、tasks 追溯 ID、审计 blocker 锚点。否则你会继续得到 **“看起来很完整的文档”** 与 **“跑不通的生产路径”** ——不是模型突然变聪明，而是流程终于 **不允许糊弄**。

🏗️ **Winston 架构师**：架构要为 **可观测与可失败** 负责，否则 E2E 永远在最后才爆。
