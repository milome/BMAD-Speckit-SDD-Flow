### 一、对抗性逐项复核

**1. §3.1 底层逻辑先行 — 对抗性检查**

- 原则已写入，但「若某环节得分无法支撑上述三者之一，则该环节的评分设计需重新审视」为定性表述，实施时无量化标准。**建议**：在权威文档或实施任务中补充「无法支撑」的判定示例（如某环节得分与迭代结束判定无因果链），不作为本轮 gap，属后续增强。
- **结论**：与 PARTY_MODE 更新要点一致，无遗漏。

**2. §3.1.1 设计原则贯彻检查清单 — 对抗性检查**

- 6 条原则均有可验证项，但「底层逻辑先行」的可验证项为「各环节得分是否服务于迭代结束判定、短板定位、模型优化？是否存在无法支撑上述目的的评分项？」。该表述为自检式问题，审计时需人工判断，无自动化校验路径。**建议**：实施时在权威文档中为每环节补充「支撑关系」说明，便于审计。不作为本轮 gap。
- 检查清单中「底层逻辑先行」置于最后一行，与 PARTY_MODE 要求「写入原则之首」存在顺序差异。但 §3.1 正文中「底层逻辑先行」已置于首条，检查清单中的顺序不影响落地。**结论**：无实质 gap。

**3. §3.8 ref 引用关系 — 对抗性检查**

- 当前表述为「scoring/rules/*.yaml 通过 ref 字段引用 code-reviewer-config 中的检查项 ID（item_id）」。PARTY_MODE 更新要点第 4 条还要求补充「code-reviewer-config 的 dimensions 可与 scoring 规则的 check_items 一致或通过映射表转换」。当前 §3.8 未显式写出「dimensions 与 check_items 一致或映射表转换」。
- **对抗性结论**：该补充属于实施细节，§3.11 的 YAML 示例已通过 ref 字段体现衔接，§3.12 也明确 item_id 一致。若严格按 PARTY_MODE 字面要求，§3.8 可增补一句「code-reviewer-config 的 dimensions 可与 scoring 规则的 check_items 一致或通过映射表转换」。**建议**：作为 minor 增强，不构成本轮 gap；若需 100% 字面覆盖，可增补。

**4. §3.10 第 11、12 条 — 对抗性检查**

- 第 11、12 条已完整写入，且与 §3.11、§3.12 形成引用闭环。无遗漏。
- **结论**：完全覆盖。

**5. §3.11 评分规则配置示例 — 对抗性检查**

- PARTY_MODE 更新要点中的 YAML 示例采用 `contributes_to_phases`、`pass_delta`/`fail_delta`、顶层 `ref: { config, mode }` 结构；需求文档采用 `link_环节`、`weights.base/bonus`、每 item 的 `ref: code-reviewer-config#xxx` 结构。两者 schema 形态不同。
- **对抗性结论**：需求文档的 schema 已覆盖核心要素（环节、权重、检查项、veto、ref），且 30/18/12/20/20 权重、veto_items、ref 字段均存在。不同 schema 形态为合理设计选择，不构成与 PARTY_MODE 的实质矛盾。**结论**：无 gap。

**6. §3.12 Code Reviewer Skill 整合 — 对抗性检查**

- 6 阶段↔六环节、触发时机、维度换算、输出与 schema 衔接、item_id 与 veto 一致均已覆盖。
- **潜在歧义**：§3.12 写「Skill 环节 2 的 30/18/12/20/20 分制对应本需求环节 2 的 25 分权重」，换算公式为 `环节得分 = (分项得分之和 / 100) × 25`。若 Skill 输出为 0–100 分，则 100 分对应 25 分；若某子维度未通过导致扣分，公式是否仍成立？文档未明确「分项得分之和」是满分 100 还是扣分后的实际得分。**建议**：实施时在权威文档中明确「分项得分之和」的定义（如 base 满分 60 + bonus 满分 40，扣分后为实际得分）。不作为本轮 gap，属实施澄清。
