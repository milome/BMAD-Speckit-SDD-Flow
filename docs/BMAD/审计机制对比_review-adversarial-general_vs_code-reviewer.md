# 审计机制对比：review-adversarial-general vs code-reviewer

**目的**：分析两者谁更适合作为「审计闭环」的主机制。  
**结论摘要**：**code-reviewer 更适合做审计闭环**；review-adversarial-general 适合做**初稿/修订稿的压力测试**（尽量挑问题），不适合作为「通过即可退出」的闭环判断依据。

---

## 一、两者定义与来源

| 项目 | review-adversarial-general | code-reviewer |
|------|----------------------------|----------------|
| **类型** | BMAD 任务（XML） | Claude/Cursor Agent（Markdown） |
| **路径** | `_bmad/core/tasks/review-adversarial-general.xml` | `.claude/agents/code-reviewer.md` 或 `.cursor/agents/code-reviewer.md` |
| **触发方式** | 通过 workflow/task 调用，传入待审内容 | 通过 Cursor Task / 子代理调用，传入**审计提示词（含检查清单）** + 待审内容 |

---

## 二、设计目标与输出

### 2.1 review-adversarial-general

- **目标**：Cynically review content and produce findings（以挑剔态度审查内容并产出发现项）。
- **人设**：Cynical, jaded reviewer, zero patience for sloppy work；假定内容有问题；关注「缺了什么」而不只是「错了什么」。
- **流程**：  
  1. 接收待审内容  
  2. 对抗性分析：**至少找出 10 个**可改进或需修复的问题  
  3. 以 Markdown 列表形式输出发现（仅描述）
- **终止条件**：  
  - 若发现数为 0 → **HALT**（视为可疑，要求重新分析或请示）  
  - 内容为空或不可读 → HALT
- **输出**：  
  - **没有**「通过 / 不通过」的结论  
  - **没有**「完全覆盖、验证通过」之类闭环标准  
  - 只有「发现项列表」

因此：该任务**天然不会给出「可以收口」的信号**；每次运行都倾向于给出至少 10 条发现，且 0 发现会被视为异常而 HALT，无法用作「通过即退出」的判据。

### 2.2 code-reviewer

- **目标**：Strict code/document audits；在实现 audit-prompts §5、验证已完成任务、核对 Story/TASKS/文档与需求时使用。
- **行为**：  
  1. **加载上下文中的审计提示词**（如 audit-prompts.md §5 或等价的 BUGFIX 审计模板）  
  2. **按检查清单逐项**对照标准验证  
  3. **在适用时执行验证命令**（pytest、grep 等）  
  4. 明确报告：通过项、未通过项、具体修改建议
- **输出格式**：  
  - 报告结尾**必须**给出明确结论：**「完全覆盖、验证通过」**，**或**列出未通过项及修改建议  
  - 不采信口头声称，以清单和（若适用）执行结果为准

因此：code-reviewer 的输出**直接支持闭环**——结论为「完全覆盖、验证通过」即可退出；否则按未通过项修改后再次调用，直到通过。

---

## 三、与「审计闭环」的匹配度

**审计闭环**在此指：  
- 有**明确的通过标准**（例如「完全覆盖、验证通过」或必达子项全部满足）  
- 审计结果有**通过 / 未通过**的二元结论  
- 未通过时给出**可操作的修改建议**，修改后可**再次审计**，直到通过才结束  

| 维度 | review-adversarial-general | code-reviewer |
|------|----------------------------|----------------|
| **通过标准** | 无；任务不定义「通过」，且要求至少 10 条发现 | 有；结论即「完全覆盖、验证通过」或未通过项列表 |
| **能否「通过即退出」** | 不能；0 发现会 HALT，且任务本身无「通过」状态 | 能；报告结论为通过即可结束迭代 |
| **检查依据** | 无固定清单；依赖审查者的通用怀疑 | 有；依赖传入的审计提示词（含必达子项/清单） |
| **可重复性** | 同一文档多次运行，可能得到不同「10 条」发现，标准不固定 | 同一审计提示词 + 同一文档，结论与清单一致，可重复 |
| **与 bmad-bug-assistant 的契合** | 技能要求「迭代至审计结论为完全覆盖、验证通过」，本任务不产出该结论 | 技能明确「优先 code-reviewer」且审计模板要求报告结尾给出该结论，完全契合 |
| **与 speckit-workflow 的契合** | speckit 要求「仅在审计报告结论为完全覆盖、验证通过时可结束」；本任务不产出该结论 | 与 speckit 的 code-review + audit-prompts §1–§4 设计一致，产出该结论 |

结论：**只有 code-reviewer 在设计和输出上支持「通过即退出」的审计闭环**；review-adversarial-general 不提供通过标准，也不产出闭环结论。

## 四、各自更适合的场景

### 4.1 review-adversarial-general 更适合

- **初稿或修订稿的压力测试**：希望「尽量多挑问题」、不预设条数上限时（任务强制至少 10 条，有利于深挖）。
- **无固定检查清单的泛化审查**：例如对一篇说明文档、一份改进任务列表做「能想到的都挑一挑」。
- **与闭环分离使用**：先跑一遍 adversarial 拿到一批改进点，修完后再用**另一套机制**（如 code-reviewer + 审计提示词）做正式闭环，而不是用 adversarial 的「发现数」判断是否通过。

### 4.2 code-reviewer 更适合

- **所有需要「审计闭环」的场景**：  
  - BUGFIX 文档审计（阶段一/二/四实施后）  
  - speckit 的 spec/plan/GAPS/tasks 审计（audit-prompts §1–§4）  
  - Story/TASKS/文档与需求的符合性验证  
- **有明确检查清单的审计**：清单来自审计提示词（如 BUGFIX 阶段一审计模板的 6 项必达子项），code-reviewer 按清单逐项验证并给出通过/未通过。
- **需要可重复、可自动化判断的结果**：同一提示词 + 同一文档，结论应一致，便于脚本或流程在「完全覆盖、验证通过」时自动结束迭代。

---

## 五、组合使用建议

- **审计闭环（主机制）**：  
  - 使用 **code-reviewer** + **对应审计提示词**（如 bmad-bug-assistant 内阶段一审计完整 prompt 模板、或 speckit 的 audit-prompts §1–§4）。  
  - 迭代至报告结论为「**完全覆盖、验证通过**」后退出。
- **可选增强（不替代闭环）**：  
  - 在重要文档（如改进任务列表、PRD、高影响 BUGFIX）定稿前，可**额外**跑一次 **review-adversarial-general**，利用其「至少 10 条、偏挑剔」的特性做压力测试。  
  - 将 adversarial 的发现作为改进输入，修完后仍由 **code-reviewer + 审计提示词** 做最终闭环判断。

这样既保留「通过即退出」的清晰闭环，又能在需要时用 adversarial 多挖一轮问题。

## 六、对照表小结

| 对比项 | review-adversarial-general | code-reviewer |
|--------|----------------------------|----------------|
| 产出结论 | 仅发现列表，无通过/不通过 | 「完全覆盖、验证通过」或未通过项+建议 |
| 通过即可退出 | 否（且 0 发现会 HALT） | 是 |
| 检查依据 | 无固定清单，通用怀疑 | 审计提示词中的清单/必达子项 |
| 可重复性 | 标准不固定，较难复现 | 同一提示词可重复 |
| 适用主场景 | 压力测试、泛化挑问题 | 审计闭环、需求符合性验证 |
| 与 bmad-bug-assistant | 不产出技能要求的结论 | 直接满足「迭代至完全覆盖、验证通过」 |
| 与 speckit-workflow | 不满足「仅结论通过时可结束」 | 满足 |

**最终建议**：以 **code-reviewer** 作为审计闭环的主机制；**review-adversarial-general** 作为可选的补充压力测试，不用于判断「是否可以退出闭环」。
