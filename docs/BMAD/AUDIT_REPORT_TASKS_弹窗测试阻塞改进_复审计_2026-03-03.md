## §3 批判性审计员多轮质疑与收敛

### 第一轮：24 个 Gap 均闭合，是否还有新 gap？

**批判性审计员**：逐项核对后，24 个 Gap 均已闭合。再检查以下潜在问题：

1. **§4.2.4 与 T11 的衔接**：§4.2.4 写「可选完整流水线：每日/每周或手动触发 `pytest tests/`」。但 addopts 含 `-m "not qt_dialog"` 后，裸 `pytest tests/` 会排除 qt_dialog，无法跑「完整全量」。此处「`pytest tests/`」表述是否与 T9 冲突？

2. **T7 涉及文件清单完整性**：T7 列出 30+ 文件，但 `chart/test_compatibility_multi_timeframe.py` 在 §1.2 表格中，是否在 T7 涉及文件清单中？需核对。

3. **V2 与「本地有显示器」的验收环境**：V2 要求「本地有显示器环境下」跑完。若验收者在 CI 无头环境执行，V2 是否应跳过？文档未说明 V2 的验收环境判定。

**审计员**：第 1 点构成潜在歧义；第 2 点需核对；第 3 点属验收流程细节，可接受。

### 第二轮：§4.2.4 歧义是否构成新 Gap？

**批判性审计员**：§4.2.4「可选完整流水线：每日/每周或手动触发 `pytest tests/`」中的「`pytest tests/`」在 addopts 修改后实际为「默认全量（排除弹窗）」，而非「完整全量」。若理解为「完整回归」，则与 T9 两阶段跑法不一致。**建议**：将 §4.2.4 改为「可选完整流水线：每日/每周或手动触发**两阶段**（`pytest tests/ -m "not qt_dialog"` + `pytest tests/ -m qt_dialog`）」，或「用于完整回归（见 T10 文档）」。

**审计员**：该歧义可能导致执行者误用裸 `pytest tests/` 作为完整回归。**Gap-25（新）**：§4.2.4「可选完整流水线」的「`pytest tests/`」与 addopts 修改后的实际行为不一致，建议改为两阶段或引用 T10。
