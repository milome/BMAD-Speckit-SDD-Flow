## §1 问题描述

### 1.1 现象

执行 `pytest tests/` 全量测试时，部分涉及 Qt 弹窗/模态对话框的测试会：

- **阻塞**：测试挂起，无法完成
- **Access Violation**：Windows 下偶发崩溃
- **全量测试无法完成**：阻塞或崩溃导致 CI/本地全量回归失败

### 1.2 影响范围

| 测试文件/模块 | 涉及弹窗/模态 | 典型用例 |
|---------------|---------------|----------|
| `test_offline_task_manager_ui.py` | QMessageBox.warning | 提交校验、删除确认、非法范围提示 |
| `test_widget_trading.py` | QMessageBox（已 mock） | 交易 UI 方向/偏移 |
| `test_chart_phase3_4.py` | qtbot + 对话框 | 方向偏移、HKFE 选项 |
| `test_widget_calendar.py` | HolidayDetailDialog.exec() | 日历假日详情 |
| `test_mainwindow_tab_position.py` | MainWindow + qapp | 停靠区域位置 |
| `chart/test_chart_window_mode_switch.py` | ChartWindow + qtbot | 单周期↔多周期模式切换、离线任务按钮 |
| `chart/test_compatibility_multi_timeframe.py` | ChartWindow + MultiTimeframeWidget | 多周期图表启动与兼容性 |
| `test_datamanager_table_models.py` | 自定义 qapp | 表格模型 |
| 其他 chart/ui 测试 | 各类 qapp/qtbot | 以 `grep -r "def qapp\|QApplication.instance" tests/` 输出为准，详见 T7 |

> **图形界面测试**：包括 ChartWindow、多周期图表（MultiTimeframeWidget）等，同样受 QApplication/qtbot 冲突影响，已纳入 T7 审查范围。阻塞场景包括：qtbot 与自定义 qapp 冲突、对话框 exec 阻塞事件循环、无头环境下 access violation 等。

### 1.3 约束

- 必须保持现有测试语义与覆盖率
- 必须兼容 CI 无头环境（无显示器）：默认流水线 `pytest tests/ -m "not qt_dialog"` 在无头下可跑；完整全量（含 qt_dialog）在无头下不保证通过，V2 仅验证「本地有显示器」场景。
- 不得引入新的 flaky 或阻塞风险

### 1.4 前置条件

- 项目已安装 `pytest-qt`（§2 根因分析及改进方案依赖 pytest-qt 的 qapp/qtbot）。

## §2 根因分析

### 2.1 阻塞的直接原因

| 根因类别 | 说明 | 证据/依据 |
|----------|------|-----------|
| **QApplication 未初始化或重复创建** | 部分测试在模块级 `QApplication.instance() or QApplication(sys.argv)`，与 pytest-qt 的 qapp 冲突或顺序错乱 | `test_widget_calendar.py` L16、`test_widget_refactor_baseline.py` L18 |
| **模态对话框 exec() 阻塞事件循环** | `QMessageBox.warning()`、`QDialog.exec()`、QFileDialog、QColorDialog 等会启动嵌套事件循环，若未 mock 或未在 qtbot 管理下，会阻塞 pytest；本改进以 QMessageBox/QDialog 为主，其他模态框可后续迭代 | `vnpy/chart/kline_style_dialog.py` L131、`vnpy/trader/ui/mainwindow.py` L594 |
| **qtbot/qapp 使用不一致** | 有的用 pytest-qt 的 qtbot，有的自定义 qapp，scope 混用（function/session/module）导致 QApplication 生命周期混乱 | `test_mainwindow_tab_position.py` 自定义 `qapp(qtbot)` |
| **子进程/多进程与 Qt 冲突** | OfflineTaskManagerUI 设计上在 ComputeClient 子进程运行，测试中若误触发子进程或 IPC，与主进程 Qt 冲突；通过 mock 弹窗或避免实际启动子进程可规避 | BUGFIX_离线任务窗口首次打开闪烁.md |
| **无头环境差异** | CI 使用 `QT_QPA_PLATFORM=offscreen` 或类似，部分平台/驱动下 Qt 行为不同，易触发 access violation；排除 qt_dialog 后剩余 Qt 测试在无头下预期可跑，若仍遇 access violation 需单独排查并可能打 qt_dialog 或进一步隔离 | pytest-qt 文档、Qt 无头模式限制 |

### 2.2 批判性审计员质疑与澄清

- **Q1**: 弹窗阻塞是 QApplication 未初始化，还是事件循环问题？  
  **结论**: 两者皆有。未初始化会导致创建 widget 失败或异常；事件循环问题则导致 exec() 后无法返回。

- **Q2**: Mock 弹窗是否足够？会不会漏测真实交互？  
  **结论**: 对「业务逻辑分支」（如校验失败弹提示）mock 足够；对「弹窗 UI 布局/样式」需隔离运行，不纳入全量。

- **Q3**: 为何不全部 mock，而要隔离？  
  **结论**: 全部 mock 会丢失对 Qt 集成（如 qtbot.addWidget、waitExposed）的验证；部分用例需真实 widget 才能测布局/可见性。

- **Q4**: pytest-qt 的 qapp 是 function scope，多测试共享会有问题吗？  
  **结论**: pytest-qt 4.5.0 的 qapp 默认 function scope，每个测试新建；问题在于**自定义 qapp 覆盖或绕过**了 pytest-qt，导致冲突。

- **Q5**: 无头环境下 QMessageBox 会怎样？  
  **结论**: 通常仍会创建，但可能无窗口句柄，部分平台会阻塞或崩溃；mock 可彻底避免。

## §3 方案对比与选型

### 3.1 方案概览

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 全量 Mock 弹窗** | 所有 QMessageBox/QDialog 在测试中 mock | 无阻塞、CI 稳定 | 丢失弹窗触发路径的集成验证 |
| **B. 标记隔离** | 弹窗相关测试打 `@pytest.mark.qt_dialog`，默认不跑，需显式 `-m qt_dialog` | 全量不阻塞，需要时单独跑 | 全量覆盖率下降，易遗忘 |
| **C. 分层策略** | 业务逻辑 mock 弹窗；UI 集成测试隔离并可选 | 平衡覆盖与稳定性 | 实施复杂度中等 |
| **D. 子进程完全隔离** | 弹窗测试在独立 pytest 进程跑 | 彻底隔离 | 维护成本高，CI 配置复杂 |

### 3.2 选型结论

**采用方案 C（分层策略）**，理由：

1. **业务逻辑优先 mock**：QMessageBox.warning/critical/information 等「提示类」弹窗，测试关心的是「是否调用、参数是否正确」，mock 即可，无需真实弹窗。
2. **UI 集成测试隔离**：涉及 `dialog.exec()`、真实窗口 show/raise 的用例，打 `@pytest.mark.qt_dialog`，全量默认排除，CI 可选跑。
3. **统一 qapp/qtbot**：移除自定义 qapp，统一使用 pytest-qt 的 qtbot/qapp，避免生命周期冲突。
4. **可演进**：若后续需更多 UI 自动化，可在此基础上扩展，无需推翻重来。

### 3.3 批判性审计员对选型的挑战与回应

- **挑战**: 方案 C 的「分层」边界模糊，谁来决定 mock 还是隔离？  
  **回应**: 建立明确规则——凡测试中会触发 `QMessageBox.*` 或 `QDialog.exec()` 的代码路径，一律 mock；凡需验证「窗口显示、布局、用户点击对话框按钮」的，一律隔离。**「窗口显示」指 QDialog/QMessageBox 等模态对话框的显示，不含 ChartWindow 等主窗口的 show()**。若某用例既触发业务逻辑又验证布局，优先 mock 业务逻辑，若同一用例需验证布局则打 qt_dialog 隔离；或拆分为两个用例（一个 mock 逻辑，一个隔离验证布局）。

- **挑战**: 隔离后全量不跑，回归会漏？  
  **回应**: CI 配置两套：`pytest tests/ -m "not qt_dialog"` 为默认快速流水线；`pytest tests/ -m qt_dialog` 或 `pytest tests/` 为完整流水线，可定时/手动触发。

## §4 改进方案

### 4.1 架构原则

1. **单一 QApplication 来源**：仅使用 pytest-qt 提供的 qapp/qtbot，禁止模块级 `QApplication.instance() or QApplication(sys.argv)`。
2. **弹窗 Mock 优先**：凡测试会触发 QMessageBox/QDialog 的，在用例或 conftest 中 mock。
3. **隔离标记**：需真实弹窗的用例打 `@pytest.mark.qt_dialog`，全量默认排除。
4. **无头兼容**：所有 Qt 测试在 `QT_QPA_PLATFORM=offscreen` 下可运行，不依赖显示器。

### 4.2 具体措施

#### 4.2.1 conftest 统一

- 在 `tests/conftest.py` 中确保 pytest-qt 的 qapp/qtbot 可用（已通过 `pytest-qt` 插件提供）。
- 新增 `mock_message_boxes` fixture：自动 mock `QMessageBox.warning`、`critical`、`information`、`question`，记录调用参数，供断言使用。若测试会触发 QFileDialog、QColorDialog、QInputDialog 等，采用相同策略（mock 或 qt_dialog 隔离）；本改进范围以 QMessageBox/QDialog 为主，其他模态框可后续迭代。
- 新增 `pytest.mark.qt_dialog` 标记，在 `pytest.ini` 中注册。

#### 4.2.2 测试改造

- **test_offline_task_manager_ui.py**：已部分 monkeypatch QMessageBox.warning，统一改为使用 `mock_message_boxes` fixture；涉及 `ui.qt_widget.show()` 且可能触发弹窗的，确保 mock 覆盖所有分支。
- **test_widget_calendar.py**：移除模块级 QApplication 创建；若测试 HolidayDetailDialog，改为 mock `dialog.exec()` 或打 `qt_dialog` 隔离。
- **test_widget_refactor_baseline.py**：移除模块级 QApplication，改用 qtbot。
- **test_mainwindow_tab_position.py**：移除自定义 `qapp(qtbot)`，直接使用 pytest-qt 的 qapp/qtbot。
- **其他**：逐个审查使用 `qapp`/`qtbot` 的测试，删除自定义 qapp 定义，统一使用插件提供。详见 T7 涉及文件清单。

#### 4.2.3 pytest 配置

```ini
# pytest.ini 新增
markers =
    qt_dialog: Tests that require real modal dialogs (excluded from default run)
```

```bash
# 默认全量（排除弹窗）：addopts 含 -m "not qt_dialog" 后，pytest tests/ 即为此
pytest tests/ -m "not qt_dialog"

# 完整全量（含弹窗）：两阶段组合，不依赖裸 pytest tests/
pytest tests/ -m "not qt_dialog"   # 阶段1
pytest tests/ -m qt_dialog         # 阶段2
```

#### 4.2.4 CI 配置

- 默认流水线：`pytest tests/ -m "not qt_dialog"`，确保不阻塞。
- 可选完整流水线：每日/每周或手动触发**两阶段**（`pytest tests/ -m "not qt_dialog"` + `pytest tests/ -m qt_dialog`），用于完整回归；见 T10 文档。
