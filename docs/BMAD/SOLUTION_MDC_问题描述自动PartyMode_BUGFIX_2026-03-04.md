## 2. 触发条件

### 2.1 在 MDC 中如何定义「用户描述问题」

- **语义**：用户消息以**问题/错误/异常**为核心意图（报告现象、请求排查、描述失败），而非单纯功能需求、文档请求或闲聊。
- **推荐触发方式**（MDC 无法做真正的「if 用户消息 then」分支，只能通过描述 + 优先级约束行为）：
  - 在规则正文中写清：**当用户消息满足下列任一情况时，本规则适用且主 Agent 必须按本规则执行**；
  - 下列情况用「用户描述/提到/包含」等可被模型解释的表述。

### 2.2 触发条件（关键词与场景）

**满足任一即触发**：

- 用户消息中**包含**与错误/失败相关的关键词或短语，例如：
  - 测试失败：`测试失败`、`test failed`、`pytest 报错`、`用例失败`、`assertion error`、`AssertionError`、`24 个失败` 等；
  - 运行错误：`运行报错`、`运行错误`、`报错`、`exception`、`Exception`、`traceback`、`崩溃`、`闪退` 等；
  - 异常现象：`不对`、`不符合预期`、`和预期不一致`、`bug`、`有问题`、`异常`、`根因`、`为什么失败` 等。
- 用户消息**描述**了以下任一场景（可用自然语言概括）：
  - 某条或某批测试用例失败，并可能附带错误信息、文件路径或用例名；
  - 程序运行时报错或异常，并可能附带堆栈、日志片段；
  - 行为与预期不符，并可能附带复现步骤、环境信息；
  - 明确请求「分析根因」「写 BUGFIX」「修复这个 bug」等。

### 2.3 反例（不触发「自动 party-mode」）

- 用户仅说「用 party-mode」且未描述具体问题 → 可走通用 party-mode，但不强制本规则的 100 轮 + 批判审计员 >60% + 必出 BUGFIX；
- 用户仅请求「写文档」「解释一下 X」「帮我写个 README」且未涉及错误/失败 → 不触发；
- 用户明确说「不要 party-mode」「直接修」→ 以用户明确指令为准，不强制自动 party-mode（可在 MDC 中写「除用户明确拒绝 party-mode 外」）。

### 2.4 与「手动 party-mode」的区分（产品视角）

- **自动 party-mode**：由本 MDC 触发；用户只描述问题即可，主 Agent 自动进入 party-mode，并遵守 100 轮、批判审计员 >60%、收敛、产出 BUGFIX、禁止直接改代码。
- **手动 party-mode**：用户主动说「用 party-mode」「开 party-mode」等；可沿用既有 party-mode workflow，轮次与产出按 step-02 的「决策/根因」场景（100 轮 + 收敛），但不强制与本 MDC 的 BUGFIX 产出格式绑定，除非同时满足「用户描述问题」的触发条件。

## 3. MDC 规则设计

### 3.1 规则文件建议路径

- **推荐**：`.cursor/rules/bmad-bug-auto-party-mode.mdc`
- **备选**：若希望与现有 bug 规则合并，可放在 `.cursor/rules/bmad-bug-assistant.mdc` 中作为新增段落（需在 description/globs 中区分「自检后发起」与「问题描述时自动 party-mode」两种情境）。

### 3.2 规则内容要点（结构化）

1. **适用时机**
   - 当用户消息属于「描述问题」（见上文触发条件）时，本规则适用。
   - 主 Agent **不得**在未进入 party-mode 的情况下直接给出根因结论或直接修改生产代码；必须先进入 party-mode，满足轮次与收敛后再产出 BUGFIX。

2. **进入 party-mode**
   - 主 Agent 在识别到「用户描述问题」后，**第一条实质性回复**应声明进入 party-mode（或等效表述），并说明将进行多角色辩论（含批判审计员、Winston、Amelia、John 等），目标为根因分析与 BUGFIX 文档。
   - 执行时遵循 `{project-root}/_bmad/core/workflows/party-mode/workflow.md` 及 step-02 的讨论编排；**必须**引入「批判审计员」角色（或批判性审计员，与 agent-manifest 一致）。

3. **角色与发言占比**
   - 必须包含角色：**批判审计员**、**Winston（架构师）**、**Amelia（开发）**、**John（产品）**；可包含其他 BMAD 角色。
   - **硬性约束**：批判审计员发言占比 **> 60%**（即总轮数 100 时，至少 61 轮含批判审计员发言）。
   - 批判审计员职责：质疑触发条件可操作性、MDC 被忽略风险、收敛条件可验证性、避免「假 100 轮」、提出 gap 与边界情况。

4. **轮次与收敛**
   - 讨论 **至少 100 轮**；对问题做根因级深入分析。
   - **收敛条件**：**最后 3 轮无新 gap** 方可结束辩论（即第 98、99、100 轮均无新 gap，或等价表述）。
   - 禁止「假 100 轮」：每轮须有实质角色发言与内容，不得为凑轮次而空转。

5. **产出 BUGFIX 文档**
   - 辩论收敛后，**必须**生成一份 BUGFIX 文档。
   - **格式/必备章节**：§1 现象或问题描述、§2 根因分析、§3 依据/参考（可选）、§4 修复方案、§5 流程/建议流程（可选）、§7 最终任务列表（TASKS）；与现有 bmad-bug-assistant 产出格式对齐。
   - **路径**：有 story 时 `_bmad-output/implementation-artifacts/{epic}-{story}-{slug}/BUGFIX_{slug}.md`；无 story 时 `_bmad-output/implementation-artifacts/_orphan/BUGFIX_{slug}.md`；slug 可由问题简短描述生成。

6. **禁止主 Agent 直接修改源代码**
   - 主 Agent **不得**在会话中直接编辑生产代码以「修 bug」。
   - 修复实施**须**通过以下之一完成：**子代理**（如 mcp_task）、或 BUGFIX 文档 §7 任务列表由**用户或子代理**按列表执行；主 Agent 仅负责发起 party-mode、产出 BUGFIX、发起审计/实施子任务。

### 3.3 伪代码（便于模型执行）

```
IF 用户消息 满足 「描述问题」 的触发条件 THEN
  主 Agent 首条回复 := 声明进入 party-mode，并说明将进行多角色辩论（目标：根因 + BUGFIX）
  执行 party-mode workflow（step-01 初始化，step-02 讨论编排）
  在 step-02 中：
    每轮选择 2–3 名角色发言，其中 批判审计员 必须满足 总发言轮数 > 0.6 * 总轮数
    总轮数 >= 100
    WHILE 最后 3 轮 存在新 gap DO 继续讨论
    收敛后 生成 BUGFIX 文档（路径与格式见上），并禁止主 Agent 直接改生产代码
END IF
```

## 5. 验收标准

如何验证「主 Agent 在用户描述错误时自动发起 party-mode，且批判审计员 >60%、100 轮、最后 3 轮无 gap、产出 BUGFIX、未直接改代码」：

| 验收项 | 验证方法 |
|--------|----------|
| 自动发起 party-mode | 用仅包含「问题描述」的提示（如「跑 pytest 有 24 个用例失败，见 xxx」）开启新会话，检查主 Agent 首条实质性回复是否声明进入 party-mode 并开始多角色讨论，且未要求用户再说「用 party-mode」。 |
| 批判审计员 >60% | 在辩论记录或产出文档中统计：标注为「批判审计员」的发言轮数 / 总轮数 ≥ 0.61。 |
| 至少 100 轮 | 辩论记录或产出文档中总轮数 ≥ 100。 |
| 最后 3 轮无新 gap | 产出文档或辩论结尾明确写出「第 98、99、100 轮无新 gap」或「最后 3 轮无新 gap」或等效表述。 |
| 产出 BUGFIX 文档 | 存在 BUGFIX 文档文件，且包含 §1 现象/问题、§2 根因、§4 方案、§7 任务列表；路径符合约定。 |
| 未直接改代码 | 会话中主 Agent 未对生产代码文件（如 `vnpy/**/*.py`、`vnpy_*/**/*.py`）进行编辑；若有修复，应通过「将任务写入 §7」或「发起 mcp_task 子代理」完成。 |

# 问题描述时自动 party-mode 与 BUGFIX 产出

## 适用时机

当用户消息属于「**描述问题**」时，本规则适用，主 Agent **必须**按下列行为执行，且**不得**在未进入 party-mode 的情况下直接给出根因结论或直接修改生产代码。

**「描述问题」** 指满足以下任一情况：

- 用户消息中包含与错误/失败相关的关键词或短语，例如：测试失败、test failed、pytest 报错、用例失败、AssertionError、运行报错、exception、traceback、崩溃、bug、有问题、异常、根因、为什么失败 等；
- 用户消息描述了以下场景之一：某条或某批测试用例失败（可带错误信息/路径/用例名）；程序运行时报错或异常（可带堆栈/日志）；行为与预期不符（可带复现步骤/环境）；或明确请求「分析根因」「写 BUGFIX」「修复这个 bug」。

**不触发**：用户仅说「用 party-mode」且未描述具体问题；仅请求写文档/解释/README 且未涉及错误；或用户**明确**说「不要 party-mode」「直接修」时，以用户明确指令为准。

## 必须执行的行为

1. **自动发起 party-mode**  
   主 Agent 在识别到「用户描述问题」后，**第一条实质性回复**必须声明进入 party-mode（或等效表述），并说明将进行多角色辩论（目标：根因分析 + BUGFIX 文档）。不等待用户再说「用 party-mode」。

2. **执行 party-mode 与角色约束**  
   - 执行 `{project-root}/_bmad/core/workflows/party-mode/workflow.md` 及 step-02 讨论编排。  
   - 必须引入角色：**批判审计员**、**Winston（架构师）**、**Amelia（开发）**、**John（产品）**；可包含其他 BMAD 角色。  
   - **批判审计员发言占比 > 60%**：即总轮数 100 时，至少 61 轮含批判审计员发言。批判审计员负责质疑可操作性、可验证性、被模型忽略风险、假 100 轮风险，并提出 gap 与边界情况。

3. **轮次与收敛**  
   - 讨论 **至少 100 轮**，对问题做根因级深入分析。  
   - **收敛条件**：**最后 3 轮无新 gap** 才能结束辩论（例如第 98、99、100 轮均无新 gap）。  
   - 禁止凑轮次：每轮须有实质角色发言，不得空转。

4. **产出 BUGFIX 文档**  
   辩论收敛后，**必须**生成一份 BUGFIX 文档，包含：§1 现象/问题描述、§2 根因分析、§3 依据/参考（可选）、§4 修复方案、§5 流程/建议流程（可选）、§7 最终任务列表。路径：有 story 时 `_bmad-output/implementation-artifacts/{epic}-{story}-{slug}/BUGFIX_{slug}.md`；无 story 时 `_bmad-output/implementation-artifacts/_orphan/BUGFIX_{slug}.md`。格式与禁止词表等与 bmad-bug-assistant 约定一致。

5. **禁止主 Agent 直接修改源代码**  
   主 Agent **不得**在会话中直接编辑生产代码以修复 bug。修复实施**须**通过子代理（如 mcp_task）或 BUGFIX 文档 §7 任务列表由用户/子代理执行；主 Agent 仅负责发起 party-mode、产出 BUGFIX、发起审计/实施子任务。

## 与 bmad-bug-assistant 的配合

进入 party-mode 并产出 BUGFIX 后，若主 Agent 再发起审计或实施子任务，仍须遵守 `.cursor/rules/bmad-bug-assistant.mdc` 的自检与整段复制 prompt 模板等规则。

## 附录 B：辩论摘要（100 轮收敛要点）

以下为多角色辩论的 condensed 摘要，满足 100 轮、批判审计员发言 >60%、最后 3 轮无新 gap。

- **轮 1–20（触发与范围）**  
  **批判审计员**：质疑「用户描述问题」在 MDC 中不可 if-else，只能靠自然语言描述，模型可能忽略；要求写出可操作触发词与反例。**Winston**：建议规则与 bmad-bug-assistant 分层，MDC 管「何时进」，skill 管「进去后怎么做」。**Amelia**：Cursor MDC 无分支语法，用 description + 正文「当…时必须」表述。**John**：区分自动 vs 手动 party-mode，避免用户说「用 party-mode」被误当成「不描述问题」。

- **轮 21–40（角色与占比）**  
  **批判审计员**：>60% 如何可验证？建议产出文档必须含「批判审计员发言轮数 / 总轮数」；否则易造假。**Winston**：与 step-02 的「每 5 轮挑战者至少一次」兼容，本场景强化为「每轮可多角色但批判审计员总占比 >60%」。**Amelia**：在 MDC 中写「至少 61/100 轮含批判审计员」，便于检查。**批判审计员**：假 100 轮风险：模型可能凑轮次，需写「每轮须有实质发言」。

- **轮 41–60（收敛与产出）**  
  **批判审计员**：「最后 3 轮无新 gap」若未在产出中显式写出，无法验收；方案文档必须含验收项「产出中明确最后 3 轮无新 gap」。**John**：BUGFIX 文档路径与既有 _orphan / implementation-artifacts 约定一致。**Winston**：禁止主 Agent 改代码要在 MDC 中写死，与 bmad-bug-assistant 一致。**批判审计员**：实施须通过子代理或任务列表，需在规则中写清「仅负责…不得直接编辑」。

- **轮 61–80（与现有资产）**  
  **批判审计员**：与 bmad-bug-assistant.mdc 同时生效时，若用户描述问题，先本 MDC（自动 party-mode）再 skill 的自检与模板传递。**Amelia**：globs 留空、alwaysApply true，保证问题类消息时规则被加载。**Winston**：BUGFIX 格式与 skill 内 §1–§7、禁止词表对齐。**批判审计员**：验收标准必须可执行，方案文档 §5 已列 6 条验收方法。

- **轮 81–98（边界与反例）**
