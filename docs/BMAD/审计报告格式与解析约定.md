# 审计报告格式与解析约定

**产出日期**：2026-03-06  
**依据**：TASKS_audit-rating-pipeline-fix、parseAndWriteScore、extractOverallGrade、parseDimensionScores

---

## 1. 概述

speckit-workflow 各阶段（spec、plan、tasks 等）的审计报告需被 `parseAndWriteScore` 解析并写入 scoring 存储，供仪表盘显示评级。解析器对报告格式有明确要求。

---

## 2. 标准格式与逐条对照格式

审计报告支持两种风格：

| 风格 | 描述 | 解析要求 |
|------|------|----------|
| **标准格式** | 含 `总体评级: A/B/C/D` 与 `维度评分: 维度名: XX/100` | 可直接解析 |
| **逐条对照格式** | 表格 + ✅ 式逐条验证，结论为「完全覆盖、验证通过」等 | **必须**在结论后追加可解析块 |

无论采用哪种格式，报告**必须**包含以下可解析块，否则 `parseAndWriteScore` 无法解析（无 `SCORING_LLM_API_KEY` 时会抛 `ParseError`）。

---

## 3. 可解析块格式

**禁止用描述代替结构化块**：不得在总结或正文中用「可解析评分块（总体评级 X，维度分 Y–Z）」等文字概括；必须在报告中输出完整的结构化块。总体评级仅限 A/B/C/D，维度分须逐行写出，不得用区间或概括代替。

### 3.1 spec/plan/GAPS/tasks 阶段（prd 模式）

报告结尾须包含：

```markdown
## 可解析评分块（供 parseAndWriteScore）

总体评级: [A|B|C|D]

维度评分:
- 需求完整性: XX/100
- 可测试性: XX/100
- 一致性: XX/100
- 可追溯性: XX/100
```

详见 audit-prompts.md §4.1。

### 3.2 Implement 阶段（code 模式）

**Implement 阶段可解析块见 audit-prompts.md §5.1**。维度须与 `config/code-reviewer-config.yaml` 的 `modes.code.dimensions` 一致，四维为：

- 功能性
- 代码质量
- 测试覆盖
- 安全性

```markdown
## 可解析评分块（供 parseAndWriteScore）

总体评级: [A|B|C|D]

维度评分:
- 功能性: XX/100
- 代码质量: XX/100
- 测试覆盖: XX/100
- 安全性: XX/100
```

缺任一维度会导致 `parseDimensionScores(mode=code)` 解析不完整，仪表盘四维显示「无数据」。

---

## 4. 解析器输入要求

### 4.1 extractOverallGrade（scoring/parsers/audit-generic.ts）

- **输入**：报告全文 `content`
- **正则**：`/总体评级:\s*([ABCD])/`
- **输出**：匹配到的 A/B/C/D 或 `null`
- **说明**：无匹配时返回 `null`；`parseGenericReport` 在无 LLM 时会抛 `ParseError`

### 4.2 parseDimensionScores（scoring/parsers/dimension-parser.ts）

- **输入**：报告全文 `content`、`mode`（由 stage 映射：spec/plan/tasks→prd，implement→code，arch→arch，pr_review→pr）
- **正则**：`/^(?:[-*]\s*|\d+\.\s*)?(.+?)\s*[：:]\s*(\d+)\s*[\/／]\s*100\s*$/`（行级）
- **输出**：`DimensionScore[]`，每项含 `dimension`、`weight`、`score`
- **说明**：维度名须与 `config/code-reviewer-config.yaml` 对应 mode 的 `modes.{mode}.dimensions` 一致；implement 阶段用 `modes.code.dimensions`（四维见 §3.2）；无匹配时返回空数组

### 4.3 维度分与逐条对照结论映射建议

- 完全覆盖、验证通过 → A，各维度 90+
- 部分覆盖、minor 问题已修复 → B，各维度 80+
- 及格、需修改 → C，各维度 60–79
- 不及格、严重问题 → D，各维度 40–59

## 5. 路径约定与历史兼容

详见 `config/eval-lifecycle-report-paths.yaml`。

- **约定路径**：`AUDIT_tasks-E{epic}-S{story}.md`（如 `AUDIT_tasks-E8-S1.md`）
- **历史命名变体**：`AUDIT_TASKS_E8_S1_逐条对照_*.md` 等
- **兼容策略**：`parseAndWriteScore` 可通过 `--reportPath` 指定任意路径；仅新产出须符合约定路径；历史报告不回溯，可选人工追加可解析块后使用 `--reportPath` 解析

---

## 6. Story 完成自检（GAP-3.3）

完成 Dev Story 实施后审计后，主 Agent 须在提供完成选项前执行自检，确认该 Story 的评分已写入 scoring 存储。

### 6.1 检查逻辑

1. 调用 `scripts/check-story-score-written.ts --epic N --story N [--dataPath path]`
2. 脚本内部：`loadAndDedupeRecords` 加载 `scoring/data`（或 `--dataPath`）下的评分记录；用 `parseEpicStoryFromRecord` 或 run_id 正则 `dev-e{N}-s{N}-` 筛选 epic/story 匹配的记录
3. 输出：`STORY_SCORE_WRITTEN:yes` 或 `STORY_SCORE_WRITTEN:no`

### 6.2 数据路径

- 默认：`scoring/data/`（由 `getScoringDataPath()` 解析）
- 可选：`--dataPath` 指定自定义路径，用于集成测试或隔离环境

### 6.3 run_id 与 epic/story 对应

- 格式：`dev-e{epic}-s{story}-{stage}-{ts}` 或 `dev-e{epic}-s{story}-{ts}`
- 同一 Story 多 stage 共享 run_id 时使用 `--runGroupId`（参见 `scoring/docs/RUN_ID_CONVENTION.md` §4.2）
