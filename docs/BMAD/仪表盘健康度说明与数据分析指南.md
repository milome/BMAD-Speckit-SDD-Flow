# 项目健康度仪表盘：维度含义、评分计算与 Action Plan 指南

**版本**：1.0  
**日期**：2026-03-06  
**适用**：BMAD-Speckit-SDD-Flow 评分体系、`/bmad-dashboard`、`/bmad-coach`

---

## 1. 概述

项目健康度仪表盘基于 **real_dev** 场景的评分数据，输出总分、四维雷达图、短板 Top 3、高迭代 Top 3、Veto 统计和趋势，帮助技术负责人快速判断项目质量状态并形成改进 Action Plan（AP）。

---

## 2. 数据源与加载方式

### 2.1 数据路径

| 配置 | 默认值 | 覆盖方式 |
|------|--------|----------|
| 数据根路径 | `scoring/data/` | 环境变量 `SCORING_DATA_PATH` |

- **默认**：`{project-root}/scoring/data/`
- **覆盖示例**：`SCORING_DATA_PATH=_bmad-output/scoring node ...`

### 2.2 数据文件形式

| 类型 | 路径/格式 | 说明 |
|------|-----------|------|
| 单 run 文件 | `scoring/data/{run_id}.json` | 单次运行的完整记录 |
| 追加模式 | `scoring/data/scores.jsonl` | 每行一条 JSON 记录 |

### 2.3 加载逻辑

1. **扫描**：读取 `scoring/data/` 下所有 `*.json` 与 `scores.jsonl`
2. **过滤**：仅保留 `scenario=real_dev`（排除 eval_question）
3. **去重**：同一 `(run_id, stage)` 取 `timestamp` 最新一条
4. **最新 run**：按 run 的 `timestamp` 降序，取最近一个 run 的 records 作为仪表盘计算基准

**调用链**：
```
getScoringDataPath() → loadAndDedupeRecords() → filter(scenario=real_dev) → getLatestRunRecords()
```

### 2.4 iteration_records 扩展（Story 9.4）

- **字段含义**：`iteration_records` 可含 optional `overall_grade`（A|B|C|D）、`dimension_scores`，用于记录审计多轮迭代的评分演进
- **写入时机**：仅 pass 时由主 Agent 传入 `--iterationReportPaths`（失败轮报告路径，不含验证轮），`parseAndWriteScore` 一次性解析并写入
- **演进轨迹展示**：Coach 与仪表盘在短板 Top 3、高迭代 Top 3 中，当 record 的 `iteration_records` 非空且至少一条含 `overall_grade` 时，追加展示格式 `第1轮 C → 第2轮 B → 第3轮 A`
- **单轮通过**：无 fail 轮时 `iteration_records` 为空，无回归

## 3. 四维雷达图：维度含义与来源

### 3.1 多模式与 stage→mode 映射

评分体系支持 **四种模式**（prd、arch、code、pr），不同 stage 映射到不同 mode，对应不同的维度定义。Epic/Story 创建与执行过程中的各阶段均有评分与统计：

| stage | mode | 对应环节 / 流程阶段 | 四维定义 |
|-------|------|---------------------|----------|
| prd | prd | Layer 1 产品定义：PRD 审计 | 需求完整性、可测试性、一致性、可追溯性 |
| arch | arch | Layer 1 架构设计：Architecture 审计 | 技术可行性、扩展性、安全性、成本效益 |
| story | code | Layer 3 Create Story：Story 文档审计 | 功能性、代码质量、测试覆盖、安全性 |
| spec、plan、tasks | prd | Layer 4 speckit：spec/plan/tasks 审计 | 需求完整性、可测试性、一致性、可追溯性 |
| implement、post_impl | code | Layer 4 执行：代码审计 | 功能性、代码质量、测试覆盖、安全性 |
| pr_review | pr | PR 合并前审计 | CI 状态、代码审查、测试覆盖、影响评估 |

**说明**：`epics`（Layer 2 Epic/Story 规划）为环节 1 的输入依据，**不单独计分**（见 `scoring/constants/table-b.ts`）。

**映射逻辑**：`scoring/parsers/dimension-parser.ts` 的 `stageToMode(stage)`。

**为何文档着重写 PRD 模式**：real_dev 场景下，**主要评分数据**来自 spec/plan/tasks（Dev Story 的 speckit 流程），三者均映射到 prd 模式，故 §3.2 的「四维」指 prd 的四维。当数据包含 arch、story、implement 等 stage 时，仪表盘会合并各 record 的 dimension_scores；若某 stage 的审计报告未输出 `维度名: XX/100`，该 record 的 dimension_scores 为空，仪表盘对该维度显示「无数据」或使用 code 模式的 fallback 维度名。

### 3.2 维度定义（PRD 模式）

当数据来自 spec/plan/tasks 时，仪表盘四维来自 **code-reviewer-config 的 prd 模式**：

| 维度 | 权重 | 含义 | 检查要点 |
|------|------|------|----------|
| **需求完整性** | 35% | 需求文档是否覆盖所有必要场景与边界 | 覆盖所有用户场景、边界条件明确、异常流程考虑 |
| **可测试性** | 25% | 需求是否可被验证 | 验收标准明确、可验证、测试场景清晰 |
| **一致性** | 25% | 文档与既有约定是否一致 | 与 Product Brief 一致、内部逻辑自洽、术语统一 |
| **可追溯性** | 15% | 需求是否可追踪 | 需求有唯一 ID、描述足够详细、依赖关系明确 |

### 3.3 维度数据来源

- **来源**：`RunScoreRecord.dimension_scores`
- **解析**：`parseAndWriteScore` 从审计报告（AUDIT_spec/plan/tasks 等）中解析 `维度名: XX/100` 格式，按 `config/code-reviewer-config.yaml` 的 `modes.prd.dimensions` 匹配权重
- **聚合**：同一维度下所有 record 的 score 取**算术平均**，四舍五入

### 3.4 无数据时的 fallback

若 `dimension_scores` 为空，仪表盘使用默认维度名 `['功能性','代码质量','测试覆盖','安全性']`（code 模式），并显示「无数据」。

### 3.5 示例：四维分数从哪里来？分别在哪儿扣的分？

以下以真实数据为例说明四维分数的来源与扣分发生位置：

| 维度       | 分数 | 权重 |
|------------|------|------|
| 一致性     | 78   | 25%  |
| 可测试性   | 80   | 25%  |
| 可追溯性   | 82   | 15%  |
| 需求完整性 | 85   | 35%  |

**来源**：上述分数来自审计报告（AUDIT_spec/plan/tasks 等）中 code-reviewer 输出的 `维度名: 分数/100` 行，由 `scoring/parsers/dimension-parser.ts` 的正则 `维度名: 分数/100` 解析并写入 `RunScoreRecord.dimension_scores`。

**扣分发生位置**：四维分数在 code-reviewer 审计阶段产出，扣分发生在**评审员对各维度检查项的评估**中，而非 scoring 代码的公式计算。各维度满分 100，评审员依据 `config/code-reviewer-config.yaml` 中 `modes.prd.dimensions` 的 checks 逐项评估后给出 0–100 分。

**评分主体与自审风险说明**：

- **评分主体**：维度分数由 **AI 模型（code-reviewer）通过推理与判断产出**。模型读取被审文档/代码，对照 checks 清单逐项评估，自行推断是否满足、不足程度，并输出 `维度名: XX/100`。此过程无预定义公式，**属于模型的主观推断与打分结论**。
- **自审风险**：当**同一模型**既生成 spec/plan/code，又在后续步骤以 code-reviewer 身份审计并打分时，存在 **自审偏袒（self-review bias）** 风险：模型可能对自身产出更宽容，扣分偏少。BMAD 流程的缓解措施包括：
  - **角色分离**：优先通过 Cursor Task 调度独立的 code-reviewer agent（`.cursor/agents/code-reviewer.md`），与执行 Dev Story 的 subagent 分离；回退时虽用 `mcp_task generalPurpose`，仍通过不同 prompt 与角色设定（如「严苛审计员」）区分。
  - **批判审计员**：审计提示词要求引入**批判审计员**视角，发言占比 >50%，从对抗角度质疑遗漏与不足，降低「自圆其说」倾向。
  - **迭代闭环**：审计不通过则修改后再次审计，多轮迭代可暴露初次自审未发现的问题。
- **建议**：对高可信度需求（如合规、安全），可辅以人工抽检或跨模型复验。

| 维度       | 检查要点（code-reviewer-config）           | 示例分数 | 扣分含义                     |
|------------|-------------------------------------------|----------|------------------------------|
| 需求完整性 | 覆盖所有用户场景、边界条件明确、异常流程考虑 | 85       | 满分 100 扣约 15，存在遗漏   |
| 可测试性   | 验收标准明确、可验证、测试场景清晰         | 80       | 满分 100 扣约 20，部分可优化 |
| 一致性     | 与 Product Brief 一致、内部逻辑自洽、术语统一 | 78       | 满分 100 扣约 22，略有不一致 |
| 可追溯性   | 需求有唯一 ID、描述足够详细、依赖关系明确   | 82       | 满分 100 扣约 18，可追溯性一般 |

**phase_score 计算**：有 dimension_scores 时，`phase_score = Σ(score × weight / 100)`：

```
81.55 = 85×0.35 + 80×0.25 + 78×0.25 + 82×0.15
      = 29.75 + 20 + 19.5 + 12.3
```

**仪表盘展示**：同一维度在多个 record（如 spec、plan、tasks）中均有分数时，仪表盘取**算术平均**；仅单 record 有分数时，直接展示该 record 的 dimension_scores。

**相关代码**：
- 解析：`scoring/parsers/dimension-parser.ts` — `parseDimensionScores(content, mode)`、正则 `(.+?)\s*[：:]\s*(\d+)\s*[\/／]\s*100`
- 权重：`config/code-reviewer-config.yaml` — `modes.prd.dimensions`
- 写入：`scoring/orchestrator/parse-and-write.ts` — `computeWeightedDimensionScore` 覆盖 `phase_score`
- 聚合：`scoring/dashboard/compute.ts` — `getDimensionScores` 按 dimension 取平均

### 3.6 示例：多模式并存时的四维展示

当同一 run 内同时存在 prd、code、arch 等多种模式的 record 时，仪表盘会**合并所有 dimension_scores**，按维度名聚合并取算术平均。

**假设最新 run 包含 3 条 record**：

| record | stage | mode | dimension_scores |
|--------|-------|------|------------------|
| 1 | spec | prd | 需求完整性 85、可测试性 80、一致性 78、可追溯性 82 |
| 2 | implement | code | 功能性 88、代码质量 75、测试覆盖 70、安全性 90 |
| 3 | arch | arch | 技术可行性 85、扩展性 80、安全性 85、成本效益 75 |

**聚合逻辑**：`getDimensionScores` 遍历所有 record，按 `dimension` 名收集分数；同名维度（如 arch 与 code 均有「安全性」）**合并后取平均**。

**仪表盘展示（维度按名称排序）**：

| 维度 | 分数 | 计算说明 |
|------|------|----------|
| 成本效益 | 75 | 仅 arch |
| 代码质量 | 75 | 仅 code |
| 功能性 | 88 | 仅 code |
| 可测试性 | 80 | 仅 prd |
| 可追溯性 | 82 | 仅 prd |
| 扩展性 | 80 | 仅 arch |
| 技术可行性 | 85 | 仅 arch |
| 需求完整性 | 85 | 仅 prd |
| 一致性 | 78 | 仅 prd |
| 安全性 | 88 | arch(85) 与 code(90) 平均 → 88 |
| 测试覆盖 | 70 | 仅 code |

**雷达图**：上述 11 个维度会同时展示；若某维度在所有 record 中均无数据，则显示「无数据」。

**总分计算**：总分 = Σ(phase_score × phase_weight) / Σ(phase_weight)（见 §4.1）。每条 record 的 phase_score 来自其 dimension_scores 的加权和；phase_weight 由 stage 对应环节权重确定。

| record | stage | phase_score（维度加权和） | phase_weight | 贡献 |
|--------|-------|--------------------------|--------------|------|
| 1 | spec | 85×0.35+80×0.25+78×0.25+82×0.15 = **81.55** | 0.25（环节 3） | 81.55×0.25 = 20.39 |
| 2 | implement | 88×0.30+75×0.30+70×0.20+90×0.20 = **80.90** | 0.10（环节 5） | 80.90×0.10 = 8.09 |
| 3 | arch | 85×0.30+80×0.25+85×0.25+75×0.20 = **81.75** | 0.25（环节 2） | 81.75×0.25 = 20.44 |

```
总分 = (20.39 + 8.09 + 20.44) / (0.25 + 0.10 + 0.25)
     = 48.92 / 0.60
     ≈ 81.53 → 四舍五入得 82
```

## 4. 总分与评分计算

### 4.1 项目健康度总分

**公式**：

```
总分 = Σ(phase_score × phase_weight) / Σ(phase_weight)
```

- **phase_score**：单条 record 的 phase_score（0–100）
- **phase_weight**：该 stage 的环节权重，来自 `RunScoreRecord.phase_weight`
- **计算范围**：最新 run 的所有 records
- **结果**：四舍五入取整

### 4.2 环节权重（PHASE_WEIGHTS）

| 环节 | 权重 | 说明 |
|------|------|------|
| 1 | 20% | 需求/PRD |
| 2 | 25% | 架构/设计 |
| 3 | 25% | spec/plan |
| 4 | 15% | tasks |
| 5 | 10% | implement |
| 6 | 5% | post_impl |

定义于 `scoring/constants/weights.ts`。

### 4.3 phase_score 的修正

- **阶梯扣分**：`iteration_count` 越高，phase_score 按系数打折（1 次 100%、2 次 80%、3 次 50%、≥4 次 0%）
- **Veto 触发**：若该 record 的 check_items 中存在 veto 项且 `passed=false`，则 `phase_score=0`

## 5. 短板识别

### 5.1 短板 Top 3

**定义**：最新 run 中 `phase_score` 最低的 3 条 record。

**算法**：
```
1. 按 phase_score 升序排序
2. 取前 3 条
3. 输出：stage、epicStory（从 run_id/path 解析）、score
```

**展示格式**：`E6.S3 spec: 81.55 分`

### 5.2 高迭代 Top 3

**定义**：`iteration_count > 0` 的 record 中，按 iteration_count 降序取前 3。

**含义**：该 stage 审计未通过次数最多，是效率短板。

**展示**：`各 stage 均为一次通过`（当全部 iteration_count=0 时）

### 5.3 Coach 的 weak_areas

Coach 诊断中：
- **weak_areas**：`phase_score < 70` 的 stage 列表
- **threshold**：70 分，可视为「需改进」边界

## 6. Veto 触发统计

### 6.1 含义

**Veto**：一票否决项。当某 check_item 的 `item_id` 属于 veto 配置且 `passed=false` 时，该 stage 的 phase_score 被置为 0。

### 6.2 统计方式

```
count = Σ record.check_items 中 passed=false 且 item_id ∈ vetoItemIds 的个数
```

**vetoItemIds** 来源：`scoring/rules/phase-2/3/4-scoring.yaml`、`stage-spec/plan/tasks-scoring.yaml`、`gaps-scoring.yaml` 中的 `veto_items[].ref` 指向的 item_id。

### 6.3 展示

- `Veto 触发：0 次`：本 run 无 veto 触发
- 若 >0：需优先修复对应 check_item

---

## 7. 趋势

### 7.1 计算逻辑

1. 取最近 5 个 run（按 run 的 max timestamp 降序）
2. 计算最新 run 与前一 run 的加权总分
3. **升**：最新 > 前一  
   **降**：最新 < 前一  
   **持平**：最新 = 前一，或仅有 1 个 run

## 8. 如何形成 Action Plan（AP）

### 8.1 输入组合

| 来源 | 信息 | 用途 |
|------|------|------|
| 仪表盘 | 短板 Top 3、高迭代 Top 3、Veto 统计、四维 | 量化短板与趋势 |
| Coach | weak_areas、recommendations | 定性建议与优先级 |

### 8.2 AP 形成步骤

1. **优先级 1：Veto**
   - 若 Veto 触发 > 0：优先修复 veto 对应的 check_item
   - Coach 建议：「存在环节 veto 触发，优先修复 veto 对应检查项后再执行下一轮迭代」

2. **优先级 2：短板 Top 3**
   - 针对得分最低的 stage（如 E6.S3 spec）
   - 行动：查阅该 stage 的审计报告，逐项修复未通过检查项
   - Coach weak_areas 中 phase_score<70 的 stage 可合并考虑

3. **优先级 3：高迭代 Top 3**
   - 针对 iteration_count 最高的 stage
   - 行动：提升该 stage 的一次通过率，减少审计轮次
   - Coach 建议：「建议关注高整改轮次 stage，提升一次通过率」

4. **优先级 4：四维均衡**
   - 若某维度明显低于其他（如一致性 78，其余 80+）
   - 行动：在后续 spec/plan/tasks 中加强该维度的检查与准备

### 8.3 AP 模板示例

```markdown
## Action Plan（基于 2026-03-06 仪表盘）

### 1. Veto
- 状态：0 次触发 ✓
- 动作：无

### 2. 短板 Top 3
- E6.S3 spec: 81.55 分
- 动作：检查 AUDIT_spec-E6-S3，补齐需求完整性/可测试性等维度的不足

### 3. 高迭代
- 状态：各 stage 均为一次通过 ✓
- 动作：维持

### 4. 四维
- 一致性 78 略低，后续 PRD/spec 起草时加强与 Product Brief 对齐、术语统一
```
