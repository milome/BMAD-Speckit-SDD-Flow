## 一、整合后的五层架构

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer 1: 产品定义层                                               │
│ ├─ Product Brief                                                 │
│ ├─ 复杂度评估 → 决定Party-Mode配置                                │
│ ├─ PRD → code-review审计(prd模式)                                │
│ └─ Architecture（如需）→ code-review审计(arch模式)                │
├─────────────────────────────────────────────────────────────────┤
│ Layer 2: Epic/Story规划层                                         │
│ ├─ create-epics-and-stories                                      │
│ ├─ 粗粒度Story拆分、依赖关系分析                                    │
│ └─ 产出：Epic列表、Story列表、依赖图                               │
├─────────────────────────────────────────────────────────────────┤
│ Layer 3: Story开发层                                              │
│ ├─ Create Story（细化）→ Party-Mode → Story文档                   │
│ ├─ PRD需求追溯、Architecture约束传递                              │
│ └─ 产出：详细Story文档、验收标准                                   │
├─────────────────────────────────────────────────────────────────┤
│ Layer 4: 技术实现层（嵌套speckit）                                 │
│ ├─ specify → plan → GAPS → tasks → TDD执行                       │
│ ├─ 每层code-review审计（§1-§5）                                   │
│ ├─ 需求映射：PRD→Story→spec→task                                  │
│ └─ 产出：spec.md, plan.md, tasks.md, 可运行代码                    │
├─────────────────────────────────────────────────────────────────┤
│ Layer 5: 收尾层                                                   │
│ ├─ 批量Push + PR自动生成                                          │
│ ├─ 强制人工审核（绝对不能自动merge）                                │
│ └─ Epic集成测试、发布                                             │
└─────────────────────────────────────────────────────────────────┘
```

## 二、复杂度评估与Party-Mode触发

### 2.1 评估问卷（Layer 1执行）

```yaml
业务复杂度:
  - 领域知识: [熟悉/部分新/全新]
  - 利益相关方数量: [≤2/3-5/>5]
  - 合规要求: [无/一般/严格]

技术复杂度:
  - 技术栈: [现有/部分新/全新]
  - 架构挑战: [无/中等/高并发大数据]
  - 集成难度: [独立/少量依赖/复杂网络]

影响范围:
  - [单个Story/单个模块/跨模块/全系统]
```

### 2.2 Party-Mode触发规则

| 总分 | PRD处理 | Architecture处理 | Create Story处理 |
|------|---------|------------------|------------------|
| ≤6分 | 直接生成 | 直接生成 | 标准流程 |
| 7-10分 | 50轮Party-Mode | 可选30轮 | 标准流程 |
| 11-15分 | 80轮Party-Mode | 80轮Party-Mode | 可选Party-Mode |
| >15分 | 80轮+专家Review | 80轮+专家Review | 强制Party-Mode |

### 2.3 PRD Party-Mode角色

- 产品经理、业务分析师、用户研究员
- 技术架构师、合规专员、**批判审计员**

### 2.4 Architecture Party-Mode角色

- 系统架构师、性能工程师、安全架构师
- 运维工程师、成本分析师、**批判审计员**

## 三、Worktree策略

### 3.1 核心原则

```yaml
Story数 ≤ 2: Story级worktree（简化流程）
Story数 ≥ 3: Epic级worktree（完整流程）
```

### 3.2 执行模式

**串行模式（默认）**：
```
Story 4.1 → merge → Story 4.2 → merge → Story 4.3
```

**并行模式（可选）**：
```
Story 4.1 ─┐
           ├─→ 并行开发 → 逐个merge + 冲突解决审计
Story 4.2 ─┘
```

### 3.3 模式切换命令

```bash
/bmad-set-worktree-mode epic=4 mode=parallel   # 切换到并行模式
/bmad-set-worktree-mode epic=4 mode=serial     # 切换到串行模式
/bmad-set-worktree-mode epic=4 mode=story-level # 回退到Story级
```

## 四、统一审计体系

### 4.1 code-reviewer多模式

| 模式 | 审计对象 | 通过标准 |
|------|---------|---------|
| code | 代码文件 | A/B级 |
| prd | PRD文档 | 完全覆盖 |
| arch | Architecture文档 | 完全覆盖 |
| pr | Pull Request | A/B级 |

### 4.2 完整审计链

```
PRD → Architecture → Story文档 → spec → plan → GAPS → tasks → 执行后
 ↓        ↓            ↓          ↓       ↓      ↓       ↓        ↓
prd模式  arch模式      默认       §1      §2     §3      §4       §5
```

### 4.3 批判审计员职责

1. **介入时机**：Create Story party-mode（强制）、PRD/Architecture审计（强制）
2. **每轮必做**：提出至少1个深度质疑、检查需求映射、验证假设
3. **退出条件**：所有质疑回应、近3轮无新gap、明确声明认可
4. **权力边界**：暂停权、记录权、否决权

## 八、关键决策记录（ADR）

### ADR-001: 采用混合方案
- **决策**：以五层架构为框架，融入v2.0的复杂度评估和PR自动化
- **理由**：兼顾流程清晰度和决策科学性

### ADR-002: Story数阈值设为2
- **决策**：Story≤2使用Story级，≥3使用Epic级
- **理由**：简化决策，覆盖大多数场景

### ADR-003: 单次渐进评估（MVP）
- **决策**：Layer 1进行一次评估，预留两次评估扩展点
- **理由**：平衡质量和效率

### ADR-004: code-reviewer 4模式
- **决策**：扩展code-reviewer支持code/prd/arch/pr四种模式
- **理由**：统一审计入口，一致标准
