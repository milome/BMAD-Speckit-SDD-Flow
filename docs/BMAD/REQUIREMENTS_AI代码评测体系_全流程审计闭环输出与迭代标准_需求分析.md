## §1 问题与目标

### 1.1 背景

- **现有体系**：BMAD+Speckit 全流程各环节已有审计闭环（spec/plan/gaps/tasks/implement），部分环节有评级（如 plan/tasks 阶段审计评级），但**缺少全体系统一评分与能力等级**，难以直接回答「模型综合能力多少分、短板在哪个环节、该轮迭代是否达标」。
- **改进目标**：在不大幅推翻现有审计闭环的前提下，增加**全体系评分评级**，并使其成为**各阶段审计闭环的重要输出与迭代结束的判定标准之一**，便于模型迭代指导、能力评估与对外能力说明等场景。
- **BMAD 五层覆盖说明**：本需求历史聚焦 Layer 4（Speckit 技术实现层）。本次更新将扩展至 **BMAD 完整五层**：Layer 1 产品定义（PRD/Architecture）、Layer 2 Epic/Story 规划、Layer 3 Story 开发、Layer 4 技术实现（speckit specify→plan→gaps→tasks→implement）、Layer 5 收尾（实施后审计、PR、强制人工审核）。全体系评分须与五层审计闭环一一映射，作为各层迭代结束的判定依据之一。
- **AI 代码教练与全链路 Skill**：本需求扩展引入**AI 代码教练（AI Code Coach）**角色与**全链路 Code Reviewer Skill**。教练用资深工程师视角，还原真实工业级开发全流程，精准定位 AI 代码模型的能力短板，设计可落地的评测/优化方案；作为「现有全流程各审计闭环的重要输出和迭代结束标准」的**承载者**。全链路 Skill 独立于现有单点 code-reviewer，通过引用 code-reviewer、audit-prompts、code-reviewer-config 完成全链路审计与评分编排。

### 1.2 目标

1. **全体系评分评级**：增加「六环节分项评分 → 四能力维度聚合 → 综合百分制 → L1–L5 能力等级」的标准化体系，与现有环节一一映射。
2. **作为全流程审计闭环的统一出口与迭代结束标准**：各阶段审计通过且（若启用评分）对应环节得分写入存储后，**可视为该阶段迭代结束**；全体系综合得分与等级作为全流程的统一输出。
3. **可落地与可复现**：主观环节可量化、评分可复现（含一致性要求）、数据污染可防、版本可追溯。

### 1.3 与全流程阶段及评测目标的对应

| 评测/建设目标 | 本需求/设计对应 |
|--------------|------------------|
| 题目与评测集构建 | 六模块评测题设计（需求拆解、代码、测试、调试、跨模块、端到端）；题池规模与版本管理 |
| 模型行为分析与迭代指导 | 环节得分与四维度聚合、短板定位、版本对比与迭代追溯 |
| 模型效果评估与强化学习配合 | 综合评分与 L1–L5 等级可作为奖励信号；优化方向对应环节与数据补充建议 |
| 工程规范与质量底线 | 规范来源于工程沉淀；一票否决项与工程底线；代码/测试/修复环节高权重 |
| 全流程能力评估 | 六模块即出题与评分维度；端到端交付为必过线 |

### 1.4 场景区分：自有真实需求开发 vs 评测题目执行

**必须区分**两种场景，避免评测逻辑绑架真实开发流程：

| 场景 | 定义 | 评分触发与存储 | 迭代结束标准 |
|------|------|----------------|---------------|
| **自有真实需求开发** | 基于真实业务需求的全流程开发（spec→plan→gaps→tasks→implement），产出为可交付项目 | 各阶段审计通过后**自动解析**既有审计报告与验收表，写入评分存储；**无额外必填步骤** | 与现有 BMAD+Speckit 一致：该阶段审计「完全覆盖、验证通过」；若启用评分则对应环节得分已写入 |
| **评测题目执行** | 基于预设评测题（题池）执行，用于模型横向对比、能力评估 | 每次题目执行产生独立 run_id，得分写入独立记录；可与同题目版本的其他 run 对比 | 题目执行完成且得分已写入；与真实开发流程**解耦** |

- **主场景**：自有真实需求开发。本改进**增强**该流程，评分为其**副产品**，不增加拖沓。
- **独立场景**：评测题目执行。用于模型评测、横向对比，与真实开发**可选并行**，不干扰主流程。

**§1.4.1 场景与 BMAD 层路径**

| 场景 | BMAD 层路径 | 说明 |
|------|-------------|------|
| **自有真实需求开发** | Layer 1→2→3→4→5 完整路径 | Product Brief→PRD→Arch→Epics→Story→specify→plan→gaps→tasks→implement→实施后审计→PR→人工审核 |
| **评测题目执行** | **必须同样走 Layer 1→2→3→4→5 完整路径** | 与自有开发一致；**不得**采用简化路径（如仅 Layer 4、Layer 3+4 等），否则评分无法全面基于五层审计闭环，无法为六环节与四能力维度提供完整得分；scoring 存储中 scenario 区分 real_dev 与 eval_question |

### 1.5 增强而非拖沓的约束（轻量化三原则）

为确保本次改进**增强**真实自有需求的开发流程，而非让流程更拖沓，须满足：

1. **同机执行**：评分写入与各阶段审计**同机执行**；审计通过即自动从既有审计报告与验收表解析并写入，**无额外人工填写、无新增必填表单**。
2. **可选启用**：全体系评分可**按项目或按运行配置关闭**；未启用时，各阶段迭代结束标准与现有 BMAD+Speckit **完全一致**，无任何新增步骤。
3. **最小侵入**：不修改现有审计闭环的输入输出格式；仅增加「解析既有产出并写入评分存储」的**可选后置步骤**；现有 progress、artifacts、_bmad-output 结构保持不变。

**gaps 双轨评审说明**：gaps 阶段产出 IMPLEMENTATION_GAPS 文档，进行**前置完整性评审**（规范全覆盖、gap 定义清晰性、风险预判），得分计入环节 1 补充；**后置闭合度评审**在 implement 阶段计入环节 2 的 gaps 闭合子维度（占环节 2 的 30%）、post_impl 阶段计入环节 6 的 gaps 闭合子维度（占环节 6 的 50%）。

### 2.2 各阶段迭代结束标准（Layer 1–5）

在**启用全体系评分**的前提下，以下条件同时满足时，可视为**该阶段迭代结束**：

**Layer 1**：
- **prd**：PRD 审计结论为「完全覆盖」，且（若录入）环节 1 需求拆解得分已写入 scoring 存储。
- **arch**：Architecture 审计结论为「完全覆盖」，且（若录入）环节 1 补充与环节 2 设计侧得分已更新。

**Layer 2**：
- **epics**：create-epics-and-stories 产出 Epic/Story 列表与依赖图；无独立评分环节，迭代结束以产出完成为准。

**Layer 3**：
- **story**：Create Story 审计结论为「完全覆盖、验证通过」，且（若录入）环节 1 补充得分已写入。

**Layer 4（speckit）**：
- **specify**：spec 审计结论为「完全覆盖、验证通过」，且（若录入）环节 1 得分已写入。
- **plan**：plan 审计结论为「完全覆盖、验证通过」，且（若录入）环节 1 补充与环节 2 设计侧得分已更新。
- **gaps**：gaps 前置完整性评审通过；IMPLEMENTATION_GAPS 审计结论为「完全覆盖、验证通过」；与 tasks 的衔接已确认；且（若录入）环节 1 补充的 gaps 前置得分已写入。
- **tasks**：tasks 审计结论为「完全覆盖、验证通过」，且各任务验收表已按实际执行填写。
- **implement**：执行 tasks 后的审计（audit-prompts §5）结论为「完全覆盖、验证通过」，且环节 2–6 得分已录入。

**Layer 5**：
- **post_impl**：实施后审计 §5 结论为「完全覆盖、验证通过」，且环节 2–6 得分已录入、综合分与等级已计算并写入存储。
- **pr_review**：强制人工审核通过；可选记录，不作为迭代结束的必填条件。

**未启用全体系评分时**：各阶段仅以该阶段审计结论「完全覆盖、验证通过」为迭代结束标准，与现有 BMAD+Speckit 行为一致。

## §3 全体系评分评级改进需求

### 3.1 设计原则

- **底层逻辑先行**：评分服务于（1）迭代结束判定、（2）短板定位、（3）模型优化。先定底层逻辑，避免为了打分而打分；若某环节得分无法支撑上述三者之一，则该环节的评分设计需重新审视。
- **全链路对齐**：100% 匹配现有「需求拆解→方案设计→代码开发→测试生成→调试修复→端到端交付」全流程，不推翻原有审计闭环，仅做标准化量化升级。
- **工业级权重**：权重贴合真实开发价值优先级（需求+测试+调试合计 60%，代码 25%，端到端 5% 锚定结果）。
- **可量化无歧义**：主观环节拆解为原子化检查项，每项有固定扣/加分，目标同交付物多评分者误差 ≤5 分。
- **可追溯可优化**：评分可拆解到环节与维度，便于定位短板；支持版本对比（同题目版本、不同模型/版本）。
- **可对标可认证**：L1–L5 等级与综合得分区间固定；主流模型对标为可替换附录，注明基准日期与参考性。

#### 3.1.1 设计原则贯彻检查清单

| 设计原则 | 可验证项（落地检查清单） |
|----------|--------------------------|
| 全链路对齐 | 六环节是否 100% 与需求拆解→方案设计→代码→测试→调试→端到端对应？阶段→评分环节映射表是否完整？ |
| 工业级权重 | 权重是否依据人力占比、落地痛点、能力区分度？是否在 scoring/rules 中可配置？ |
| 可量化无歧义 | 每检查项是否有明确通过/不通过条件？同交付物多评分者误差是否目标 ≤5 分？ |
| 可追溯可优化 | 是否支持 run_id、question_version、stage 维度追溯？是否支持同题目不同模型版本对比？ |
| 可对标可认证 | L1–L5 等级与得分区间是否固定？对标表是否可替换并注明基准日期？ |
| 底层逻辑先行 | 各环节得分是否服务于迭代结束判定、短板定位、模型优化？是否存在无法支撑上述目的的评分项？ |

### 3.2 四层架构

1. **第一层：六环节分项评分**（满分 100，按权重分配）
   - 环节 1：需求拆解与方案设计（20 分）
   - 环节 2：代码生成与工程规范（25 分）
   - 环节 3：测试用例与质量保障（25 分）
   - 环节 4：调试与 bug 修复（15 分）
   - 环节 5：跨模块/存量项目集成（10 分）
   - 环节 6：端到端全流程交付（5 分）

2. **第二层：四能力维度聚合**
   - 需求与设计能力（环节 1）
   - 代码与工程能力（环节 2+5 加权）
   - 质量与闭环能力（环节 3+4 加权）
   - 端到端交付能力（环节 6）

3. **第三层：综合百分制**  
   综合得分 = Σ(环节得分 × 对应权重)，0–100 分。

4. **第四层：L1–L5 能力等级**
   - L5 专家级：90–100；L4 资深级：80–89；L3 进阶级：60–79；L2 入门级：40–59；L1 玩具级：0–39。  
   - 定义与得分区间固定；主流模型对标为附录，可替换与定期复核。

#### 3.2.1 Epic 综合评分

当评测对象为 Epic（含多个 Story）时，采用 Epic 综合评分：

- **单 Story 综合分**：综合得分 = Σ(环节得分 × 环节权重)，0–100 分。
- **Epic 综合分**：Epic 综合分 = Σ(Story 综合分 × Story 权重)，Story 权重可配置（默认均等）。
- **L1–L5 映射**：Epic 综合分采用与单次运行相同的得分区间（L5 90–100、L4 80–89、L3 60–79、L2 40–59、L1 0–39）。

### 3.3 权重依据

- 真实开发人力占比：需求/设计/测试/调试占 70%+，代码编写约 30%。
- 企业落地痛点：规范、测试、安全、可维护性为卡点，故设高权重与一票否决。
- 能力区分度：需求拆解与全流程质量闭环区分度高，故提高相应权重。  
以上为工业依据；具体权重数值为可调参数，可在 scoring/rules/ 中配置。

### 3.4 一票否决项与环节映射

| 一票否决项 | 所属环节 | 后果 |
|------------|----------|------|
| OWASP Top 10 高危、硬编码敏感信息 | 环节 2 | 该环节 0 分，综合等级下调一级（L1 不再下调） |
| 核心业务需求完全遗漏、功能不符预期 | 环节 1 | 同上 |
| 代码无法编译/运行、致命语法错误 | 环节 2 | 同上 |
| 修复引入次生高危 bug、破坏现有业务 | 环节 4 | 同上 |

#### 3.4.1 OWASP Top 10 高危、硬编码敏感信息 — 判定标准与权威文档

**一、OWASP Top 10 高危**

- **定义**：OWASP Top 10 是 OWASP 发布的 Web 应用安全风险标准清单，代表业界对最常见、最严重安全风险的共识。
- **判定依据**：OWASP Top 10 官方版本（当前为 2025 版）；发现与下列任一类别对应的**可利用**漏洞即触发一票否决。
- **官方文档与链接**：
  - 主站：https://owasp.org/Top10/
  - 2025 版：https://owasp.org/Top10/2025/
  - 项目仓库：https://github.com/OWASP/www-project-top-ten

**OWASP Top 10:2025 列表**：

| 编号 | 名称 | 官方链接 |
|------|------|----------|
| A01 | Broken Access Control（访问控制失效） | https://owasp.org/Top10/2025/A01_2025-Broken_Access_Control/ |
| A02 | Security Misconfiguration（安全配置错误） | https://owasp.org/Top10/2025/A02_2025-Security_Misconfiguration/ |
| A03 | Software Supply Chain Failures（软件供应链失效） | https://owasp.org/Top10/2025/A03_2025-Software_Supply_Chain_Failures/ |
| A04 | Cryptographic Failures（加密失效） | https://owasp.org/Top10/2025/A04_2025-Cryptographic_Failures/ |
| A05 | Injection（注入） | https://owasp.org/Top10/2025/A05_2025-Injection/ |
| A06 | Insecure Design（不安全设计） | https://owasp.org/Top10/2025/A06_2025-Insecure_Design/ |

- **判定方式**：静态扫描（如 OWASP ZAP、npm audit、Snyk）+ 人工复核；各条目官方页面含攻击示例与修复建议。

**二、硬编码敏感信息**

- **定义**：产品在代码或配置中直接包含固定凭证（密码、密钥、API Key 等），对应 **CWE-798: Use of Hard-coded Credentials**。
- **判定依据**：CWE-798 官方定义；https://cwe.mitre.org/data/definitions/798.html
- **两种典型形态**：
  - **Outbound（出站）**：连接外部系统时在代码中硬编码后端凭证（如数据库连接串、API 密钥）。
  - **Inbound（入站）**：内置默认管理员账号，密码或密钥写死在代码中，且通常无法通过配置修改。
- **常见硬编码敏感信息类型**：密码、数据库凭证（连接串/用户名/密码）、API 密钥/Token、加密密钥与 IV、盐值、主机名/IP（用于连接后端）、默认用户名。
- **典型代码表现**：`DriverManager.getConnection(url, "scott", "tiger");`、`if (strcmp(password, "Mew!")) {...}`、`const apiKey = "sk-xxx";` 等。
- **判定方式**：grep/正则检测密码、密钥、API Key 等模式；或使用 truffleHog、git-secrets 等工具。
- **补充参考**：OWASP DevSecOps 硬编码密钥检测，https://github.com/OWASP/www-project-devsecops-verification-standard/blob/main/document/CODE-002-Hardcoded-Secrets-Detection.md

**三、一票否决触发条件**

- **OWASP Top 10 高危**：发现与 A01–A10 任一类别对应的可利用漏洞。
- **硬编码敏感信息**：生产代码或配置中包含硬编码的密码、密钥、API Key、数据库凭证等（CWE-798）。

#### 3.4.2 角色一票否决权

除上述**评分规则**的一票否决（影响环节得分与等级）外，**批判审计员**与**AI 代码教练**均拥有对**迭代结束/达标**的一票否决权：

| 角色 | 一票否决权含义 | 触发条件 | 后果 |
|------|----------------|----------|------|
| **批判审计员** | 对审计闭环收敛的一票否决 | 审计结论为「本轮存在 gap」或「未通过」 | 本轮不收敛，迭代不结束；须修复后再次审计，直至连续 3 轮无 gap |
| **AI 代码教练** | 对全流程迭代达标的一票否决 | 教练输出 `iteration_passed: false` 或等效判定 | 本轮迭代不达标；须按教练建议改进后重新触发教练诊断 |

- 两者**独立行使**：批判审计员否决阶段级审计收敛；教练否决全流程级迭代达标。实施时两者均可触发，任一否决则对应层级不通过。

#### 3.4.3 Epic 级一票否决

当评测对象为 Epic 时，以下任一条件触发时，Epic 综合等级直接降为 L1 或标记「不达标」：

| 条件 | 定义 | 默认阈值 |
|------|------|----------|
| 单阶段 veto 次数 ≥3 | 全流程中环节级一票否决触发次数合计 | 3 次 |
| 需求交付率 < 阈值 | 已通过验收的 Story 数 / Epic 总 Story 数 | 80% |
| 高危漏洞 ≥ 阈值 | OWASP Top 10 高危或 CWE-798 等 | 2 个 |
| 测试通过率 < 阈值 | 通过验收的测试用例数 / 总测试用例数 | 80% |
| 整改≥4 次未通过 Story 数 ≥1 | 全流程中任一 Story 某阶段整改迭代≥4 次仍未通过 | 1 个 |
| 一次通过率<50% | 一次通过即达标的 Story 数 / Epic 总 Story 数 | 50% |
| 整改≥3 次 Story 数 ≥2 | 全流程中整改迭代≥3 次才通过的 Story 数 | 2 个 |
| 致命问题整改≥3 次 Story 数 ≥1 | 含致命问题且整改≥3 次才通过的 Story 数 | 1 个 |

- 阈值可在 scoring/rules 或 _bmad-output/config 的 code-reviewer-score 中配置，默认见上表。
- 与 §3.4 环节级 veto 的关系：环节级 veto 触发时计入「单阶段 veto 次数」；Epic 级 veto 为全流程级判定，与角色一票否决（§3.4.2）独立并存。
- 上述第 5–8 项与现有第 1–4 项**合并**为 Epic 级 veto 条件表；任一条件触发即 veto。

#### 3.4.4 Implementation Gaps 评审规则

**定位**：gaps 作为 SDD 全流程的**核心前置校验项 + 闭环评审核心维度**，权重中高优先级；契合 Speckit 规范驱动开发，量化需求对齐能力。

**双轨评审**：
- **前置完整性评审**（gaps 阶段）：规范全覆盖、gap 定义清晰性、风险预判。得分计入环节 1 补充，占 gaps 阶段总分 40%。
- **后置闭合度评审**（implement、post_impl 阶段）：gaps 闭合率、无新增偏离。implement 阶段占环节 2 的 30%；post_impl 阶段占环节 6 的 50%。

**原子化评分规则**：

| 维度 | 类型 | 扣分/否决规则 |
|------|------|---------------|
| 规范全覆盖 | 前置 | spec/plan 需求在 IMPLEMENTATION_GAPS 中均有对应或显式标注「无偏离」；否则扣分 |
| gap 定义清晰性 | 前置 | 每个 gap 有 id、描述、风险等级、闭合标准；缺失扣分 |
| 风险预判 | 前置 | 高/中风险 gap 有应对方案；缺失扣分 |
| gaps 闭合率 | 后置 | 已闭合 gap 数 / 总 gap 数；<80% 扣分 |
| 无新增偏离 | 后置 | 实现未引入规范未覆盖的偏离；违反扣分 |

**gaps 一票否决项**：

| 条件 | 适用 | 后果 |
|------|------|------|
| 核心需求>20%未映射 | 前置 | 前置 0 分，计入环节级 veto |
| gaps 与规范冲突 | 前置 | 前置 0 分，计入环节级 veto |
| 闭合率<80% | 后置 | 后置 0 分，计入环节级 veto |
| 未定义 gap≥3 | 后置 | 后置 0 分，计入环节级 veto |

#### 3.4.5 多次迭代阶梯式扣分

**原则**：禁止「一次通过和改 10 次通过拿一样分」；量化工业级交付能力，区分玩具级与企业级；可作为强化学习奖励函数。

**强制整改迭代定义**：某阶段审计结论为「未通过」或「存在 gap」，经修复后再次审计，每次「修复→再审计」计为 1 次整改迭代。**不计入豁免**：格式类、表述类等非功能性问题的微调，若审计结论仍为「未通过」，仍计入；仅当审计结论为「通过」后的补充优化（非整改）不计入。

**阶梯扣分规则**：

| 整改迭代次数 | 阶梯系数 | 说明 |
|--------------|----------|------|
| 1 次通过 | 100% | 首次审计即通过 |
| 2 次通过 | 80% | 1 次整改后通过 |
| 3 次通过 | 50% | 2 次整改后通过 |
| ≥4 次 | 0% | 3 次及以上整改 |

**计算方式**：该阶段最终得分 = 原始得分 × 阶梯系数。例如环节 2 原始得分 85 分，3 次通过则 85×0.5=42.5 分。

**问题严重等级差异化**：

| 等级 | 规则 |
|------|------|
| 致命 | 致命问题整改≥3 次→该阶段阶梯系数强制为 0 |
| 严重 | 严重问题整改≥2 次→系数降一档（如 2 次通过按 1 次计 100%，3 次按 2 次计 80%） |
| 一般/轻微 | 仅累计迭代次数，不单独降档 |

**Story/Epic 累计**：单 Story 各阶段整改迭代次数**分别**计入各阶段阶梯扣分；Story 综合分 = Σ(环节得分×阶梯系数×环节权重)。Epic 一次交付通过率 = 一次通过的 Story 数 / Epic 总 Story 数，可作为 Epic 综合分的调节系数（默认权重 0.1，可配置）或独立指标写入报告。

### 3.5 主观环节量化（示例：需求拆解）

至少 **5 项强制项**、**3 项加分项**（可扩展，但 5+3 为最低可数示例）。

**强制项（至少 5 项，每项固定扣分）：**
1. 需求覆盖完整度：核心业务需求无遗漏，每遗漏 1 项扣分；  
2. 边界条件与异常场景标注：明确标注所有边界与异常，每遗漏 1 类扣分；  
3. 模块拆分颗粒度合理性：符合单一职责，无过度拆分或耦合严重，每处不合理扣分；  
4. 技术选型与业务场景匹配：选型匹配业务非盲目跟风，不匹配扣分；  
5. 可测试性与验收标准可验证：需求可拆为可验证验收项，缺失扣分。  

**加分项（至少 3 项，每项固定加分）：**
1. 技术风险与依赖卡点预判，并给出应对方案；  
2. 架构设计可扩展、符合行业最佳实践；  
3. 技术选型最优且与团队/现有栈一致。  

**执行方式**：原子化检查清单 + 明确判定标准；可选 LLM-as-Judge + 抽样人工校准，Judge 与受测模型隔离。

#### 3.5.1 环节 1 完整评分维度表（与环节 2 同级）

环节 1（需求拆解与方案设计）须有与环节 2 同级的完整维度表，便于 scoring/rules 配置与实施。以下为示例结构，实施时以 scoring/rules/spec-scoring.yaml（或等效）为准：

| 维度 | 权重 | 类型 | 扣分/加分规则 |
|------|------|------|---------------|
| 需求覆盖完整度 | 15 | 强制 | 每遗漏 1 项核心需求扣 3 分 |
| 边界条件与异常标注 | 12 | 强制 | 每遗漏 1 类扣 2 分 |
| 模块拆分颗粒度 | 10 | 强制 | 每处不合理扣 2 分 |
| 技术选型与业务匹配 | 12 | 强制 | 不匹配扣 5 分 |
| 可测试性与验收标准 | 11 | 强制 | 缺失扣 4 分 |
| 技术风险预判 | 15 | 加分 | 有预判且给出方案加 5 分 |
| 架构可扩展 | 15 | 加分 | 符合最佳实践加 5 分 |
| 选型与团队栈一致 | 10 | 加分 | 一致加 3 分 |

- 强制项 5 项、加分项 3 项，可扩展；合计 100 分。与 §3.5 的「5+3」示例对应，本表为完整可配置版本。

### 3.6 版本追溯与存储：每次审计闭环输出的数据保存

**原则**：每一个审计闭环迭代的输出都应是最终评分的重要依据；每次评分环节的具体数据**必须**保存到结构化文件中，便于追溯与版本对比。

**存储位置**：`scoring/data/` 或 `_bmad-output/scoring/`（可配置）；与现有 _bmad-output、progress、artifacts 兼容。

**推荐数据结构**：优先 **JSON/JSONL**（支持嵌套、易扩展）；若需表格化分析，可同时导出 **CSV**。每次阶段审计通过后，对应环节得分及明细须**立即追加**到结构化文件。

**每次评分环节的必存字段（schema）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| run_id | string | 单次运行唯一标识（真实开发场景可为 spec/epic-story 标识；评测场景为题目执行 ID） |
| scenario | enum | `real_dev` \| `eval_question`，区分自有真实需求开发 vs 评测题目执行 |
| stage | string | prd \| arch \| epics \| story \| specify \| plan \| gaps \| tasks \| implement \| post_impl \| pr_review |
| path_type | string | 可选。两场景均须走完整五层，path_type 仅用于记录或审计追溯；若记录则应为 `full` |
| phase_score | int | 该阶段对应的评分环节（1–6）得分 |
| phase_weight | float | 该环节权重（如 0.20, 0.25 等） |
| check_items | array | 各检查项通过/不通过、扣分/加分明细（见下） |
| timestamp | string | ISO 8601 |
| model_version | string | 可选，受测模型版本/配置 |
| question_version | string | 可选，评测题目版本（评测场景必填；真实开发可空或填 spec 版本） |
| iteration_count | int | 该 stage 整改迭代次数（0 表示一次通过） |
| iteration_records | array | 每次整改记录：{timestamp, result, severity, note}，result 为 pass/fail，severity 为 fatal/serious/normal/minor |
| first_pass | bool | 是否一次通过（true=阶梯系数 100%） |

**check_items 明细结构**（每项一条）：

| 字段 | 类型 | 说明 |
|------|------|------|
| item_id | string | 检查项标识（与 scoring/rules 中定义一致） |
| passed | bool | 是否通过 |
| score_delta | float | 扣分（负）或加分（正） |
| note | string | 可选，备注 |

**文件命名与格式**：
- 单次运行单文件：`scoring/data/{run_id}.json`，内含各 stage 的 phase_score 与 check_items；
- 或追加模式：`scoring/data/scores.jsonl`，每行一条 JSON，含 run_id、stage、phase_score、check_items 等。
- CSV 导出（可选）：`scoring/data/scores.csv`，列与上表字段对应，便于 Excel/BI 分析。

**与迭代结束标准的关系**：该阶段「得分已写入」指上述结构化文件中已存在该 stage 的完整记录（含 phase_score 与 check_items）。

### 3.7 数据污染防护

- **题目来源与时间隔离**：评测题基于自有业务需求或历史项目原创设计，不采用公开题库，避免进入训练数据。  
- **定期迭代题池**：按周期（如 1–2 个月）更新题池，淘汰旧题，降低被训练数据覆盖风险。  
- **混淆变量校验（可选）**：对题目做逻辑等价改写（变量名、表述替换），若受测模型在改写题上通过率下降 >20%，触发人工复核，不自动淘汰题目。  
- **私有闭卷与评测接口分离**：完整题库不对外公开，仅开放评测接口或脱敏样本，避免被爬取加入训练数据。  

具体操作细节与触发阈值由实施时在 scoring/ 或项目内单独 checklist 中定义；若需与 §3 同文档存放，可增加「§3 附录：数据污染防护操作与触发条件」小节并写入上述四条的操作要点与建议触发条件。

### 3.8 评分规则版本与目录

- 评分规则置于 `scoring/rules/`，可版本化、可插拔；支持多套方案（如 default、algorithm-heavy）按需切换。
- **引用关系**：scoring/rules/*.yaml 通过 `ref` 字段引用 `code-reviewer-config` 中的检查项 ID（item_id），实现规则与审计检查的衔接；与 audit-prompts 的引用在实施时明确并写入实施文档。

### 3.9 题量表述

- 区分「已实现并验证的题数」与「目标题池规模」（如需求拆解目标 20+，全流程题池目标 50+）；在文档与产出中保持一致，避免与实现状态不符。

### 3.10 评分标准权威文档产出要求

**评分标准必须有权威和详细的细节和标准作为文档产出**，不能仅靠 scoring/rules/ 的配置文件；需有**可读、可审计、可版本化**的权威文档。

**产出物**：
- **权威文档路径**：`scoring/docs/SCORING_CRITERIA_AUTHORITATIVE.md`（或等效，如 `docs/scoring/SCORING_CRITERIA_AUTHORITATIVE.md`）。
- **与 scoring/rules/*.yaml 的关系**：权威文档为**人类可读**的完整标准；scoring/rules/*.yaml 为**机器可读**配置；两者须**一致且可追溯**（规则版本号、文档修订日期对应）。

### 3.11 评分规则配置示例

评分规则以 YAML 配置，置于 `scoring/rules/`，与 code-reviewer-config 通过 `ref` 字段衔接。以下为**环节 2（代码开发与工程规范）**的配置示例，其余环节可类比扩展。

**YAML schema 示意**：

```yaml
# scoring/rules/default/implement-scoring.yaml（环节 2 示例）
version: "1.0"
stage: implement
link_stage: [tasks, post_impl]
link_环节: 2

weights:
  base:
    functional_correctness: 30   # 功能正确性
    code_standards: 18           # 代码规范符合度
    exception_handling: 12      # 异常处理完整性
  bonus:
    security_compliance: 20     # 安全合规性
    performance_maintainability: 20  # 性能与可维护性

items:
  - id: func_correct
    name: 功能正确性
    weight: 30
    type: base
    ref: code-reviewer-config#functional_correctness
    deduct: 10  # 每核心逻辑错误扣 10 分
  - id: code_standards
    name: 代码规范符合度
    weight: 18
    type: base
    ref: code-reviewer-config#naming_conventions
    deduct: 2   # 每处违规扣 2 分；无类型注解/关键注释扣 5 分
  - id: exception_handling
    name: 异常处理完整性
    weight: 12
    type: base
    ref: code-reviewer-config#exception_handling
    deduct: 4   # 每遗漏 1 类核心异常扣 4 分
  - id: security
    name: 安全合规性
    weight: 20
    type: bonus
    ref: code-reviewer-config#owasp_top10
    deduct: 10  # 低危问题扣 10 分
  - id: perf_maintain
    name: 性能与可维护性
    weight: 20
    type: bonus
    ref: code-reviewer-config#complexity_maintainability
    deduct: 10  # 合理加 10 分，优秀加满分

veto_items:
  - id: veto_core_logic_error
    ref: code-reviewer-config#veto_core_logic
    consequence: stage_0_level_down
  - id: veto_high_security
    ref: code-reviewer-config#veto_owasp_high
    consequence: stage_0_level_down
  - id: veto_compile_fail
    ref: code-reviewer-config#veto_compile
    consequence: stage_0_level_down
  - id: veto_hardcoded_secret
    ref: code-reviewer-config#veto_cwe798
    consequence: stage_0_level_down
```

**环节 3（测试保障）、环节 4（Bug 修复）YAML schema 示意**：

环节 3 与环节 4 须有与环节 2 同级的 weights、items 结构。以下为简化示意，完整配置见 scoring/rules/test-scoring.yaml、bugfix-scoring.yaml（或等效）：

- **环节 3**：`link_环节: 3`；weights 含 test_coverage、case_quality、regression、edge_cases、performance_tests 等；items 与 code-reviewer-config 的 test 模式对应。
- **环节 4**：`link_环节: 4`；weights 含 fix_correctness、no_regression、fix_efficiency、documentation 等；items 与 code-reviewer-config 的 bugfix 模式对应。

**gaps-scoring.yaml 示意**：

```yaml
# scoring/rules/gaps-scoring.yaml
version: "1.0"
stage: gaps
link_stage: [plan, tasks]
link_环节: 1  # 前置计入环节 1 补充

weights:
  base:
    spec_coverage: 40      # 规范全覆盖
    gap_definition: 30    # gap 定义清晰性
    risk_anticipation: 30  # 风险预判

veto_items:
  - id: veto_core_unmapped
    condition: "核心需求>20%未映射"
    consequence: gaps_pre_0_veto
  - id: veto_gaps_conflict
    condition: "gaps 与规范冲突"
    consequence: gaps_pre_0_veto
```

**iteration_tier 配置示意**（可置于各环节 YAML 或全局 scoring/rules/iteration-tier.yaml）：

```yaml
iteration_tier:
  1: 1.0   # 一次通过 100%
  2: 0.8   # 两次通过 80%
  3: 0.5   # 三次通过 50%
  4: 0     # 四次及以上 0%

severity_override:
  fatal: 3    # 致命问题整改≥3 次→系数 0
  serious: 2  # 严重问题整改≥2 次→降一档
```

- **ref**：指向 code-reviewer-config 中的检查项 ID，实现规则与审计检查的衔接。
- **veto_items**：一票否决项，触发时该环节 0 分且综合等级下调一级。

### 3.12 Code Reviewer Skill 与需求的整合说明

全流程 Code Reviewer Agent 评分体系 Skill 与本需求的整合关系如下：

| 整合项 | 说明 |
|--------|------|
| **6 阶段 ↔ 六环节** | Skill 的 6 个评审阶段（需求拆解与方案设计、代码开发、测试保障、Bug 修复、跨模块集成、Story 交付验收）与本需求六环节一一对应；阶段权重 20/25/25/15/10/5 与本需求 §3.3 一致。 |
| **触发时机** | Skill 的触发（Story 状态变更、MR 创建、Speckit 测试完成、Bug 状态变更、Epic 待验收）与本需求 §2.1、§2.2 的阶段与迭代结束标准对应；自动触发与手动触发均可写入 scoring 存储。 |
| **维度换算** | Skill 环节 2 的 30/18/12/20/20 分制对应本需求环节 2 的 25 分权重；换算公式：`环节得分 = (分项得分之和 / 100) × 25`，或由 scoring/rules 定义等效换算。 |
| **输出与 schema 衔接** | Skill 输出的分项得分、问题清单、一票否决检查结果，须写入 §3.6 定义的 scoring 存储 schema；item_id、veto 与 scoring/rules 的配置一致，便于追溯与短板定位。 |
| **迭代结束标准** | Skill 单阶段评审通过且得分写入后，对应本需求 §2.2 的该阶段迭代结束；Epic 综合评分报告对应全流程综合得分与 L1–L5 等级认证。 |

**Epic 综合报告六部分结构**（当评测对象为 Epic 时）：

1. 摘要：Epic 综合分、L1–L5 等级、是否达标、Epic 级 veto 检查结果；
2. 各 Story 得分明细：Story ID、综合分、环节得分、权重；
3. 环节短板：环节 1–6 最低分项及改进建议；
4. 一票否决检查结果：环节级 veto 触发记录、Epic 级 4 项条件检查；
5. L1–L5 等级与得分区间说明；
6. 改进建议：基于短板与 veto 的优化方向。

**触发模式表**（全链路 Skill 编排）：

| 事件类型 | 触发方式 | 对应 stage/环节 |
|----------|----------|-----------------|
| stage 审计产出完成 | 自动 | 该 stage 对应环节 |
| Story 状态变更 | 自动（可配置） | 环节 1–6 |
| MR 创建 | 自动（可配置） | 环节 2–6 |
| Epic 待验收 | 手动或自动 | 环节 6、Epic 综合 |
| 用户显式请求「全链路评分」 | 手动 | 全环节 |

- 实施时，Code Reviewer Skill 作为本需求「全流程审计闭环的重要输出与迭代结束标准」的**执行载体**，其配置与输出须与 scoring/rules、scoring 存储、权威文档保持一致。

### 3.13 全链路 Code Reviewer Skill 的独立与引用关系

将「全链路 Code Reviewer 质量审计与生命周期评分」单独作为**新的 Skill**（如 `bmad-code-reviewer-lifecycle` 或等效命名），与现有 `receiving-code-review`、`requesting-code-review` 及 Cursor Task 调度的 `code-reviewer` **区分**。

**独立 Skill 边界**：
- **全链路**：覆盖 BMAD 五层、六环节、各 stage 审计与评分写入；负责**编排**（何时触发、传参、解析产出、写入 scoring 存储）。
- **单点评审**：由现有 code-reviewer（Cursor Task/agents）、audit-prompts、code-reviewer-config 提供；新 Skill **引用**而非重复实现。

**引用关系**（新 Skill 通过引用使用以下组件）：

| 引用组件 | 职责 | 引用方式 |
|----------|------|----------|
| code-reviewer | 执行各 stage 审计 | Cursor Task 调度，按 stage 传 mode 与 prompt_template |
| audit-prompts | 各 stage 审计提示词 | audit-prompts-prd.md、audit-prompts-arch.md、audit-prompts-code.md、audit-prompts-pr.md 等 |
| code-reviewer-config | 多模式配置（prd/arch/code/pr） | config/code-reviewer-config.yaml，按 mode 读取 dimensions、pass_criteria |
| scoring/rules | 解析规则、item_id、veto_items | scoring/rules/*.yaml，用于解析审计产出并映射环节得分 |

**与 _bmad-output/config 的 code-reviewer-score 关系**：

- scoring/rules/*.yaml 为**权威规则定义**（完整 check_items、veto_items、权重）。
- _bmad-output/config 的 code-reviewer-score 为**项目级覆盖**，可配置 weight、block_rules、code_style、security_scan、test_coverage 等，**不得替代** scoring/rules 的完整定义。
- 实施时：优先加载 scoring/rules；若 _bmad-output/config 存在 code-reviewer-score，则覆盖对应字段；未配置字段使用 scoring/rules 默认。

**避免重复实现**：新 Skill 不重新实现审计逻辑；仅定义触发时机、与 stage 的映射、输出解析规则、scoring 写入格式。审计执行由被引用的组件完成。

**与 speckit-workflow、bmad-story-assistant 协同**：speckit-workflow 的 clarify/checklist/analyze 嵌入各审计闭环；bmad-story-assistant 的审计步骤调度 code-reviewer；新 Skill 作为全链路评分编排，在上述流程的**各 stage 审计通过后**触发评分解析与写入，三者协同。实施时需明确触发链：stage 完成 → 调用新 Skill 的「解析并写入」逻辑。

### 3.14 AI 代码教练（AI Code Coach）

**定位**：用资深工程师视角，还原真实工业级开发全流程，精准定位 AI 代码模型的能力短板，设计可落地的评测/优化方案。作为「现有全流程各审计闭环的重要输出和迭代结束标准」的**承载者**。

**与全流程 Code Reviewer 的关系**：非替代、非并列。教练**消费** Reviewer（及全链路 Skill）的产出，做汇总、解读与优化方案设计；不重复执行审计。

**职责边界**：

| 职责 | 执行方 | 说明 |
|------|--------|------|
| 评审 | 全流程 Code Reviewer / 全链路 Skill | 各 stage 审计、检查项通过/不通过、得分写入 |
| 教练 | AI 代码教练 | 解读评分、定位短板、给出改进方向 |
| 优化方案设计 | AI 代码教练 | 设计题池补充、权重调整、检查项细化等可落地方案（建议型，非直接执行） |

**人格定义**：资深工程师视角、工业级标准、可落地导向；输出风格：精准、可执行、无模糊表述。

**技能配置**：
- **引用 Skill**：全链路 Code Reviewer Skill（必引，优先级最高）；speckit-workflow（可选）；bmad-story-assistant（可选）。
- **fallback**：若全链路 Skill 不可用，可降级为「仅解读既有 scoring 存储中的已有数据」，不执行新审计。

**工作流定制**：
- **与 BMAD 五层、六环节、迭代结束标准的衔接**：教练在各 stage 审计完成且得分写入后可被触发；输入为 scoring 存储中 run_id 对应数据；输出为综合诊断报告。
- **触发时机**：手动触发（用户请求「对本轮迭代做教练诊断」）或阶段式触发（post_impl 完成后自动可选触发）；**可选**，不强制，满足轻量化三原则。
- **输出格式**：含 `summary`、`phase_scores`、`weak_areas`、`recommendations`、`iteration_passed` 等字段；支持 JSON 与 Markdown。
- **一票否决权**：教练输出 `iteration_passed: false` 时，对全流程迭代达标行使一票否决（见 §3.4.2）。

**禁止表述**：教练产出不得以「面试」为主导；仅可保留「成果可用于对外能力说明等场景」等中性表述。

## §4 与主流评测体系的关系

- **底层兼容**：代码功能正确性可沿用 Pass@k 逻辑；存量项目集成可对齐 SWE-bench 思路。
- **全链路升级**：本体系覆盖从需求到交付的全流程，解决单点评测「分数高但落地差」的痛点。
- **工业级落地**：面向企业落地与模型迭代指导，可与 HumanEval/SWE-bench 组合使用，不替代、不否定。

---

## §5 实施步骤（MVP 与迭代）

- **MVP**：将现有审计闭环转化为标准化检查清单；确定权重与评分表模板；用 1–2 个主流模型做单次实测并记录标杆分。
- **依赖与前置**：受测模型接入方式、可选 Judge 模型与人工抽样量、运行环境（Python/Node、API 等）。
- **后续迭代**：自动化评测流水线、多模型批量评测、题池扩展、定期对标表复核。

---

## §6 风险与假设

- **假设**：权重与检查项经讨论确定后在一段时间内稳定；Judge 与受测模型可隔离；题目版本与 run_id 可正确记录。
- **风险**：主观环节即使用清单仍可能存在边界歧义；对标表需定期维护。缓解：明确判定标准、双盲或抽样校准；对标表注明基准日期并约定复核周期。

## §7 收敛声明

本需求分析经**七轮** Party-Mode 辩论（第一轮 200 轮 + 第二轮 200 轮 + 第三轮 200 轮 + 第四轮 200 轮 + 第五轮 200 轮 + 第六轮 200 轮 + 第七轮 200 轮，合计 1400 轮），批判审计员发言占比 >70%，最后 3 轮无新 gap 后收敛，可直接用于：

1. 指导「全体系评分评级」的落地设计与实施任务分解；  
2. **作为现有全流程各审计闭环的重要输出与迭代结束标准**，与 BMAD 五层（Layer 1 PRD/Arch、Layer 2 Epics、Layer 3 Story、Layer 4 speckit spec/plan/gaps/tasks/implement、Layer 5 实施后审计与 PR 审核）各阶段审计协同；  
3. **区分自有真实需求开发与评测题目执行**两种场景，确保改进**增强**真实开发流程而非拖沓（轻量化三原则）；  
4. **每次审计闭环输出的数据**须保存到结构化文件（JSON/JSONL/CSV），schema 见 §3.6；  
5. **评分标准权威文档**须产出（scoring/docs/SCORING_CRITERIA_AUTHORITATIVE.md），含详细检查项、扣分/加分规则、一票否决等；  
6. 与现有 BMAD+Speckit 审计闭环及 scoring/、docs/BMAD 文档协同，不替代现有环节审计，仅增加统一出口与等级认证；  
7. 本体系成果可用于模型迭代指导、能力评估与对外能力说明等场景；  
8. **AI 代码教练**作为全流程审计闭环的重要输出与迭代结束标准的**承载者**，与全链路 Code Reviewer Skill 协同；教练消费 Reviewer 产出，做汇总、解读与优化方案设计；  
9. **全链路 Code Reviewer Skill**独立于现有单点 code-reviewer，通过引用 code-reviewer、audit-prompts、code-reviewer-config、scoring/rules 完成全链路编排，与 speckit-workflow、bmad-story-assistant 协同；
10. **Epic 综合评分**（§3.2.1）与 **Epic 级一票否决**（§3.4.3）支持多 Story 聚合；分阶段评分含环节 1 完整维度表（§3.5.1）、环节 3–6 YAML schema；Epic 综合报告六部分结构、触发模式表、_bmad-output/config 与 scoring/rules 关系均已定义；
11. **Implementation Gaps** 作为核心前置校验项与闭环评审维度（§3.4.4）；**多次迭代阶梯式扣分**（§3.4.5）与 Epic 级 veto 扩展（§3.4.3 第 5–8 项）；schema 扩展 iteration_count、iteration_records、first_pass。
