# Party-Mode 200 轮辩论 — Code Reviewer 分阶段评分与 Epic 综合评分需求更新要点

**产出说明**：本文档由 Party-Mode 辩论产出。针对需求分析文档 `REQUIREMENTS_AI代码评测体系_全流程审计闭环输出与迭代标准_需求分析.md`，聚焦**全流程 Code Reviewer Agent 评分体系 Skill 定义修改方案**与现有需求文档的整合与优化，进行 200 轮深度辩论，**批判审计员发言占比 >70%**，**最后 3 轮无新 gap** 后收敛。

**用户 Skill 定义方案要点**：
- **6 阶段**：需求拆解与方案设计(20%)、代码开发(25%)、测试保障(25%)、Bug修复(15%)、跨模块集成(10%)、Story交付验收(5%)

- **配置项**：.bmad/config.yaml 中 code-reviewer-score（weight、block_rules、code_style、security_scan、test_coverage）
- **输出规范**：各阶段 Markdown 报告模板、Epic 综合报告六部分结构

## 核心结论（逐议题）

### 议题 1：分阶段评分标准与需求文档的衔接

| 结论项 | 内容 |
|--------|------|
| **6 阶段与六环节对应** | 用户方案的 6 阶段与需求 §3.3 六环节**一一对应**；命名差异（如「代码开发」vs「代码生成与工程规范」）在需求中采用**统一术语表**，实施时以需求为准。 |
| **各阶段 100 分制与 §3.11 映射** | 每环节内部采用**100 分制子维度**，再按环节权重（20/25/25/15/10/5）折算入综合分。scoring/rules/*.yaml 的 `weights`、`items` 与用户方案维度**一一映射**；环节 3–6 的 YAML 配置**须在需求中补充示例**（至少环节 3 测试保障、环节 4 Bug 修复的 schema 示意）。 |
| **环节 1 完整维度表** | 需求 §3.5 仅有「5 项强制+3 项加分」示例，**不足以**与环节 2 同级的完整维度表对标。**须扩展**：新增「§3.5.1 环节 1 完整评分维度表」，含至少 8 个分项（覆盖强制+加分）、每项权重、扣分/加分规则，与环节 2 的 implement-scoring.yaml 结构同级。 |
| **阶段→评分环节表** | §2.1 表 B 已覆盖；用户方案阶段与表 B 的「对应评分环节」列**一致**，无需修改。 |

### 议题 2：Epic 综合评分体系与需求文档的衔接

| 结论项 | 内容 |
|--------|------|
| **Epic 综合分计算** | 需求 §3.2 四层架构仅有「综合得分 = Σ(环节得分 × 对应权重)」，**未覆盖 Epic 级**。**须新增 §3.2.1 Epic 综合评分**：Epic 综合分 = Σ(Story 综合分 × Story 权重)，Story 权重可配置（默认均等）；单 Story 综合分 = Σ(环节得分 × 环节权重)。折算公式写入需求，便于实施与审计。 |
| **Epic 一票否决** | 需求 §3.4 仅有**环节级**一票否决（该环节 0 分、等级下调）。用户方案的 Epic 级 veto（3 次及以上单阶段 veto、需求交付率<80%、高危漏洞≥2、测试通过率<80%）**须新增 §3.4.3 Epic 级一票否决**。与 §3.4 环节级 veto 的关系：环节级 veto 触发时计入「单阶段 veto 次数」；Epic 级 veto 为**全流程级**，触发时 Epic 综合等级直接降为 L1 或标记「不达标」。 |
| **L1–L5 一致性** | 用户方案与需求 §3.2 的 L1–L5 定义**一致**（L5 90–100、L4 80–89、L3 60–79、L2 40–59、L1 0–39）。得分区间**已统一**，无需修改。Epic 综合分映射到 L1–L5 时，采用相同区间。 |

### 议题 3：Skill 定义与配置的标准化

| 结论项 | 内容 |
|--------|------|
| **.bmad/config.yaml 与 scoring/rules 关系** | 用户方案的 `.bmad/config.yaml` 中 `code-reviewer-score` 与 `scoring/rules/*.yaml` 的关系：**引用而非替代**。.bmad/config.yaml 为**项目级开关与覆盖**（如启用/禁用评分、覆盖权重、block_rules 快捷配置）；scoring/rules/*.yaml 为**权威规则定义**（完整 check_items、veto_items、权重）。实施时：scoring/rules 为默认；.bmad/config.yaml 的 code-reviewer-score 可覆盖部分字段（weight、block_rules 等），**不得替代** scoring/rules 的 check_items 与 veto_items 完整定义。 |
| **全链路 Skill 触发模式** | §3.13 的「编排」职责与用户方案「事件监听、自动触发、手动触发」的对应：**编排** = 定义何时触发；**事件监听** = 监听 stage 完成、Story 状态变更、MR 创建等；**自动触发** = 事件满足时自动调用解析与写入；**手动触发** = 用户显式请求「对本 Epic 做全链路评分」。需求 §3.13 须**补充触发模式表**：事件类型 → 触发方式（自动/手动）→ 对应 stage/环节。 |
| **Skill 输出与 schema、AI 教练衔接** | Skill 输出的阶段报告、Epic 报告须与 §3.6 schema 兼容；阶段报告模板（Markdown）与结构化 JSON 双格式；Epic 综合报告六部分结构（摘要、各 Story 得分、环节短板、一票否决检查、综合等级、改进建议）须写入需求；AI 代码教练输入 = scoring 存储 + Epic 报告，输出格式含 `iteration_passed` 等，与 §3.14 一致。 |

### 议题 4：与现有审计闭环、迭代结束标准的关系

| 结论项 | 内容 |
|--------|------|
| **承载关系** | 用户方案作为「全流程审计闭环的重要输出和迭代结束标准」的**承载**，与 §2.2 各阶段迭代结束标准**衔接**：各 stage 审计通过且得分写入 → 该阶段迭代结束；Epic 综合报告产出且无 Epic 级 veto → 全流程迭代达标。§3.4.2 角色一票否决（批判审计员、AI 代码教练）**独立于** Epic 级 veto：角色否决 = 迭代不收敛/不达标；Epic 级 veto = 综合等级降级或标记不达标。两者**并存**，不冲突。 |
| **禁止「面试」** | 全文无「面试」主导表述；保持「成果可用于对外能力说明等场景」中性表述。 |

### 修改 1：§3.2 四层架构 — 新增 §3.2.1 Epic 综合评分

**位置**：§3.2 第四层（L1–L5）之后，§3.3 权重依据之前

**新增 §3.2.1**：

```markdown
#### 3.2.1 Epic 综合评分

当评测对象为 Epic（含多个 Story）时，采用 Epic 综合评分：

- **单 Story 综合分**：综合得分 = Σ(环节得分 × 环节权重)，0–100 分。
- **Epic 综合分**：Epic 综合分 = Σ(Story 综合分 × Story 权重)，Story 权重可配置（默认均等）。
- **L1–L5 映射**：Epic 综合分采用与单次运行相同的得分区间（L5 90–100、L4 80–89、L3 60–79、L2 40–59、L1 0–39）。
```

### 修改 2：§3.4 一票否决项 — 新增 §3.4.3 Epic 级一票否决

**位置**：§3.4.2 角色一票否决权之后，§3.5 主观环节量化之前

**新增 §3.4.3**：

```markdown
#### 3.4.3 Epic 级一票否决

当评测对象为 Epic 时，以下任一条件触发时，Epic 综合等级直接降为 L1 或标记「不达标」：

| 条件 | 定义 | 默认阈值 |
|------|------|----------|
| 单阶段 veto 次数 ≥3 | 全流程中环节级一票否决触发次数合计 | 3 次 |
| 需求交付率 < 阈值 | 已通过验收的 Story 数 / Epic 总 Story 数 | 80% |
| 高危漏洞 ≥ 阈值 | OWASP Top 10 高危或 CWE-798 等 | 2 个 |
| 测试通过率 < 阈值 | 通过验收的测试用例数 / 总测试用例数 | 80% |

- 阈值可在 scoring/rules 或 .bmad/config.yaml 的 code-reviewer-score 中配置，默认见上表。
- 与 §3.4 环节级 veto 的关系：环节级 veto 触发时计入「单阶段 veto 次数」；Epic 级 veto 为全流程级判定，与角色一票否决（§3.4.2）独立并存。
```
