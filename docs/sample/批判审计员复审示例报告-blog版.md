# 第 3 轮复审是如何拿到“完全覆盖、验证通过”的

> 历史样例说明（2026-04-13）：本文是基于更早一轮 reviewer 复审报告整理出的 blog 化示例，保留当时的工具名、路径和审计证据语境。若与当前 reviewer 运行事实有出入，以 2026-04-13 的 shared reviewer core / host carrier / `runAuditorHost` close-out 为准。

很多审计报告的问题，不在于结论错，而在于读者根本看不出结论是怎么来的。

“通过了”三个字很容易写，但如果中间没有足够的证据链、没有对抗性质疑、没有对上一轮问题的回查，那这类通过就很像一种气氛，而不是一种判断。

这份示例报告想展示的，正是另一种写法：**不是先宣布结论，而是先让读者一步步看到，为什么这次可以判定通过。**

这是一份第 3 轮的 §5 执行阶段审计报告。它的价值不在“又过了一轮”，而在于它清楚回答了三个关键问题：

1. 上一轮要求修的内容，真的修了吗？
2. 全文重新审一遍之后，还有没有新的结构性漏洞？
3. 如果站在最挑剔、最不信任结果的角度，再去质疑一次，还能不能守住结论？

## 先看模型选择：这份报告由谁来写，为什么这样配

先把基础信息摆出来，是为了让读者知道这份报告的“观察视角”来自哪里。

| 项目 | 值 |
| --- | --- |
| 配置来源 | `.claude/agents/code-reviewer.md` |
| 指定模型 | `inherit`（来自 frontmatter，继承主 Agent 模型） |
| 选择依据 | 子代理 YAML 中 `model: inherit`；可改为 `fast` 以节省审计成本（若环境支持） |

这一段看起来像元数据，但其实很重要。因为它告诉读者，这不是凭空冒出来的一份“主观意见”，而是一个带有明确调度来源、明确模型继承规则的审计产物。

## 第一件事不是重审全文，而是先验上一轮的 4 个修复项

这是很多报告最容易偷懒的地方。

上一轮说“已经修复”，这一轮最容易直接接受这个前提，然后跳到新的大结论。但批判审计员不会这么做。**真正严谨的复审，第一步永远是回头看：上一轮的修复，到底有没有真实落地。**

这一次，第 2 轮留下了 4 个修复项，结果是 **4/4 全部通过**。

### 1. T2a 范围表补充：不是“差不多对”，而是数字逐项对齐

这里最关键的不是“表格补了”，而是补得是否完整，数字是否经得起交叉验证。

| 验证点 | 实际内容 | 判定 |
| --- | --- | --- |
| `.cursor/skills/` 行 | 第 375 行：「`.cursor/skills/ | 9 | 21 | ...T2 删除 skills/ 后，.cursor/skills/ 中幸存的副本其内部交叉引用...将失效`」 | PASS |
| `scoring/` 行 | 第 380 行：「`scoring/ | 1 | 3 | EVAL_AGENT_BMAD_FLOW_IMPROVEMENT.md`」 | PASS |
| 总数更新 | 第 369 行：「`52+ 个文件共 137+ 处引用`」 | PASS |
| 交叉验证 | 实际 grep 验证：`19+9+7+13+1+1+1+1=52` 文件，`56+21+25+28+2+1+3+1=137` 引用 | PASS |

这类验证的价值，在于它不是“我看上去差不多”，而是**数字能自己闭环**。表里写 52，就真能从各目录分项加回 52；表里写 137，就真能从引用总数加回 137。读者读到这里，才会开始相信后面的“通过”不是空话。

### 2. T2a 替换命令：不是能跑就行，而是要防误替换

第二个修复点聚焦在替换命令本身是否足够安全。

| 验证点 | 实际内容 | 判定 |
| --- | --- | --- |
| 负向断言 | 第 389 行：`perl -pi -e "s|(?<![./])skills/$skill/|.cursor/skills/$skill/|g"` | PASS |
| `sed` 警告 | 第 394 行明确标注：「`sed 不支持负向断言，实际执行建议使用 perl -pi -e`」 | PASS |

这里的重点，其实是作者有没有把“能替换”升级为“能安全替换”。

只写一条 `sed` 很容易，难的是你有没有意识到：如果没有负向断言，批量替换就有机会伤到已经带前缀的路径。审计员在这里判 PASS，不是因为命令长得专业，而是因为它明确处理了这个边界。

### 3. `code-review` 的 superpowers 依赖：是不是公共技能，不能靠感觉分

这是一个很典型的分类问题。

表面上看，它依赖 `superpowers:code-reviewer`，那它到底是“平台专用”，还是“接口通用但实现依赖插件”？

| 验证点 | 实际内容 | 判定 |
| --- | --- | --- |
| 子代理依赖栏 | 第 128 行完整标注：「使用 Task tool with `superpowers:code-reviewer` type...但 `superpowers:code-reviewer` 子代理类型来自 superpowers 插件...Cursor 中为 superpowers 扩展，Claude Code CLI 中为 oh-my-claudecode」 | PASS |
| 判定依据栏 | 第 129 行：「使用通用 Task tool 接口（非平台专用调度），按 §1.2 四项标准属公共；插件依赖不改变调度机制本身的通用性」 | PASS |

这段判定很有代表性，因为它不是围绕“名字像不像平台专用”来判断，而是回到标准本身：**看调度接口是否是平台专用，而不是看背后插件来自哪里。**

### 4. §7.2 的 `wc -l` 验证：防止“完整复制”变成主观感受

最后一个修复点，看起来最小，实际上很关键。

| 验证点 | 实际内容 | 判定 |
| --- | --- | --- |
| `wc -l` 验证说明 | 「验证方式：手动 `wc -l` 对比 Cursor 版与 Claude 适配版中同名模板 ID 对应的段落行数，确认比值 `>= 0.9`」 | PASS |

为什么这个点重要？因为“整段复制”“不要概括”“不要摘要”这些要求，如果没有量化检查，很容易沦为一种态度表达。加入 `wc -l` 之后，它就从风格要求，变成了可检查的规则。

**这一轮回查的结论很明确：4 个修复点，4 个都真正落地了。**

## 真正的复审，从这里才开始：不是看 1 个点，而是看整份文档能不能站住

上一轮的问题补齐之后，接下来不是宣布胜利，而是进入全面重审。这里最重要的一点是：**批判审计员不是按“章节顺序”读文档，而是按“失效风险”来拆解检查维度。**

这次完整复审，一共覆盖了 6 大维度：

1. 结构完整性
2. 数据一致性
3. 分类逻辑
4. 可操作性
5. 禁止词
6. §7 自洽性

### 结构完整性：先确保这不是一份“排版完整、逻辑残缺”的文档

很多文档表面上节数很齐，但真正的问题是跳号、断层、任务编号缺失、章节职责不清。

这次检查的结果是：

| 检查项 | 结果 | 判定 |
| --- | --- | --- |
| 文档层级完整性 | 一（目录定位）→ 二（现状分析）→ 三（逐技能分析）→ 四（分类汇总）→ 五（任务列表）→ 六（执行优先级）→ 七（提示词模板完整性规则）→ 八（注意事项），共 8 节，无跳号 | PASS |
| 各节目的说明 | 每节均有标题 + 正文首段说明用途，§7 有 blockquote 总结 | PASS |
| 任务编号连续性 | `T1, T2, T2a, T3, T4, T5, T6, T7, T8, T9`，共 10 个任务，无遗漏 | PASS |

结构完整性为什么要先看？因为如果文档骨架本身不完整，后面所有“细节正确”都没有意义。你甚至不知道缺的是哪一块。

### 数据一致性：同一个数字，不能在三处长成三个版本

这是这份报告最强的部分之一。

审计员不是只看“某张表内部对不对”，而是强制把 §3、§4、§5、§6 之间的数字互相拉通：分类统计是否一致、步骤和任务是否一致、优先级和依赖图是否一致。

例如分类统计：

| 检查项 | §3 值 | §4.1 值 | 一致? |
| --- | --- | --- | --- |
| 公共技能数 | 7 | 7 | PASS |
| Cursor 专用（已适配） | 1 | 1 | PASS |
| Cursor 专用（待适配） | 7 | 7 | PASS |
| 总计 | 15 | 15 | PASS |

再比如执行步骤与任务映射：

| 检查项 | §4.2 | §5 | 一致? |
| --- | --- | --- | --- |
| 步骤1：清空公共副本 7 个 | 第 279 行 | T1：删 7 个 | PASS |
| 步骤2：移 8 个 Cursor 专用 | 第 280 行 | T2：删 `skills/` 中 8 个 | PASS |
| 步骤3：新增 7 个 Claude 适配 | 第 281 行 | T3-T9：7 个适配任务 | PASS |
| T2a（引用更新） | §4.2 未单列 | §5 独立列出 | PASS |

这类对照最能暴露“文档后半段忘记更新”的问题。只要有一个总数没同步、一个步骤忘了映射、一个任务依赖写岔，整份文档就不再可信。

### 分类逻辑：不是“看起来像公共”，而是要拿真实文件来验

这一步非常像法庭证据，而不是写作修辞。

审计员直接抽取真实技能文件做交叉验证：

| 技能文件 | 文档分类 | 实际内容验证 | 判定 |
| --- | --- | --- | --- |
| `skills/code-review/SKILL.md` | 公共 | 无 Cursor Task 关键词；使用 `superpowers:code-reviewer` 通用 Task tool 接口 | PASS |
| `skills/bmad-orchestrator/SKILL.md` | 公共 | 无 Cursor Task 或 `.cursor/agents/` 关键词；frontmatter 标注多平台 | PASS |
| `skills/bmad-customization-backup/SKILL.md` | 公共 | 无 Cursor Task 关键词；`~/.cursor/skills/` 仅为路径描述 | PASS |
| `.cursor/skills/speckit-workflow/SKILL.md` | Cursor 专用 | 明确依赖 Cursor Task 与 `.cursor/agents/code-reviewer.md` | PASS |

如果没有这一步，所谓“分类正确”就很容易退化成作者自己的直觉分类。只有拉真实文件进来，读者才会相信“分类逻辑”不是拍脑袋。

### 可操作性：一份任务文档，最终还是要有人真的照着做

这是审计里非常接地气的一环。

批判审计员在这里切换成“执行工程师”视角，不再问“写得对不对”，而是问“我真的能照着做吗”。

结果是 T1 到 T9 全部通过：

| T4 | ✓ | N/A | ✓ | PASS |

这个维度的意义在于，很多文档在“分析层”写得很漂亮，但一到执行层就变成了口号。审计员在这里确保：每个任务不仅说得通，而且做得下去。

### 禁止词检查：不是形式主义，而是防止文档偷偷把问题往后拖

搜索模式是：

`后续迭代|待定|酌情|视情况|可选|可考虑|暂不|将来|以后`

结果：`No matches found`

这一步看似机械，实则非常必要。因为太多“暂时通过”的文档，真正的问题不是逻辑错，而是它把不确定性藏在措辞里，把该解决的问题悄悄延后。没有这些词，至少说明作者没有靠模糊措辞逃避决断。

### §7 自洽性：规则、清单、示例、错误类型，四个地方要能互相扣住

这一块的意思很简单：如果 §7.1 说了规则，那 §7.2 应该能检查它，§7.3 应该能示范它，§7.4 应该能反例化它。

换句话说，**规则不能只存在于规则区，它必须贯穿到检查、案例和错误识别里。**

这轮复审确认：这四层之间是扣得上的。

## 真正拉开差距的地方：不是“审一遍”，而是再从四个高压视角重打一遍

如果说前面的六大维度是在看“文档整体是否成立”，那么后面的四个视角，就是在看“这份通过是否只是表面上的通过”。

这也是为什么这份报告里，批判审计员的篇幅占比会超过 70%。因为在高质量复审里，真正有价值的不是复述事实，而是主动制造压力。

### 视角一：对抗视角

这里最有意思的是，批判审计员不再站在作者一边，而是假装自己是一个会被这份文档坑到的工程师。

于是问题变成：

- `sed` 和 `perl` 混写，会不会让我困惑？
- `diff -rq` 在 Windows PowerShell 下是不是不成立？
- `.claude/agents/` 先改到 `.cursor/skills/`，中间过渡期会不会有逻辑问题？
- T3-T9 的提示词完整性约束，到底够不够具体？

这些问题的共同特点是：它们都不质疑“意图”，而是质疑“执行现场会不会踩坑”。

最后的结论是，这些点都成立为信息项，但没有形成阻塞性 gap。换句话说，**它们值得被看见，但不足以推翻这份文档的可执行性。**

### 视角二：遗漏视角

这一部分看的是失败场景。

不是问“正常做是否可行”，而是问：

- 如果 T2 后 Cursor 突然识别不了 `.cursor/skills/` 怎么办？
- 如果 T2a 批量替换替错了怎么办？
- 如果 T3-T9 适配过程中发现原始 Cursor 版技能自己就有 bug，怎么办？

这类问题很重要，因为很多文档只描述“成功路径”，但真正决定能否上线的，往往是失败路径是否被考虑过。

这次复审给出的答案是：

- 回退路径存在，而且有 `git commit` 作为前置保护。
- 批量替换存在负向断言、`git diff --stat` 和全量搜索三层防护。
- Cursor 版技能本身有 bug 的场景，属于独立修复流程，不属于本文档职责范围。

这是一种很健康的边界感：**该负责的负责到底，不该负责的明确切开。**

### 视角三：一致性视角

这部分像是一场数字审判。

审计员把文档中的关键数字一个一个拎出来，要求它们在所有上下文里都自洽：

- 7 个公共技能
- 8 个 Cursor 专用
- 15 个技能总数
- 52+ 个受影响文件
- 137+ 处引用
- 19 / 56+、9 / 21、7 / 25+、13 / 28+、1 / 2、1 / 1、1 / 3、1 / 1
- 2112 行的 Claude 版 `bmad-story-assistant`

为什么这一步重要？因为数字最不会给面子。段落可以写得模糊，表格可以看起来完整，但数字一旦前后打架，读者马上就知道这份文档没收口。

而这份报告的强处就在于：**这些数字不仅内部一致，还能和实际 grep 结果对上。**

### 视角四：边界视角

最后一个问题，往往是最容易被忽略的：

如果将来再新增一个技能，这份文档的分类标准还够用吗？

这个问题其实是在测一件更深的事：它到底是“只够解释这一次”，还是“有能力解释下一次”。

报告给出的答案是积极的。

因为 §1.2 的判定框架不是写死在某个工具名上，而是落在一个更高层的原则上：**是否存在平台专用调度机制。** 只要这个原则还在，新机制也可以被纳入同一套判断框架。

这说明这份文档不只是完成了一次分类，更搭出了一个可扩展的分类规则。

## 最后的 6 个质疑都没有形成阻塞项，这意味着什么？

复审最后还做了一轮“假 100 轮”风险扫描，列出了 6 个依然值得讨论的点：

| 质疑项 | 结论 |
| --- | --- |
| §7.4 未单列“仅改执行体”错误类型 | 信息项 |
| T2a `sed/perl` 混用 | 信息项 |
| bash 命令假设 Unix 环境 | 信息项 |
| T3 “约 10 个文件”是估数 | 信息项 |
| §4.2 写“移到”，T2 写“删除” | 信息项 |
| 依赖图中 T5 箭头方向 | 信息项 |

这里最值得注意的不是“还有 6 个问题”，而是**这 6 个问题都没有升级为阻塞 gap**。

也就是说，这份文档不是完美无瑕，而是已经达到另一种更成熟的状态：还有细节可优化，但这些细节不会动摇整体正确性、完整性和执行性。

这其实比“假装没有任何问题”更可信。

## 为什么这份报告可以判定“完全覆盖、验证通过”

最后把证据收回来，其实结论已经很自然了。

它之所以能通过，不是因为某一张表写得好，也不是因为某一个任务描述得细，而是因为这几个层次同时成立：

- 上一轮的 4 个修复项，已经真实落地。
- 全文在结构、数据、分类、执行、措辞、自洽性六个维度上站得住。
- 再从对抗、遗漏、一致性、边界四个方向施压，也没有打出新的阻塞性 gap。
- 所有关键数字都能回到真实文件和 grep 结果上。

因此，这一轮的正式结论是：

> 「完全覆盖、验证通过」

不过，这个结论仍然保留了一个严谨的尾注：

> 本轮无新 gap，第 3 轮（累计 1/3 轮无 gap）；仍需继续第 4 轮、第 5 轮审计，才能满足 3 轮无 gap 收敛。

这句尾注非常重要，因为它提醒读者：**“通过”不等于“流程结束”。**

在高标准审计里，通过只是说明这一轮站住了，不代表可以跳过后续的收敛要求。

## 这份 blog 版示例，真正想示范的写法是什么

如果把这份内容当成一篇写给团队成员、项目维护者或未来审计者的 blog，它真正示范的，不只是“怎么写一份通过报告”，而是：

- 怎么先交代上一轮问题，而不是直接跳到结论。
- 怎么把“通过”拆解成读者能理解的证据链。
- 怎么让表格、数字、grep、对抗质疑和边界讨论彼此支撑。
- 怎么在保持正式性的同时，让读者愿意一路读下去。

一份好的批判审计员报告，不应该只服务于“归档”，也应该服务于“说服”。

读者读完之后，不只是知道这次通过了，更应该知道：
