## 2. 逐项详情

### 2.1 Story / plan / GAPS 覆盖

- **T1**（run 命令与参数解析）：eval-questions-cli.ts 实现 run 子命令，解析 --id、--version、--reportPath；校验必填、version ∈ {v1,v2}；loadManifest 按 id 查找；题目不存在时输出「题目 {id} 在版本 {version} 中不存在」。
- **T2**（报告来源）：支持 --reportPath；未传时使用 `path.join(versionDir, entry.path)`；解析失败捕获并输出「题目解析失败：<原因>」。
- **T3**（runId、scenario、question_version）：generateEvalRunId 生成 `eval-q{id}-{version}-{ts}`；validateQuestionVersionForEval 在校验；parseAndWriteScore 传入 scenario=eval_question、question_version。
- **T4**（失败路径与版本隔离测试）：run-core.test.ts 单元测试 runId 格式与 question_version 校验；cli-integration.test.ts 含 run 题目不存在、缺 --id、缺 --version、端到端写入评分。

### 2.2 集成测试与 run 用例

```
npx vitest run scoring/eval-questions
✓ 35 tests passed (run-core、template-generator、manifest-loader、cli-integration)
```

run 相关用例：

- run 题目不存在时输出明确错误
- run 缺 --id 时输出用法
- run 缺 --version 时输出用法
- run 使用有效报告时写入评分（端到端：tmpRoot 下复制 rules/config/schema + sample 报告，执行 run，断言输出与 record）

### 2.3 关键路径导入链

```
main() → cmdRun()
  → loadManifest(versionDir)           [manifest-loader]
  → validateQuestionVersionForEval()   [run-core]
  → generateEvalRunId()                [run-core]
  → parseAndWriteScore()               [scoring/orchestrator]
    → writeScoreRecordSync()           [writer] → validateScenarioConstraints()
```

### 2.4 prd、progress 与 TDD 标记

- prd：4 个 US（US-001–US-004），均 passes: true。
- progress：每个 US 含 [TDD-RED]、[TDD-GREEN]、[TDD-REFACTOR]；格式为 `N failed` / `N passed`（未填写具体数字，但结构完整）。

### 2.5 question_version 校验

- **run-core**：`validateQuestionVersionForEval('eval_question', version)`，缺失或空字符串时 throw。
- **writer**：`validateScenarioConstraints(record)` 在 writeScoreRecordSync 写入前调用；scenario=eval_question 且 question_version 缺失时 throw。
- **单元测试**：run-core.test.ts 覆盖缺失、空字符串、仅空格、有效值、real_dev 不校验。

### 2.6 runId 格式

- 实现：`eval-${questionId}-${version}-${ts}` → `eval-q001-v1-1730812345`
- 约定：`scoring/docs/RUN_ID_CONVENTION.md` §2.2 `eval-q{id}-{version}-{ts}` ✅

### 2.7 写入记录字段

- 集成测试断言：`record.scenario === 'eval_question'`、`record.question_version === 'v1'`、`record.run_id` 匹配 `/^eval-q001-v1-\d+$/`。

## 3. 批判审计员结论（占比 >50%）

### 3.1 对抗性检查：遗漏任务

**批判审计员**：tasks T4.2 要求「v1、v2 各 run 一次，run_id 与 question_version 可区分」。当前集成测试仅覆盖 v1 单次 run，未显式执行 v2 run 并断言两版数据可区分。

**复核**：runId 格式与 question_version 的注入逻辑已由单元测试与单次 run 端到端验证；v2 会生成 `eval-q001-v2-*` 与 `question_version=v2`，与 v1 自然隔离。实现满足 AC-5；增加 v2 用例可增强回归，但不构成必须 gap。

**结论**：不构成阻塞 gap；建议后续迭代补充 v1+v2 双 run 集成用例。

**结论**：与已批准 tasks/plan 一致，无 gap。
