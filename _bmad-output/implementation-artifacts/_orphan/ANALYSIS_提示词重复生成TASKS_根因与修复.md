# 根因分析：为何提示词会不断重复生成 TASKS 文档

**议题**：SFT 提取扩展与 eval 题集生成 — Party-Mode + 审计子任务流程中，提示词导致不断重复生成 TASKS  
**日期**：2026-03-07  
**产出路径**：`_bmad-output/implementation-artifacts/_orphan/ANALYSIS_提示词重复生成TASKS_根因与修复.md`

---

## §1 现象描述

用户提示词要求：

1. **Party-Mode**：100 轮讨论，批判审计员 >30%，最后 3 轮无 gap 收敛，产出最终任务列表
2. **合并**：更新并合并 `TASKS_SFT提取扩展与eval题集生成_PARTY_MODE_100轮产出.md` 与 `TASKS_SFT提取扩展与eval题集生成.md`
3. **审计子任务**：完成后发起审计，要求：
   - `subagent_type: code-reviewer`
   - 使用 `audit-prompts.md §5` 提示词（或用户指定 §4）
   - 迭代至「完全覆盖、验证通过」
   - 最后 3 轮无 gap 才能收敛

**现象**：流程不断重复，TASKS 文档被反复生成或更新，无法稳定收敛。

---

## §2 根因分析

### 2.1 根因一：Prompt 与审计对象不匹配（主因）

**audit-prompts §5 的定位**：

> 请帮我仔细审阅**目前基于 tasks.md 的执行所做的代码实现** 是否完全覆盖了原始的需求设计文档、plan.md 以及 IMPLEMENTATION_GAPS.md 所有章节……

§5 是**执行 tasks 后的代码实现审计**，检查项包括：

- 代码实现是否覆盖需求
- 是否已执行集成测试与端到端功能测试
- 模块是否被生产代码关键路径导入、实例化并调用
- 是否已创建并维护 prd.json、progress.txt
- 涉及生产代码的每个 US 是否含 [TDD-RED]、[TDD-GREEN]、[TDD-REFACTOR]

**审计对象**：`TASKS_SFT提取扩展与eval题集生成.md` —— 这是一份**任务列表文档**，描述「要做什么」，**尚未实施**，因此：

- 无代码实现
- 无 prd.json、progress.txt
- 无 TDD 标记
- 无集成测试执行结果

**后果**：用 §5 原版提示词审计 TASKS 文档时，审计员会按 §5 检查项逐项核对，必然发现：

- 「无代码实现」
- 「无 prd/progress」
- 「无 TDD 三项标记」
- 「无集成测试执行」

这些 gap **无法通过修改 TASKS 文档解决**（TASKS 文档只描述任务，不包含实现）。每轮结论均为「未通过」，永远达不到「完全覆盖、验证通过」，形成死循环。用户或主 Agent 可能因此反复触发 Party-Mode 或「更新任务列表」，导致 TASKS 文档被不断重新生成。

---

### 2.2 根因二：§4 与 §5 的适用场景混淆

| 审计对象 | 适用 prompt | 说明 |
|----------|-------------|------|
| **tasks.md / TASKS 文档**（任务列表，未实施） | audit-prompts **§4** | 检查 tasks 是否覆盖 spec、plan、GAPS；验收标准、依赖关系、可执行性 |
| **代码实现**（已执行 tasks） | audit-prompts **§5** | 检查代码、TDD、集成测试、prd/progress |

若提示词要求「使用 §5」审计 TASKS 文档，则必然不匹配。**正确做法**：审计 TASKS 文档时使用 **§4** 或 **§4 精神适配**。

---

### 2.3 根因三：迭代责任与收敛条件

speckit-workflow 约定：

> **仅在审计报告结论为「完全覆盖、验证通过」时** 可结束该步骤；**否则根据报告迭代修改文档并再次审计**。

- **code-reviewer**：只负责审计，不负责修改
- **主 Agent**：根据 gap 列表修改 TASKS 文档，然后再次发起审计

若使用 §5 审计 TASKS，gap 列表恒为「无实现」类，主 Agent 无法通过修改 TASKS 消除，循环无法收敛。

---

### 2.4 根因四：Orphan TASKS 与 §4 的适配

audit-prompts §4 原文引用「需求设计文档、plan.md、IMPLEMENTATION_GAPS.md」。本 TASKS 为 **orphan**（无 speckit spec→plan→gaps 前置），故需 **§4 精神适配**：

- 以**需求映射表**替代「需求设计文档」
- 以**GAP 闭合表**替代 IMPLEMENTATION_GAPS 覆盖检查
- 检查项：需求映射、验收标准可量化、依赖关系、边界处理、GAP 闭合、禁止词、可执行性
- **不**检查：代码实现、prd、TDD、集成测试（因尚未实施）

---

## §3 修复建议

### 3.1 提示词修正

将用户原提示词中的：

> 使用 audit-prompts.md §5 提示词

修正为：

> 使用 **audit-prompts.md §4** 或 **TASKS 文档专用审计 prompt**（§4 精神适配）：审计对象为 TASKS 文档本身，检查需求映射、验收标准可量化、依赖关系、边界处理、GAP 闭合、禁止词、可执行性；**不**检查代码实现、prd、TDD、集成测试（因尚未实施）。参考 `AUDIT_TASKS_SFT提取扩展与eval题集生成_§5_round1.md` 的检查项结构。

### 3.2 审计对象与 prompt 对应关系

| 审计对象 | 适用 prompt |
|----------|-------------|
| TASKS 文档（任务列表，未实施） | §4 或 §4 精神适配 |
| 代码实现（已执行 tasks） | §5 |

### 3.3 收敛条件

对 TASKS 文档审计，收敛条件应基于**文档质量**：需求映射完整、验收可执行、GAP 闭合、无禁止词等。不应要求「有代码实现」「有 TDD」等实现阶段才有的证据。

---

## §4 总结

| 根因 | 影响 | 修复方向 |
|------|------|----------|
| §5 与 TASKS 文档不匹配 | 每轮必现「无实现」类 gap，无法消除，导致重复生成 | 使用 §4 或 TASKS 专用 prompt |
| §4 与 §5 适用场景混淆 | 错误选用 §5 审计 TASKS | 明确：TASKS 用 §4，实现用 §5 |
| 迭代责任未分离 | 无人修改文档，或修改无效 | 明确主 Agent 修改、子代理审计 |
| 收敛条件与对象不匹配 | 永远达不到 3 轮无 gap | 收敛条件基于文档质量 |
