### 2.1 T12 与 §4.1 引用关系

- **§4.1 当前定义**：`audit-prompts.md` §4.1 规定 tasks 阶段报告结尾必须含「可解析评分块」（总体评级 + 维度评分），供 `parseAndWriteScore` 解析。

- **维度映射**：§4.1 表格给出「需求完整性、可测试性、一致性、可追溯性」四维度。T12 说「与 tasks 阶段一致」，即 spec/plan/GAPS/执行复用同一格式。若各阶段用不同维度名，解析器需支持；当前 parseAndWriteScore 假定固定维度。**实施时需确认**：各阶段是否严格复用 §4.1 表格维度，或解析器是否支持维度名映射。任务描述未显式约束，属实施细节，**不构成本轮阻塞 gap**，建议实施时在 T12 验收中补充「各阶段块格式与 §4.1 一致」的说明。

### 2.2 T12 与 §4 表述重复性

- **§4（tasks）**：已强制「报告结尾必须包含 §4.1 规定的可解析评分块」。
- **T12**：为 §1、§2、§3、§5 新增同类要求。五阶段中 tasks 已满足，T12 补足其余四阶段。表述上「五阶段 prompt 均要求」包含 §4，无矛盾；重复仅在于强调一致性，**可接受**。

## 三、批判审计员结论（强制，占比 >70%）

### 3.1 对抗视角一：T12 验收标准伪可验证性

**GAP-T12-1：T12 验收标准「在 §1、§2、§3、§5 各有匹配」无法用 grep 独立验证（高）**

- **现象**：T12 验收标准为「1) grep `可解析评分块`、`§4.1` skills/speckit-workflow/references/audit-prompts.md 在 §1、§2、§3、§5 各有匹配或引用；2) 五阶段（spec/plan/GAPS/tasks/执行）的 prompt 均要求报告结尾含可解析块」。
- **问题**：
  1. **grep 无 section 上下文**：`grep "可解析评分块" audit-prompts.md` 仅返回匹配行，无法自动判断每行属于 §1、§2、§3 还是 §5。若实施者在 §1 内重复插入 4 次、§2/§3/§5 不加，grep 仍会返回多行匹配，**无法判定「各有」**。
  2. **§4.1 已有匹配**：当前 audit-prompts 的 §4.1 标题与 tasks prompt 已含「§4.1」「可解析评分块」。grep 会命中这些行，与 T12 新增的 §1/§2/§3/§5 处混在一起，**无法区分**。
  3. **「各有匹配或引用」歧义**：若解释为「整个文件中 grep 到 4 次以上即通过」，则仅在 §4 加 2 次也可满足，§1/§2/§3/§5 未改仍可误判通过。若解释为「每个 section 内至少 1 次」，grep 无法按 section 分组。
- **批判审计员**：该验收标准表面可执行（有 grep 命令），实则**伪可验证**——实施者无法在不依赖主 Agent 解释的情况下，仅凭验收标准独立判定通过/不通过。
- **修改建议**：
  - 方案 A：改为「`grep -c '可解析评分块' audit-prompts.md` 至少为 5」且「在 §1、§2、§3、§4、§5 各章节标题与下一 `##` 标题之间的内容中，分别包含至少 1 处『可解析评分块』或『§4.1』」，并给出具体验证命令（如 `awk` 按 `## 1.`、`## 2.` 等分段后逐段 grep）。
  - 方案 B：要求每处修改有唯一锚点，如 §1 prompt 后增加「`【§1 可解析块要求】`」等标记，验收时 `grep "【§1 可解析块要求】"` 等四条各有一匹配。
  - 方案 C：简化为「grep `可解析评分块` audit-prompts.md 输出行数 ≥ 5」且「grep `报告结尾必须包含.*可解析` audit-prompts.md 输出行数 ≥ 5」，明确每阶段 prompt 内须含「报告结尾必须包含…可解析…」的完整句。
