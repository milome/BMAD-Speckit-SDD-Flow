## §2 对抗性复验（批判审计员视角，占比 >50%）

### 2.1 对第 2 轮「无阻塞性模糊」结论的挑战

**批判审计员**：第 2 轮裁定 T4、T5「或等价逻辑」可接受，因验收标准通过行为约束功能等价。**对抗性质疑**：若实施者实现「等价逻辑」为在 instructions 内嵌 3 行 PowerShell 片段，而另一实施者实现为调用独立 check-sprint-ready.ps1，二者在可维护性、T5 的 check-prerequisites 复用性上是否等价？T5 明确写「调用 check-sprint-ready 或等价逻辑」——若采用内联实现，T5 的 check-prerequisites 如何「调用」？

**裁定**：T5 的「调用」隐含对外部可调用实体的引用；内联实现不满足「调用」语义。但 T5 为**可选任务**，且 T4 可独立使用「等价逻辑」（如主 Agent 内联检查文件存在性）。T5 若不实施，此歧义不触发。若实施 T5，则「调用 check-sprint-ready」应理解为调用 T1 产出的脚本或等效可调用单元，内联片段不满足。**实施指引**：T5 实施时优先调用 T1 脚本；若项目约束不允许新增脚本，则实现为 check-prerequisites 内部复用相同判定逻辑（逻辑等价，非「不调用」）。**结论**：可接受，不构成阻塞；实施时按上述理解执行即可。

---

### 2.2 对 T1「项目根可由 -RepoRoot 或环境推断」的挑战

**批判审计员**：第 1、2 轮未单独审查「环境推断」。**对抗性质疑**：「环境推断」的判定标准是什么？若实施者 A 使用 `$PWD`，实施者 B 使用 `git rev-parse --show-toplevel`，实施者 C 使用 `$env:REPO_ROOT`，三者在不同场景下可能指向不同目录，导致 sprint-status 路径解析不一致。

**裁定**：T1 验证命令使用 `./scripts/check-sprint-ready.ps1`，隐含当前工作目录为项目根或 scripts 的父级。常见约定为「在项目根执行」或「-RepoRoot 显式传入」。**建议**：实施时在脚本文档中注明「环境推断」的默认行为（如：未传 -RepoRoot 时，以 `Get-Location` 或调用方工作目录为项目根，并说明 typical 调用场景）。当前未强制要求，**低风险**，不构成阻塞；与第 2 轮对「或等价」的裁定一致（可接受边界）。

---

### 2.3 对 T2「继续」「force」「bypass」大小写与空格的挑战

**批判审计员**：第 2 轮确认 Challenger Condition 1 已落地，token 明确。**对抗性质疑**：用户输入 `Force`、`FORCE`、` 继续 `（首尾空格）是否视为确认？LLM 解析用户意图时可能对大小写、空格敏感度不同，导致误判放行或误拒。

**裁定**：T2 验收写「仅当用户输入 继续、force 或 bypass 时」。实施时应在 instructions 或等价逻辑中明确：建议采用**大小写不敏感**匹配，并对输入做 trim。此属实施细节，可由实施者在 instructions 中补充；TASKS 文档层级可不强制写「大小写不敏感」，因 Challenger Condition 1 的目标是「明确 token 避免 LLM 误判」，已通过列举具体 token 达成。**结论**：不构成阻塞；实施时采用保守匹配（如 case-insensitive + trim）可进一步降低风险。

### 2.4 对 T3「选项 3」在 sprint_status 不存在时的适用性挑战

**批判审计员**：T3 验收写「仅接受：(a) 用户显式提供的 story_path，或 (b) 用户选择选项 3 后提供的路径」。**对抗性质疑**：当 sprint_status **不存在**时，dev-story 的 instructions 进入「Non-sprint story discovery」分支，该分支当前无「选项 1、2、3、4」菜单（选项 3 存在于 sprint_status 存在但无 ready-for-dev 时的分支）。故 sprint_status 不存在时，(b) 实际 N/A。

**裁定**：经核查 `dev-story/instructions.xml`，sprint_status 不存在时进入 L79–106 的 Non-sprint 分支，该分支无编号选项，仅「提供路径」提示。T3 要求移除自动搜索后，唯一合法路径为用户直接提供 story_path，对应 (a)。(b) 在 sprint_status 不存在场景下不适用，但不与验收矛盾；(a) 已覆盖该场景。**结论**：无歧义，通过。

### 2.6 对 T8 场景完整性——sprint-status 存在但损坏的挑战

**批判审计员**：T8 场景 1 为「sprint-status 存在」，场景 2 为「sprint-status 缺失」。**对抗性质疑**：若 sprint-status 存在但 YAML 损坏或结构异常（如 development_status 为 null），T1 的 SPRINT_READY 应为何？该场景是否覆盖？
