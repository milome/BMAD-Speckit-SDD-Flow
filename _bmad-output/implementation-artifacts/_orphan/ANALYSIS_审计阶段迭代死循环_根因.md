# 根因分析：审计阶段迭代陷入死循环

**议题**：SFT 提取扩展与 eval 题集生成 — Party-Mode 产出 TASKS 后，审计子任务（code-reviewer + audit-prompts §5）迭代陷入死循环  
**日期**：2026-03-07  
**产出路径**：`_bmad-output/implementation-artifacts/_orphan/ANALYSIS_审计阶段迭代死循环_根因.md`

---

## §1 现象描述

用户提示词要求：

1. **Party-Mode**：100 轮讨论，批判审计员 >30%，最后 3 轮无 gap 收敛，产出最终任务列表
2. **审计子任务**：完成后发起审计，要求：
   - `subagent_type: code-reviewer`
   - 使用 `audit-prompts.md §5` 提示词
   - 迭代至「完全覆盖、验证通过」
   - 批判审计员 >50%，最后 3 轮无 gap 才能收敛

**现象**：审计阶段的迭代陷入死循环，无法收敛。

---

## §2 根因分析

### 2.1 根因一：Prompt 与审计对象不匹配（主因）

**audit-prompts §5 的定位**：

> 请帮我仔细审阅**目前基于 tasks.md 的执行所做的代码实现** 是否完全覆盖了原始的需求设计文档、plan.md 以及 IMPLEMENTATION_GAPS.md 所有章节……

§5 是**执行 tasks 后的代码实现审计**，检查项包括：

- 代码实现是否覆盖需求
- 是否已执行集成测试与端到端功能测试
- 模块是否被生产代码关键路径导入、实例化并调用
- 是否已创建并维护 prd.json、progress.txt
- 涉及生产代码的每个 US 是否含 [TDD-RED]、[TDD-GREEN]、[TDD-REFACTOR]
- branch_id、parseAndWriteScore 参数、scoring 配置等

**审计对象**：`TASKS_SFT提取扩展与eval题集生成.md` —— 这是一份**任务列表文档**，描述「要做什么」，**尚未实施**，因此：

- 无代码实现
- 无 prd.json、progress.txt
- 无 TDD 标记
- 无集成测试执行结果

**后果**：用 §5 原版提示词审计 TASKS 文档时，审计员会按 §5 检查项逐项核对，必然发现：

- 「无代码实现」
- 「无 prd/progress」
- 「无 TDD 三项标记」
- 「无集成测试执行」

这些 gap **无法通过修改 TASKS 文档解决**（TASKS 文档只描述任务，不包含实现）。每轮结论均为「未通过」，永远达不到「完全覆盖、验证通过」，形成死循环。

---

### 2.2 根因二：迭代责任未分离

speckit-workflow 约定：

> **仅在审计报告结论为「完全覆盖、验证通过」时** 可结束该步骤；**否则根据报告迭代修改文档并再次审计**。

隐含流程：**审计 → 修改 → 再审计**。

- **code-reviewer** 子代理：只负责**审计**，不负责**修改**
- **主 Agent**：负责根据 gap 列表修改文档，然后再次发起审计

若主 Agent：

1. 未明确执行「根据 gap 修改 TASKS 文档」步骤，或
2. 修改后仍用 §5 原版提示词审计（对象仍是 TASKS 文档，非实现），

则每次审计得到的 gap 列表相同（无代码、无 prd、无 TDD 等），主 Agent 无法通过修改 TASKS 文档消除这些 gap，循环无法收敛。

---

### 2.3 根因三：收敛条件与审计对象不匹配

audit-prompts-critical-auditor-appendix 的 **strict** 模式要求：

- 需**连续 3 轮**结论均为「完全覆盖、验证通过」
- 每轮批判审计员段落均注明「本轮无新 gap」

当 prompt 与对象不匹配时，审计员无法给出「完全覆盖」，故永远达不到「连续 3 轮无 gap」的收敛条件。

---

### 2.4 根因四：code-reviewer 默认 mode 与 TASKS 审计需求不符

`config/code-reviewer-config.yaml`：

- `default_mode: "code"`
- `code` 模式使用 `audit-prompts-code.md`，内容等效 audit-prompts §5（实现审计）

若调用 code-reviewer 时未显式指定 mode，会使用 code 模式，即实现审计 prompt，与 TASKS 文档审计需求不符。

---

### 2.5 根因五：批判审计员占比与收敛的张力

用户要求：批判审计员发言 >50%。

批判审计员的职责是：质疑、发现 gap、挑战共识。若每轮都「必须」大量发言，且其角色设定为「找问题」，可能倾向于持续提出新 gap 或潜在风险，使「本轮无新 gap」难以达成，形成结构性难以收敛。

---

## §3 现有成功案例的差异

`AUDIT_TASKS_SFT提取扩展与eval题集生成_§5_round1.md` 实际采用的是**§5 精神适配**，而非 §5 原版：

- 检查项：需求映射表、验收标准可量化、依赖关系、边界处理、GAP 闭合、禁止词等
- **未**检查：代码实现、prd、TDD、集成测试

即：对 TASKS 文档使用了**定制化**的审计逻辑，将 §5 的「逐项验证」精神迁移到 TASKS 文档场景，而非直接使用 §5 的检查项。

---

## §4 修复建议

### 4.1 明确审计对象与 prompt 的对应关系

| 审计对象           | 适用 prompt                         | 说明                         |
|--------------------|-------------------------------------|------------------------------|
| 代码实现（已执行 tasks） | audit-prompts §5 / audit-prompts-code.md | 实现审计，检查代码、TDD、集成测试等 |
| TASKS 文档（任务列表）  | §5 精神适配 或 专用 TASKS 审计 prompt   | 检查需求映射、验收标准、GAP 闭合等 |
| spec / plan / GAPS | audit-prompts §1 / §2 / §3          | 各阶段文档审计               |

### 4.2 审计 TASKS 文档时的正确做法

1. **不要**直接使用 audit-prompts §5 原版提示词
2. **应**使用「TASKS 文档专用」审计 prompt，或显式说明：
   - 审计对象为 TASKS 文档本身（非实现）
   - 检查项：需求映射、验收标准、依赖关系、边界处理、GAP 闭合、禁止词、可执行性
   - **不**检查：代码实现、prd、TDD、集成测试（因尚未实施）

### 4.3 迭代流程的明确分工

- **主 Agent**：根据审计报告的 gap 列表，修改 TASKS 文档，然后再次发起审计
- **code-reviewer**：仅负责审计，产出报告；不负责修改
- 若 gap 为「可通过修改 TASKS 文档解决」的类型，主 Agent 修改后重审；若 gap 为「需实施后才能解决」的类型，则不应在 TASKS 文档审计阶段要求消除

### 4.4 收敛条件的可达成性

- 对 TASKS 文档审计，收敛条件应基于「文档质量」：需求映射完整、验收可执行、GAP 闭合、无禁止词等
- 不应要求「有代码实现」「有 TDD」等实现阶段才有的证据

---

## §5 建议的 prompt 修正

将用户原提示词中的：

> 使用 audit-prompts.md §5 提示词

修正为：

> 使用 **TASKS 文档专用审计 prompt**（§5 精神适配）：审计对象为 TASKS 文档本身，检查需求映射、验收标准可量化、依赖关系、边界处理、GAP 闭合、禁止词、可执行性；**不**检查代码实现、prd、TDD、集成测试（因尚未实施）。参考 `AUDIT_TASKS_SFT提取扩展与eval题集生成_§5_round1.md` 的检查项结构。

或创建独立的 `audit-prompts-tasks.md`，供 TASKS 文档审计专用。

---

## §6 总结

| 根因                         | 影响                         | 修复方向                         |
|------------------------------|------------------------------|----------------------------------|
| §5 与 TASKS 文档不匹配       | 每轮必现「无实现」类 gap，无法消除 | 使用 TASKS 专用 prompt           |
| 迭代责任未分离               | 无人修改文档，或修改无效     | 明确主 Agent 修改、子代理审计    |
| 收敛条件与对象不匹配        | 永远达不到 3 轮无 gap        | 收敛条件基于文档质量            |
| code-reviewer 默认 code 模式 | 自动用实现审计 prompt        | 显式指定 mode 或传入定制 prompt  |
| 批判审计员占比与收敛张力    | 可能阻碍「无新 gap」         | 收敛时可允许「有条件同意」      |

---

*本分析基于 party-mode、audit-prompts、code-reviewer-config、speckit-workflow、bmad-bug-assistant 等既有约定与实现。*
