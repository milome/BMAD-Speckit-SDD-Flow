## §3 批判审计员裁定（占比 >50%）

### 3.1 关于「或等价」「或等效」的可操作性

**批判审计员**：T1、T4、T5 多处出现「或等价」「或等效结构」。**质疑**：若实施者 A 实现「等价脚本」为内联 PowerShell 片段，实施者 B 实现为独立 .ps1 文件，二者是否均算通过？若无功能边界定义，「等价」即模糊。

**证据请求**：T1 验收标准要求输出 JSON `{ "SPRINT_READY": boolean, "SPRINT_STATUS_PATH": string, "MESSAGE": string }`，且 `SPRINT_READY=true` 当且仅当指定路径存在且可解析。此条件是否足以唯一约束「等价」？

**裁定**：**可接受但存在边界风险**。验收标准通过「输出格式 + 判定逻辑」间接定义了功能等价。若实施者产出符合验收的脚本，即视为等价。建议在 T1 文档说明中补充：「等价指输出 JSON 格式与判定逻辑与本文档一致，实现形式可为独立脚本或可复用片段。」当前未补充，属低风险遗漏，不构成阻塞性模糊。

---

### 3.2 关于 T6「项目可修改」bmad-help.csv

**批判审计员**：T6 要求「若 `_bmad/_config/bmad-help.csv` 存在且**项目可修改**，则补充」。**质疑**：「项目可修改」的判定标准是什么？若文件由 BMAD 核心安装且每次升级覆盖，是否算「不可修改」？若项目 fork 了 _bmad 则可编辑，是否算「可修改」？实施者可能做出不同判定，导致验收时分歧。

**证据请求**：本项目 `_bmad` 的来源——是否为 git submodule、npm 包、或项目自有副本？若为 submodule，则项目无法直接修改 _bmad 内文件，应为「不可修改」。

**裁定**：**存在可判定性不足**。建议 T6 验收标准补充可执行判定步骤：「尝试在 bmad-help.csv 中追加或修改 Create Story、Dev Story 对应行；若文件为只读或受版本控制保护无法提交，则判定为不可修改，仅在 commands 文档中补充并在验收中注明。」当前文档仅有「可修改则补充；不可修改则注明」，未给出「如何判定可修改」的操作步骤。**列为未通过项**，需修改。

### 3.3 关于 T1「或等效结构」中的 development_status

**批判审计员**：T1 要求 sprint-status.yaml 含 `development_status` 或等效结构。**质疑**：sprint-status.yaml 的实际 schema 是什么？若仅有 `epics` 而无 `development_status`，是否算等效？若结构为 `sprint.status.by_epic`，是否等效？

**裁定**：**存在判定歧义**。若 sprint-planning 产出的 sprint-status 结构已在本项目或 BMAD 规范中定义，应引用该规范；否则「等效」无据可依。建议 T1 验收标准修改为：「sprint-status.yaml 存在且可解析，且包含可识别的 sprint 跟踪结构（如 development_status、epics、或项目约定的等效字段）；若项目使用非标准结构，须在脚本文档中说明支持的字段。」当前未细化，**中低风险**，建议补充以消除歧义。

### 3.5 关于 §2.2 豁免规则中「但仍建议检查」

**批判审计员**：显式 story 路径放行时，「但仍建议检查 sprint-status 是否存在以决定是否更新」。**质疑**：此「建议」是否需在 T2、T3、T4 中实现？若实现，具体行为是「提示用户」还是「静默检查并更新」？若未实现，是否违反共识？

**裁定**：**豁免规则中的建议不强制落地**。§1.2 目标为「强制 sprint-planning 前置」，显式 story_path 已视为用户已知目标而放行。「建议检查」为增强型行为，非门控必选。T2、T3、T4 未要求实现此建议，与当前任务范围一致。**通过**。

### 3.6 关于 Greenfield「story docs path」路径的存在性

**批判审计员**：T2、T6、T8 均以「若 instructions.xml 含」「若有 greenfield 路径」为前提。**质疑**：若 create-story instructions.xml 实际**不含**该分支，则 T2 的 GAP-002 验收项、T6 的文档说明、T8 的场景 6a 均变为 N/A。实施者如何知道该分支是否存在？

**裁定**：**需在实施前做一次性判定**。建议在 T2 描述中补充：「实施前须检查 create-story instructions.xml 是否含『提供 story docs path』分支；若含，则须在验收中说明该分支在 sprint-status 缺失时的行为；若不含，则 T2、T6 中相关验收项标注为 N/A，T8 场景 6a 跳过。」当前 T2 验收已写「若 instructions.xml 含……需在验收中说明」，隐含「先检查再决定」的逻辑，但未显式要求「实施前检查并记录分支存在性」。**低风险**，建议 T2 增加此步骤以消除歧义。

### 3.9 关于 T8 场景 6a 与 场景 7 的可验证性

**批判审计员**：T8 场景 6a 写「若有 greenfield『story docs path』路径，sprint-status 缺失时该路径行为符合 §2.2 豁免（放行）」；场景 7 写「直接 BUGFIX 实施（如通过 bmad-bug-assistant，不经过 standalone-tasks）→ 不受影响」。**质疑**：场景 6a 的「符合 §2.2 豁免」具体指什么行为？放行的验收标准是「用户能成功执行 create-story 且无需 sprint-status」，还是「有明确提示但可 bypass」？若未定义「符合豁免」的判定标准，不同实施者可能给出不同结论。

**裁定**：§2.2 豁免明确「提供 story docs path 路径保留，作为合法入口」，即放行 = 可进入。**场景 6a 的验收标准可解读为**：sprint-status 缺失时，用户通过 story docs path 入口可正常执行 create-story，无额外门控阻断。此解读与 §2.2 一致。建议在 T8 场景 6a 中补充「放行 = 可进入 create-story，无需 sprint-status」，以消除歧义。当前为低风险，不构成阻塞。
