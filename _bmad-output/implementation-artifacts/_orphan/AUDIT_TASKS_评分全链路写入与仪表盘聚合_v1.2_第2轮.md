## 二、T12 验收标准可执行性复验

### 2.1 独立可验证性

| 验收项 | 命令 | 预期输出 | 判定方式 |
|--------|------|----------|----------|
| 1 | `grep -c '【§1 可解析块要求】' <path>` | 1 | 输出 = 1 即通过 |
| 2 | `grep -c '【§2 可解析块要求】' <path>` | 1 | 输出 = 1 即通过 |
| 3 | `grep -c '【§3 可解析块要求】' <path>` | 1 | 输出 = 1 即通过 |
| 4 | `grep -c '【§5 可解析块要求】' <path>` | 1 | 输出 = 1 即通过 |

- **grep -c 行为**：匹配 0 次时退出码 1、输出 0；匹配 1 次时退出码 0、输出 1。实施者可运行命令后比较输出与 1，或使用 `[ $(grep -c '...' file) -eq 1 ]` 判定。

- **GAP-T12-1 消除判定**：修订后验收标准可通过 4 条独立 grep -c 命令判定通过/不通过，无需按 section 分组，无「各有匹配」歧义。**GAP-T12-1 已消除**。

## 四、批判审计员结论（强制，占比 >70%）

### 4.1 对抗视角一：T12 修订后验收是否真正可独立验证？

**检查点**：实施者能否在不依赖主 Agent 解释的情况下，仅凭验收标准独立判定通过/不通过？

- **修订前（GAP-T12-1）**：grep「可解析评分块」无法区分 §1/§2/§3/§5，易误判；需人工逐 section 核对。
- **修订后**：4 条 `grep -c '【§N 可解析块要求】'` 各输出 1。每条命令独立、输出可比较、无歧义。
- **文件路径**：若实施者从项目根执行 `grep -c '【§1 可解析块要求】' audit-prompts.md`，会得到「No such file」。验收标准未显式写路径。但 修改路径 与 验收标准 同处 T12 表格，实施者修改的是 `skills/speckit-workflow/references/audit-prompts.md`，自然会对此文件执行 grep。**可推断**，非致命缺陷。建议下轮或文档迭代时补全路径。
- **判定**：修订后验收**可独立验证**，GAP-T12-1 已消除。

### 4.2 对抗视角二：grep 退出码与「通过」判定

**检查点**：若文件不存在或路径错误，grep 退出码如何？实施者如何判定「通过」？

- **grep -c 行为**：文件不存在 → exit 2；匹配 0 次 → exit 1、输出 0；匹配 1 次 → exit 0、输出 1。
- **「为 1」含义**：输出数字等于 1。实施者检查 `$(grep -c ...)` 是否等于 1，或直接目视输出。
- **边界**：文件不存在时 grep 报错，不会输出 1，故 implicitly 判定为不通过。逻辑自洽。
- **判定**：退出码与判定逻辑明确，无需额外说明。

### 4.3 对抗视角三：锚点是否可能与文档其他内容冲突？

**检查点**：`【§N 可解析块要求】` 会否与 audit-prompts 其他章节、示例、历史内容冲突？

- **audit-prompts 结构**：§1～§5 为 spec/plan/GAPS/tasks/执行 五阶段 prompt。§4.1 定义可解析块格式，不含【§N 可解析块要求】。
- **锚点用途**：仅在 §1、§2、§3、§5 的 prompt 内各插入一次，作为验收锚点。
- **示例/历史**：若将来在示例中写「例如：`【§1 可解析块要求】`」，会导致 grep -c 为 2，验收不通过。当前文档无此类内容；T12 具体修改 明确「每处插入的句首或句尾增加唯一锚点」，即每 section 仅一处。实施者按此执行即可。
- **与 §4 关系**：§4（tasks）已强制可解析块，T12 不为其增加锚点；验收标准 5) 说明「§4 已有，§1/§2/§3/§5 通过上述锚点落实」。无冲突。
- **判定**：锚点格式唯一，与现有内容无冲突；实施时遵守「每处各一」即满足验收。

### 4.4 对抗视角四：锚点是否需说明「仅用于验收、不出现于面向用户的 prompt 输出」？

**检查点**：锚点会否被误读为「审计报告必须输出」？

- **插入位置**：在 prompt 的「报告结尾必须明确给出结论」之后，与「报告结尾必须包含 §4.1 规定的可解析评分块」一并加入。锚点作为该要求的元标记。
- **Agent 行为**：code-reviewer 收到 prompt，产出审计报告。prompt 要求报告含可解析块，未要求输出【§N 可解析块要求】。通常 agent 不会将指令中的元标记写入报告。
- **风险**：若 agent 误将锚点复制进报告，报告会含【§1 可解析块要求】等。但验收 grepping 的是 audit-prompts.md，非报告文件，故不影响 T12 验收。报告解析器（parseAndWriteScore）预期的是 §4.1 定义的可解析块格式，非锚点。低风险。
- **建议**：在 T12 注中加一句「锚点仅用于本任务验收，不要求出现在审计报告正文」可消除歧义，**非阻塞**。

### 4.5 对抗视角五：有无修订引入的新 gap？

**检查点**：v1.2.1 修订是否引入遗漏、矛盾或不可执行项？

- **遗漏**：4 个锚点对应 §1、§2、§3、§5，与 具体修改 的 4 处插入一一对应；§4 已有要求，无需锚点。无遗漏。
- **矛盾**：验收标准 5) 与 1-4) 无矛盾；5) 为说明性，1-4) 为可执行验收。
- **不可执行项**：grep -c 为标准化命令，Windows/Linux/macOS 均支持。路径建议改进但非不可执行。
- **与第 1 轮方案 B 一致性**：第 1 轮建议「§1 prompt 后增加【§1 可解析块要求】等标记，验收时 grep 四条各有一匹配」。v1.2.1 采用唯一锚点 + grep -c 各为 1，与方案 B 一致。
- **判定**：**无修订引入的新 gap**。
