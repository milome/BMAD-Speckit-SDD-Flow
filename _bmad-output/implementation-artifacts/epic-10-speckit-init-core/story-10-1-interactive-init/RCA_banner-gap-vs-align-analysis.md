# Banner「分隔符–SPECKIT 空隙」与「SPECKIT 错位」深入分析

## 目标

在**既无分隔符与 SPECKIT 之间的空隙**又**保证 SPECKIT 竖线不错位**的前提下，寻找可行且可实施的最优方案。

---

## 一、根因：为何会二选一（有空隙 vs 错位）

### 1.1 终端光标的两种推进方式

输出一串字符后，光标下一列由终端实现决定，有两种可能：

| 推进方式 | 含义 | 典型环境 |
|----------|------|----------|
| **按字符数** | 每输出 1 个 code unit，光标 +1（或按码点 +1） | 部分 Windows ConHost/PowerShell、部分老旧终端 |
| **按显示列（cell）** | 每输出 1 个「占 1 列的字符」光标 +1，占 2 列的（如 █）光标 +2 | 多数现代终端（Windows Terminal、VS Code 集成终端等） |

`getRenderWidth()` 的语义是「显示列数」：ASCII 空格=1 列，█/═=2 列。  
若终端按**字符数**推进，则「输出 N 个字符」≠「光标前进 getRenderWidth(N 个字符) 列」，二者脱节。

### 1.2 为何「仅空格填充到 82 列再拼 SPECKIT」会错位

- 做法：`bmad + separator + ' '.repeat(gap) + SPECKIT`，其中 `gap = 82 - endCol`（按显示列算）。
- 假设：输出上述字符串后，光标在**显示列 82**。
- 事实：若终端按**字符数**推进，输出后光标在「字符位置」= `len(bmad)+len(separator)+gap`，**不是**显示列 82。下一字符（SPECKIT）就从错误列开始画，出现错位。

因此：**不强制到 82 列**时，SPECKIT 起始列依赖终端如何推进光标；在你当前环境下该推进方式与 `getRenderWidth` 不一致，导致错位。

### 1.3 为何「\x1b[82G + SPECKIT」会保留空隙

- 做法：`Part1 + Part2 + \x1b[82G + Part3`，不再在 Part2 后填空格。
- 效果：`\x1b[82G` 显式把光标移到**显示列 82**，SPECKIT 一定从 82 列开始，竖线对齐。
- 代价：Part2 结束列 = Part1 显示宽度 + Part2 显示宽度，每行不同（约 44～48）；到 82 列之间这段就是「分隔符与 SPECKIT 之间的空隙」。

所以：**要保证不错位，就必须用 \x1b[82G；在 Part2 后不填任何东西时，这段空隙必然存在。**

---

## 二、在「必须保证不错位」下的可选方向

前提：**SPECKIT 前必须保留 \x1b[82G**（或等价列定位），否则在你当前终端上错位无法避免。在此前提下讨论「能否同时消除空隙」。

### 方案 A：中间区按显示宽度填满到 82，再 \x1b[82G + SPECKIT（仍用 82G 保底）

- 构造：`Part1 + \x1b[sepCol]G + middle + \x1b[82G + Part3`。  
  `middle` 按**显示宽度**凑满到 82 列（例如 Part2 + 空格，总宽度 = 82 - sepCol）。
- 意图：中间区「画满」到 82，视觉上无空隙；再用 \x1b[82G 保证即使终端算错，SPECKIT 也在 82。
- 结果：
  - 若终端**按显示列**推进：画完 middle 后光标已在 82，再 \x1b[82G 是冗余，**无空隙、不错位**。
  - 若终端**按字符数**推进：画完 middle 后光标不在 82，\x1b[82G 把光标拉到 82，SPECKIT 对齐，但「middle 在屏上画到的列」与「82」之间仍可能有一段空白（终端画了 middle，但光标没到 82，82G 再跳到 82），**可能有空隙、但不错位**。

结论：方案 A 在「按字符数推进」的终端上**不能保证**完全消除空隙，但能保证不错位；在「按显示列推进」的终端上可以既无空隙又不错位。可作为**统一实现**：始终「middle 填满 + \x1b[82G + SPECKIT」，在不同终端上得到不同视觉效果，但对齐一致。

### 方案 B：运行时检测「光标是否按显示列推进」，再分支

- 思路：在输出 banner 前，做一次**探测**（例如输出已知字符序列 + 查询光标位置，或输出测试图案观察是否对齐），判断当前终端是「按字符数」还是「按显示列」推进。
- 分支：
  - 若判定为**按显示列**：用「Part1 + Part2 + 按显示宽度填满到 82 的空格 + SPECKIT」，**不用** \x1b[82G，无空隙、不错位。
  - 若判定为**按字符数**或探测失败：用「Part1 + Part2 + \x1b[82G + SPECKIT」，接受空隙，保证不错位。
- 优点：在支持显示列的终端上可做到**既无空隙又不错位**；在其它终端上保底不错位。
- 成本：需要设计并维护探测逻辑、可能误判、需在不同终端上验证。

### 方案 C：固定「中间区」总显示宽度为 82 - sepCol，且中间区仅用 ASCII 空格（无宽字符）

- 思路：Part2 不用 `═══`，改为「仅 ASCII 空格」的中间区，总长度 = 82 - getRenderWidth(Part1)。这样「字符数」=「显示列数」，按字符推进的终端也会把光标推到 82。
- 问题：Part2 不再有 `═══`，视觉上不再是「BMAD ═══ SPECKIT」，与现有设计不符；若仍要保留 ═══，则中间区仍有宽字符，按字符推进时仍会错位。
- 结论：仅当可以放弃中间区 ═══ 时，方案 C 才能在「按字符推进」的终端上同时无空隙、不错位；否则不能。

### 方案 D：接受当前取舍，文档化「为何有空隙」

- 保持：Part1 + Part2 + \x1b[82G + Part3，不填中间空格。
- 在代码或文档中明确：为保证 SPECKIT 在各类终端（含按字符推进）上不错位，必须使用 \x1b[82G；因此分隔符与 SPECKIT 之间会有一段空隙，为已知取舍。

---

## 三、推荐最优方向（可实施）

1. **短期/稳妥**：采用 **方案 A**  
   - 实现：每行 `Part1 + \x1b[sepCol]G + middle + \x1b[82G + Part3`。  
   - `middle` = Part2（分隔符或 9 空格）+ 若不足则用 ASCII 空格填满到总显示宽度 (82 - sepCol)（用现有 `getRenderWidth` 与 BMAD_RENDER_WIDTHS 计算）。  
   - 效果：在「按显示列」的终端上可减少或消除空隙；在「按字符数」的终端上至少保证不错位，空隙可能仍存在。
2. **中期/更优**：在方案 A 基础上增加 **方案 B** 的探测与分支  
   - 若探测为「按显示列」：仅用「Part1 + Part2 + 填满到 82 的空格 + Part3」（不再用 \x1b[82G），实现无空隙且不错位。  
   - 否则：使用方案 A（或当前 Part1+Part2+\x1b[82G+Part3），保证不错位。
3. **文档**：在 TASKS_RCA 或 BUGFIX 中写明「为何可能仍有空隙」及「方案 A/B 的适用条件」，便于后续在不同终端上再优化。

---

## 四、小结

| 方案 | 无空隙 | 不错位 | 实现难度 | 备注 |
|------|--------|--------|----------|------|
| 仅空格填充、不用 82G | 理论可 | 在你终端否 | 低 | 已证实错位，不可接受 |
| Part2 + \x1b[82G + Part3（当前） | 否 | 是 | 低 | 接受空隙 |
| A：middle 填满 + \x1b[82G + Part3 | 视终端而定 | 是 | 中 | 推荐先做 |
| B：探测 + 分支 | 在支持终端可 | 是 | 高 | 推荐中期 |
| C：中间区仅 ASCII | 可 | 仅当无 ═══ | 中 | 需改视觉 |

**当前最大问题的本质**：终端「光标按字符数推进」与「按显示列推进」的差异，导致「无空隙」与「不错位」在你环境下不能同时满足，除非（1）用方案 A 在部分终端上减轻空隙，或（2）用方案 B 在可探测的终端上做到既无空隙又不错位。

建议：先按 **方案 A** 实现并保留 \x1b[82G，再视需要加 **方案 B** 的探测与分支；同时在文档中明确上述根因与取舍。
