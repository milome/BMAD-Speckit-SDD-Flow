## 5. 模糊表述与边界条件检查

### 5.1 已消除的模糊表述（上一轮）

- ~~veto_items 映射与合并策略未定义~~ → 已明确并入格式与语义
- ~~AC-B05-7 验证范围不明确~~ → 已列出两种集成场景

### 5.2 当前是否仍存在模糊表述

| 检查项 | 结论 | 说明 |
|--------|------|------|
| issues 与 veto_items 重叠 | **无模糊** | GAP-B05 schema 中 issues 与 veto_items 独立；issues 为 `{severity, description}`，veto_items 为 `item_id[]`，语义不同，实现时分别映射再合并即可 |
| veto_items 的 item_id 是否须在 YAML 中定义 | **轻微可选澄清** | spec 写「与现有 veto 模块的 check_items 语义一致」；veto 模块根据 rules YAML 的 vetoItemIds 判定。若 LLM 返回的 item_id 不在 YAML 中，仍会写入 check_items 但不会触发 veto_triggered。此为合理默认行为，spec 可接受 |
| resolveItemId 导入路径 | **无模糊** | 「从 audit-item-mapping 导入」足以指导实现；plan/tasks 会细化 |
| audit-generic 的 stage 参数 | **无模糊** | spec 写「stage 作为 llmStructuredExtract 的第二参数传入」；parseGenericReport 可传入 spec/plan/tasks 等，与 resolveItemId 的 AuditStage 兼容 |

### 5.3 术语与边界

| 术语/边界 | 定义清晰度 | 说明 |
|-----------|------------|------|
| ParseError | 清晰 | 与当前行为一致，指解析失败时抛出的异常 |
| GRADE_TO_SCORE | 清晰 | 现有映射表，spec 引用即可 |
| CheckItem | 清晰 | 含 item_id、passed、score_delta、note |
| 正则成功/失败 | 清晰 | 指 extractOverallGrade 返回非 null / null |
