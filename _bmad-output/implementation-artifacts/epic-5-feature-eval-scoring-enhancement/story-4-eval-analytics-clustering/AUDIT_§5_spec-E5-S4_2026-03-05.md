## 3. spec 存在模糊表述（需 clarify 澄清）

以下位置存在需求描述不明确、边界条件未定义或术语歧义，建议通过 clarify 流程补充。

### 3.1 cluster_id 生成规则未定义

- **位置**：§3.1.1 `WeaknessCluster.cluster_id: string`
- **问题**：未说明 `cluster_id` 的生成规则（如是否等于 `primary_item_ids[0]`、是否使用 hash、或其它规则）
- **影响**：实现时可能产生不一致

### 3.2 score_delta 缺失/异常时的 severity 处理

- **位置**：§3.1.5 severity_distribution
- **问题**：当 `score_delta` 为 `null`、`undefined` 或非数字时，severity 归属未定义
- **边界**：需明确 fallback（如归入「低」或单独计数）

### 3.3 空 note 或 note 缺失

- **位置**：§3.1.4 关键词提取
- **问题**：当 check_item 的 `note` 为空字符串、`null` 或 `undefined` 时，是否参与统计、是否影响 keywords 数组未定义

### 3.4 top-5 同频 tie-breaking

- **位置**：§3.1.4「取 top-5 关键词（按词频降序）」
- **问题**：多个词同频时，排序或选取规则未说明（如字母序、出现顺序等）

### 3.5 types.ts 修改未显式列出

- **位置**：§2.1、§3.2
- **问题**：`scoring/coach/types.ts` 的修改未作为独立变更点列出，仅通过 CoachDiagnosisReport 字段提及
- **影响**：实施时可能遗漏

### 3.6 vitest 未在 spec 中提及

- **位置**：全文
- **问题**：原始需求 §5.3 要求「单测使用 vitest」，spec 未显式写出测试框架
- **影响**：次要，通常沿用项目惯例
