# 批判审计员结论格式（所有阶段通用）

本附录定义 audit-prompts §1–§5 及 PRD/Arch/PR 审计中**批判审计员（Critical Auditor）**段落的必填格式与结构。当严格度为 standard 或 strict 时，审计报告**必须**包含符合本格式的批判审计员段落；simple 模式可省略。

---

## 1. 必填结构

批判审计员段落必须包含以下三项，**缺任一项则判「审计报告格式不合格」，需重新生成**：

| 序号 | 必填项 | 说明 |
|------|--------|------|
| 1 | 已检查的维度列表 | 列出本审计中实际检查的质疑维度 |
| 2 | 每维度结论 | 每个维度的检查结果（通过/未通过及说明） |
| 3 | 本轮 gap 结论 | 明确写出「本轮无新 gap」或「本轮存在 gap」；若存在 gap，须列出具体项 |

---

## 2. 输出格式

审计报告须包含独立段落，标题为：

```
## 批判审计员结论
```

该段落**字数或条目数不少于报告其余部分**（即占比 ≥50%），以确保模型分配足够 token 给对抗视角，避免敷衍。

---

## 3. 批判审计员检查维度（文档/代码审计场景）

从对抗视角检查，聚焦文档/代码审计，非多方案讨论。建议覆盖以下维度（可按阶段裁剪）：

| 维度 | 说明 |
|------|------|
| 遗漏需求点 | 原始需求/前置文档中有但未覆盖的要点 |
| 边界未定义 | 边界条件、异常路径未明确定义 |
| 验收不可执行 | 验收标准不可量化、不可验证 |
| 与前置文档矛盾 | 与 spec/plan/IMPLEMENTATION_GAPS 等矛盾 |
| 孤岛模块 | 模块内部完整但未被生产代码关键路径导入、调用 |
| 伪实现/占位 | 假完成、TODO、预留、占位式实现 |
| TDD 未执行 | 涉及生产代码的 US 缺 [TDD-RED]/[TDD-GREEN]/[TDD-REFACTOR] 任一项 |
| 行号/路径漂移 | 引用的行号、路径失效或与当前实现不符 |
| 验收一致性 | 验收命令未实际执行或结果与宣称不符 |

---

## 4. 示例结构

```markdown
## 批判审计员结论

**已检查维度**：遗漏需求点、边界未定义、验收不可执行、与前置文档矛盾、孤岛模块、伪实现、TDD 未执行、行号漂移、验收一致性。

**每维度结论**：
- 遗漏需求点：已逐条对照需求文档，无遗漏。
- 边界未定义：边界条件已在 spec 中明确。
- 验收不可执行：验收命令已执行，结果可验证。
- …（其余维度）

**本轮结论**：本轮无新 gap。第 N 轮；建议累计至连续 3 轮无 gap 后收敛。
```

若存在 gap，则写：

```markdown
**本轮结论**：本轮存在 gap。具体项：1) XXX；2) YYY。不计数，修复后重新发起审计。
```

---

## 5. 与收敛条件的关系

- **standard**：单次审计通过且批判审计员段落格式合格即结束。
- **strict**：需**连续 3 轮**结论均为「完全覆盖、验证通过」且每轮批判审计员段落均注明「本轮无新 gap」；任一轮为「存在 gap」则从下一轮重新计数。
- **simple**：可省略批判审计员段落；单次通过即结束（不保证质量，仅用于快速验证）。

---

## 6. 报告格式校验（GAP-CONV-12）

主 Agent 或在子代理 prompt 中可执行格式校验，作为审计通过条件之一：

- 检查报告是否包含「## 批判审计员结论」段落
- 检查该段落字数占比 ≥50%（该段落字数 ÷ 报告总字数 ≥ 0.5）
- 缺任一项则判「审计报告格式不合格」，需重新生成

校验逻辑可内嵌于主 Agent 流程或由独立脚本执行；脚本示例：解析报告 markdown，定位 `## 批判审计员结论` 至下一 `##` 或文末，计算字数并与全文比较。

---

## 7. 逐条对照格式报告的可解析评分块（tasks 阶段强制）

当审计报告采用**逐条对照格式**（表格 + ✅ 式逐条验证 + 结论）时，**必须在 §5 结论之后**追加可解析评分块，否则 `parseAndWriteScore` 无法解析，仪表盘不显示评级。

**要求**：逐条对照格式报告须在「完全覆盖、验证通过」等结论后追加以下块：

```markdown
## 可解析评分块（供 parseAndWriteScore）

总体评级: [A|B|C|D]

维度评分:
- 需求完整性: XX/100
- 可测试性: XX/100
- 一致性: XX/100
- 可追溯性: XX/100
```

**禁止用描述代替结构化块**：不得在总结或正文中用「可解析评分块（总体评级 X，维度分 Y–Z）」等文字概括；必须在报告中输出**完整的结构化块**，包含独立一行 `总体评级: X` 以及四行 `- 维度名: XX/100`。总体评级只能是 **A/B/C/D**（不得使用 A-、C+ 等）。维度分须逐行写出，不得用区间或概括代替。

**反例（无效输出）**：

| 无效写法 | 问题 |
|----------|------|
| `可解析评分块（总体评级 A，维度分 92–95）` | 描述句，非结构化块 |
| `总体评级: A-`、`C+` | 非 A/B/C/D，正则不匹配 |
| `维度分 92–95`、`各维度 90+` | 区间、缺具体维度名、格式不符 |
| 仅 `维度评分:` 标题无下文 | 无 `- 维度名: XX/100` 行，解析不到 |

**维度分与逐条对照结论的映射建议**：

| 逐条对照结论       | 建议总体评级 | 建议各维度分 |
|--------------------|--------------|--------------|
| 完全覆盖、验证通过 | A            | 90+          |
| 部分覆盖、minor 问题 | B            | 80+          |
| 需修改后重新审计   | C            | 70+          |
| 严重问题、不通过   | D            | 60 及以下    |

**示例**（在结论段落后追加）：

```markdown
## 5. 结论

**完全覆盖、验证通过。**

（逐条对照正文...）

## 可解析评分块（供 parseAndWriteScore）

总体评级: A

维度评分:
- 需求完整性: 95/100
- 可测试性: 92/100
- 一致性: 90/100
- 可追溯性: 88/100
```
